# 執筆スタイルガイド（The Internals of PostgreSQL 風）

参考: The Internals of PostgreSQL（interdb.jp/pg）。データベース／システムソフトウェアの
「内部構造」を、エンジニア向けに図と実例で解き明かす日本語技術ドキュメントの作法。

## 全体構成

- **章（Chapter）→ 節（Section）→ 小節（Subsection）** の3階層。
  - 章番号 `第N章`、節番号 `N.1`、小節 `N.1.1`。
- 各章は **冒頭の概要段落** で「この章で何を扱い、なぜ重要か」を 2〜4 文で述べる。
- 章末に **まとめ** または **次章への橋渡し** を置くと読みやすい。
- 全体は「基礎概念 → データ構造 → 動作（アルゴリズム）→ 具体例 → 応用・注意点」の順で積み上げる。

## 1ページ（1章）の典型パターン

1. 概要（この章のスコープと前提）
2. 対象の **データ構造／レイアウト** を図で提示（メモリ構造、ページ構造、ヘッダ等）
3. その構造を **操作する処理の流れ** をシーケンス図／フローチャートで提示
4. **実際のSQL・シェル・出力例** を示して内部状態を観察させる（`pageinspect`, `pg_freespace` 等）
5. ソースコード上の該当箇所・構造体名（例: `HeapTupleHeaderData`）への言及
6. バージョン差異・性能上の含意・運用上の注意

## 文体

- 常体（である調）で簡潔に。一文を短く、主語と述語を明確に。
- 用語は初出で定義し、英語の正式名と略語を併記する（例: 「先行書き込みログ（Write Ahead Logging, WAL）」）。
- 推測と事実を区別する。仕様・実装に基づく事実を優先し、曖昧な箇所は明示する。
- 過度な比喩を避け、図とデータで説明する。

## 図（Mermaid）の使いどころ

- **データ構造／レイアウト**: `flowchart` または `classDiagram` でフィールドを矩形に。
- **処理の流れ**: 複数コンポーネント間のやり取りは `sequenceDiagram`。単一処理の分岐は `flowchart`。
- **状態遷移**: `stateDiagram-v2`（トランザクション状態、リカバリ状態など）。
- すべての図に `図 N.M: 説明` のキャプションを付け、本文から「図 N.M に示すように…」と必ず参照する。
- 1つの図に詰め込みすぎない。複雑な対象は複数図に分割する。

## 例（SQL/コード）の示し方

- 入力（SQL/シェル）と出力（psql の結果）をセットで示す。
- 実行可能・再現可能な最小例にする。スキーマやデータの前提も明記する。
- 内部を覗くための拡張（`pageinspect`, `pg_buffercache`, `pg_visibility` 等）を活用した観察例は特に有効。

## 品質チェック

- 章番号・節番号・図番号が連番で、本文参照と一致しているか。
- すべての内部リンク（サイドバー・前後ナビ・章間リンク）が実在ファイルを指すか。
- Mermaid 構文が壊れていないか（ノードIDの重複、未閉じの括弧など）。
- 技術的主張に裏付けがあるか（ソース・公式ドキュメント・実測）。
