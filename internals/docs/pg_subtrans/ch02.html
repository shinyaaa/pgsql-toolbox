第2章　pg_subtrans の全体像と SLRU 基盤

本章では pg_subtrans がどのような基盤の上に構築され、ディスクとメモリにどう配置されるかを 概観する。pg_subtrans は SLRU（Simple LRU）と呼ばれる汎用機構の上に載る、薄い マネージャである。

2.1　SLRU マネージャの役割

SLRU は、固定長スロットからなる共有メモリ上のページバッファと、ディスク上のセグメントファイルを管理する 汎用機構で、src/backend/access/transam/slru.c に実装される。pg_xact（コミット状態）、 pg_multixact、pg_commit_ts なども同じ SLRU を土台にしている。

図 2.1 に示すように、subtrans.c は SLRU の API を呼び出すだけの薄い層であり、ページの読み書き・ キャッシュ・ディスク入出力はすべて SLRU 側が担う。subtrans.c は xact.c をベースにして書かれているが、 コメントが述べるとおり堅牢性の要件はまったく異なる。

SLRU への登録は SUBTRANSShmemRequest() 内の SimpleLruRequest() で行われる。主な パラメータは次のとおりである。

  • .name = "subtransaction" … 統計ビュー pg_stat_slru 等で使われる名前。
  • .Dir = "pg_subtrans" … ディスク上のディレクトリ名。
  • .long_segment_names = false … セグメントファイル名は短い形式（4桁16進）。
  • .sync_handler = SYNC_HANDLER_NONE … チェックポイント時の fsync 管理対象外。
  • .PagePrecedes = SubTransPagePrecedes … 切り詰め時のページ順序判定（5.4節）。

2.2　ディスク上の pg_subtrans ディレクトリ

実体は $PGDATA/pg_subtrans/ 以下のセグメントファイルである。1 エントリは 4 バイト、1 ページは BLCKSZ（既定 8 KB）なので 1 ページに 2,048 エントリ、1 セグメントは SLRU_PAGES_PER_SEGMENT（= 32）ページ＝ 65,536 エントリを収める（数え方は第3章）。 ディレクトリの中身は次のように観察できる。

SQL 
SELECT * FROM pg_ls_dir('pg_subtrans') ORDER BY 1;
実行結果（例） 
 pg_ls_dir
-----------
 0000
(1 row)

稼働初期はセグメント 0000 だけだが、XID が進むにつれて新しいセグメントが追加され、不要になった 古いセグメントはチェックポイント時に削除される（5.3節）。

2.3　クラッシュをまたがない設計

pg_subtrans の最大の特徴は、クラッシュ耐性を持たないことである。 subtrans.c のコメントが明記するとおり、XLOG（WAL）との連携は一切ない。サーバ起動時には、現在 アクティブなページを単純にゼロで初期化する（StartupSUBTRANS()、5.1節）。

これは「pg_subtrans が必要なのは現在実行中のトランザクションについてだけ」という性質 （1.3節）から導かれる。クラッシュ後に再起動すれば、それ以前に実行中だった トランザクションはもう存在しないため、古い内容を保存する意味がない。チェックポイントでダーティページを書き出す CheckPointSUBTRANS() も、正当性のためではなく、書き込み負荷をバックエンドからチェックポイント プロセスへ移すための最適化にすぎない。

注意この非永続性ゆえ、ベースバックアップやリカバリで pg_subtrans の 内容が引き継がれることはない。ストリーミングレプリケーションのスタンバイでは、プライマリから受け取った 「XID 割り当てレコード」を適用して SubTransSetParent() を呼び、必要な親子関係を再構築する （4.4節）。
