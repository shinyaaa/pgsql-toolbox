第4章　親の記録と最上位XIDの解決

本章は pg_subtrans の中心となる 3 つの関数 — 親を記録する SubTransSetParent()、親を 引く SubTransGetParent()、最上位の親までさかのぼる SubTransGetTopmostTransaction() — を コードに即して読み解き、最後に呼び出し元である xact.c との連携を見る。

4.1　SubTransSetParent — 親の記録

サブトランザクションの XID とその親 XID を受け取り、対応するエントリへ親を書き込む。図 4.1 の手順で、 バンクロックを排他取得し、ページを読み込み、エントリへ親を書いてダーティに印を付ける。

書き込みは冪等になっている。すでに同じ親が入っていれば何もしない。コメントが警告するとおり、ある有効な親 XID から別の有効な親 XID へ書き換えることは決して起きてはならない（起きればデータ構造の破損 である）。そのため書き込み前のエントリは InvalidTransactionId であることがアサートされる。

subtrans.c（抜粋） 
if (*ptr != parent)
{
    Assert(*ptr == InvalidTransactionId);
    *ptr = parent;
    SubTransCtl->shared->page_dirty[slotno] = true;
}

4.2　SubTransGetParent — 親の参照

ある XID の親を 1 段だけ引く。ブートストラップXIDや凍結XIDなど「通常でない」XID には親がないので InvalidTransactionId を返す。読み取りは SimpleLruReadPage_ReadOnly() を使い、これが バンクロックの取得まで面倒を見る。読み終えたらバンクロックを解放する。

先頭にある Assert(TransactionIdFollowsOrEquals(xid, TransactionXmin)) が重要である。 TransactionXmin より古い XID については、すでにセグメントが切り詰められている可能性があるため、 そもそも問い合わせてはならない（切り詰めは5.3節）。

4.3　SubTransGetTopmostTransaction — 最上位の解決

可視性判定で実際に使われるのはこの関数である。親チェーンを子から親へたどり、親が無効になるか、 あるいは TransactionXmin より前に達した時点で止め、その直前の有効な XID を「最上位」として返す。 図 4.2 にループの構造を示す。

2 つの設計上の工夫がある。第一に、TransactionXmin より前で打ち切るため、真の最上位ではなく 途中のサブXIDを返す（"嘘をつく"）ことがある。だがコメントが述べるとおり、関心があるのは 「最上位が今も実行中か、現在のスナップショットの実行中リストに含まれるか」だけなので、 TransactionXmin より前の XID はどれも同じく「十分に古い」とみなせて問題にならない。第二に、 慣例として親は子より先に割り当てられる（親XID < 子XID）。これが崩れていれば無限ループに陥る破損なので、 elog(ERROR, "pg_subtrans contains invalid entry ...") で即座に打ち切る。

4.4　xact.c との連携

親子関係を実際に書き込む主な呼び出し元は、トランザクションへ XID を割り当てる AssignTransactionId()（src/backend/access/transam/xact.c）である。新しい FullTransactionId を採番した直後、サブトランザクションであれば SubTransSetParent() を 呼ぶ。

xact.c（抜粋） 
s->fullTransactionId = GetNewTransactionId(isSubXact);
...
if (isSubXact)
    SubTransSetParent(XidFromFullTransactionId(s->fullTransactionId),
                      XidFromFullTransactionId(s->parent->fullTransactionId));

この順序には理由がある。コメントいわく「subtrans エントリは、その XID が PGPROC 以外の共有領域に 現れる前に作らねばならない。なぜなら PGPROC に空きがない（オーバーフロー）場合、 他のバックエンドがその XID を『実行中』と認識するために subtrans エントリが必要だからである」。つまり pg_subtrans は、PGPROC のサブXIDキャッシュが溢れたときのバックアップ経路として 機能する（6.1節）。

関連リカバリ中は、プライマリの「XID 割り当てレコード」を ProcArrayApplyXidAssignment()（procarray.c）が適用し、各サブXIDについて SubTransSetParent() を呼んで親子関係を再構築する。2相コミット（twophase.c）でも、 準備済みトランザクションの復元時に同様の設定を行う。
