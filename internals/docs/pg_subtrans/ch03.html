第3章　データ構造とページレイアウト

本章では pg_subtrans のデータ構造を、ページレイアウトと XID からの写像、そして並行アクセスを 保護するロックの観点から詳しく見る。値はすべて src/backend/access/transam/subtrans.c の定義に基づく。

3.1　エントリとページ

pg_subtrans の 1 エントリは TransactionId（4 バイト）で、ある XID の親XIDを 表す。XID をそのまま配列の添字とみなし、エントリ番号 i の中身が「XID i の親」になる、と 考えればよい。図 3.1 にページの内部を示す。

関連する定義は次のとおりである。

定義値意味
SUBTRANS_XACTS_PER_PAGEBLCKSZ / sizeof(TransactionId) = 8192 / 4 = 20481 ページあたりのエントリ数
SLRU_PAGES_PER_SEGMENT321 セグメントのページ数
1 セグメントのエントリ数2048 × 32 = 65,5361 セグメントファイルがカバーする XID 数

3.2　XID からページ・エントリへの写像

ある XID のエントリがどのページのどの位置にあるかは、単純な除算と剰余で求まる （TransactionIdToPage() と TransactionIdToEntry()）。

subtrans.c 
#define SUBTRANS_XACTS_PER_PAGE (BLCKSZ / sizeof(TransactionId))

/* ページ番号 = xid / 2048 */
static inline int64
TransactionIdToPage(TransactionId xid)
{
    return xid / (int64) SUBTRANS_XACTS_PER_PAGE;
}

/* エントリ番号 = xid % 2048 */
#define TransactionIdToEntry(xid) ((xid) % (TransactionId) SUBTRANS_XACTS_PER_PAGE)

図 3.2 に、XID = 5000 を例にとった写像を示す。ページ番号は 5000 / 2048 = 2、エントリ番号は 5000 % 2048 = 904 となる。XID は 32 ビットで 0xFFFFFFFF を超えると 0 へ回り込むため、 ページ番号も 0xFFFFFFFF / 2048 でラップする（5.4節）。

3.3　バンクとロック、バッファ数

共有メモリ上のページバッファは複数のバンク（bank）に分割され、各バンクは専用の LWLock で保護される。あるページを操作する際は、SimpleLruGetBankLock(SubTransCtl, pageno) でそのページが属するバンクのロックを取得する。書き込み系（SubTransSetParent）は LW_EXCLUSIVE で明示的に取得し、読み取り系は SimpleLruReadPage_ReadOnly() が内部で ロックを取得・解放する（第4章）。

バッファ（スロット）数は GUC subtransaction_buffers で決まる。0 なら自動チューニングで、 共有バッファ 1 GB あたり 2 MB、上限 8 MB を割り当てる（SimpleLruAutotuneBuffers(512, 1024)）。明示 指定時は [16, SLRU_MAX_ALLOWED_BUFFERS] の範囲にクランプされる。

SQL 
SHOW subtransaction_buffers;
SELECT name, blks_zeroed, blks_hit, blks_read, blks_written
FROM pg_stat_slru WHERE name = 'subtransaction';
実行結果（例） 
 subtransaction_buffers
------------------------
 0

      name      | blks_zeroed | blks_hit | blks_read | blks_written
----------------+-------------+----------+-----------+--------------
 subtransaction |           1 |        0 |         0 |            0

blks_read（ディスクから読み込んだページ数）は、サブトランザクションのオーバーフローが起きて pg_subtrans が実際に参照されると増えていく。この観察は6.3節で運用上の 指標として再登場する。
