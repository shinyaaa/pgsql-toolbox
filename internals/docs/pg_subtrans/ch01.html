第1章　サブトランザクションと親子関係

PostgreSQL のトランザクションは入れ子（ネスト）にできる。トップレベルのトランザクションの内部に サブトランザクション（subtransaction）を作り、その一部だけを個別に取り消せる。本章では サブトランザクションの概念と、PostgreSQL がなぜ各トランザクションの親を記録する必要があるのか を整理し、pg_subtrans が解決する問題を明らかにする。

1.1　サブトランザクションとは

サブトランザクションは SAVEPOINT コマンドや PL/pgSQL の BEGIN ... EXCEPTION ... END ブロックによって生成される、トップレベルトランザクション内部の入れ子トランザクションである。書き込みを行った サブトランザクションには、トップレベルと同様に独自のトランザクションID（XID）が割り当てられる。

各トランザクションは「親」を1つ持つ。トップレベルトランザクションの親は無効値 InvalidTransactionId であり、各サブトランザクションの親は直近の親トランザクションの XID である。図 1.1 に示すように、この親子関係は木構造をなす。子から親へは容易にたどれるが、逆方向（親から子）には たどれない点が後の設計に効いてくる。

subtrans.c 冒頭のコメントが、この役割を端的に述べている。すなわち pg_subtrans は 「各トランザクションについて親トランザクションIDを保存する、pg_xact 風のマネージャであり、入れ子 トランザクション実装の根幹をなす」ものである。

1.2　SAVEPOINT による入れ子の例

具体的に、セーブポイントでサブトランザクションが生まれる様子を見る。次の例では、各 SAVEPOINT 以降の書き込みがそれぞれ別のサブトランザクションに属する。

SQL 
BEGIN;
INSERT INTO accounts VALUES (1, 100);   -- トップレベルのXIDが割り当てられる
SAVEPOINT s1;
INSERT INTO accounts VALUES (2, 200);   -- サブトランザクション s1
SAVEPOINT s2;
INSERT INTO accounts VALUES (3, 300);   -- サブトランザクション s2
ROLLBACK TO SAVEPOINT s1;               -- s1 以降（s2 含む）を取り消す
INSERT INTO accounts VALUES (4, 400);   -- s1 を巻き戻した後の新たなサブトランザクション
COMMIT;

ROLLBACK TO SAVEPOINT s1 は、s1 を設定した時点以降の変更だけを取り消し、トップレベルや それ以前の変更は保持する。サブトランザクションごとに XID を分けて管理することで、このような部分的な巻き戻しが 可能になる。

関連PL/pgSQL の EXCEPTION 句を持つブロックも、内部的には サブトランザクションを開始する。例外ハンドラで処理を継続できるのは、ブロック内の変更をサブトランザクション 単位で巻き戻せるからである。この点は第6章の性能の議論につながる。

1.3　親子関係を記録する理由

では、なぜ親子関係を共有領域に記録する必要があるのか。鍵となるのは他のバックエンドからの可視性判定 である。あるバックエンドがタプルに刻まれた XID（それがサブトランザクションのものかもしれない）を見たとき、 「この XID の最上位の親は誰か」「その親はまだ実行中か」を判定できなければならない。サブ トランザクションの可視性は、その最上位の親トランザクションの状態に従うからである。

各バックエンドは自分のサブXIDを共有メモリ上の PGPROC に最大 PGPROC_MAX_CACHED_SUBXIDS（= 64、src/include/storage/proc.h）個までキャッシュ する。しかしサブトランザクションが 64 を超えるとキャッシュは溢れ（overflow）、個々のサブXIDは 共有スナップショットからは見えなくなる。この溢れたケースで、子XID→親XIDの写像を全バックエンドへ提供するのが pg_subtrans である（詳細は6.1節）。

メモpg_subtrans が必要なのは現在実行中の トランザクションについてだけである。subtrans.c のコメントが明言するとおり、堅牢性要件は pg_xact（CLOG）とは全く異なり、データをクラッシュ・再起動をまたいで保存する必要はない。この 割り切りが、次章で見る軽量な設計を可能にしている。
