第6章　スナップショット・可視性での利用と運用

最終章では、pg_subtrans が実運用でいつ・どのように参照されるかを、サブXIDキャッシュの オーバーフローを起点に追い、可視性判定での利用と、性能・運用上の注意点をまとめる。

6.1　subxid キャッシュの溢れ（オーバーフロー）

1.3節で触れたとおり、各バックエンドは自分のサブXIDを共有メモリ上の PGPROC に最大 PGPROC_MAX_CACHED_SUBXIDS（= 64）個までキャッシュする （subxidStatus: nxids と overflowed）。図 6.1 のように、65 個目以降は overflowed = true となり、個々のサブXIDは共有スナップショットに載らなくなる。

このとき、他バックエンドがそのサブXIDの状態を知る唯一の手段が pg_subtrans である。 4.4節で見たように、AssignTransactionId() が XID を共有領域に出す前に 必ず親を記録しておくのは、まさにこのオーバーフロー時の参照に備えるためである。

6.2　可視性判定での参照

スナップショット取得時（GetSnapshotData()）に、いずれかのバックエンドがオーバーフローして いれば、スナップショットに suboverflowed 印が付く。以後、その印が付いたスナップショットで サブXIDの可視性を判定する XidInMVCCSnapshot()（src/backend/utils/time/snapmgr.c）は、 図 6.2 のように SubTransGetTopmostTransaction() で最上位XIDを求め、それをスナップショットの 実行中リストと照合する。

同様に、トランザクション完了を待つロック機構 XactLockTableWait() （src/backend/storage/lmgr/lmgr.c）や、ヒープの可視性・待機処理 （src/backend/access/heap/heapam.c）も、サブXIDをいったん最上位XIDへ写してから扱う。いずれも 「サブトランザクションの状態は最上位の親に従う」という原則の現れである。

6.3　性能と運用上の注意

pg_subtrans は普段は静かだが、次の条件が重なると性能問題の原因になりうる。

  • 1 トランザクションあたりのサブトランザクションが 64 を超える（深い／多数の SAVEPOINT、 PL/pgSQL の EXCEPTION ブロックの多用）。
  • 同時に長時間トランザクションが存在し、TransactionXmin が進まない。

これらが重なると、読み取り側が頻繁に pg_subtrans を引き、SLRU のバンクロック競合やディスク I/O （待機イベント SubtransSLRU / SubtransBuffer）が顕在化する。次の PL/pgSQL は、 EXCEPTION ブロックが 1 つのサブトランザクションを生むことを利用して、意図的にオーバーフロー （> 64 サブトランザクション）を起こす例である。

SQL 
CREATE TABLE t (id int);
BEGIN;
DO $$
BEGIN
  FOR i IN 1..100 LOOP
    BEGIN
      INSERT INTO t VALUES (i);      -- EXCEPTION 句を持つブロック = サブトランザクション
    EXCEPTION WHEN others THEN NULL;
    END;
  END LOOP;
END $$;
-- この時点でこのバックエンドの subxid キャッシュはオーバーフローしている。
-- 別セッションから t を読むと、可視性判定で pg_subtrans が参照され blks_read が増える。
COMMIT;

影響は pg_stat_slru と pg_stat_activity で観察・診断できる。

SQL 
-- SLRU の読み込み・ヒット状況（subtransaction 行に注目）
SELECT name, blks_zeroed, blks_hit, blks_read FROM pg_stat_slru
WHERE name = 'subtransaction';

-- どのセッションが Subtrans 系の待機をしているか
SELECT pid, wait_event_type, wait_event, state
FROM pg_stat_activity
WHERE wait_event LIKE 'Subtrans%';

緩和策としては、(1) subtransaction_buffers を増やして SLRU のバッファを拡張する、 (2) 長時間トランザクションを避けて TransactionXmin を進める、(3) 1 トランザクション内の サブトランザクション数を 64 以下に抑える（不要な SAVEPOINT や EXCEPTION の濫用を避ける）、 が挙げられる。

まとめpg_subtrans は、サブトランザクションの親子関係を SLRU 上に 記録する軽量・非永続のマネージャである。普段は PGPROC のサブXIDキャッシュで足りるが、 キャッシュが溢れたときに「子XID→最上位の親XID」を全バックエンドへ提供する最後の拠り所として働く。クラッシュを またいで保存しないという割り切りが、WAL 連携を持たない簡潔な設計を可能にしている。
