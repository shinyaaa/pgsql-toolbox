第7章　バックエンドによるサブトランザクション状態判定

あるバックエンドが、タプルの xmin や xmax に記録されたサブXIDについて 「コミット済みか・実行中か・中止済みか」を決める一連の処理を、ソースコードを参照しながら順を追って解説する。 pg_subtrans が登場するのはこの判定の中の特定のステップに限られており、その手前でいくつかの 高速パスが設けられている。

7.1　エントリポイントと全体像

ヒープタプルの可視性判定は HeapTupleSatisfiesMVCC() （src/backend/access/heap/heapam_visibility.c）が行う。内部で xmin の状態を調べる 際に実行する処理の全体像を図 7.1 に示す。大きく 5 段階あり、早いステップで答えが出るほどコストが低い。 pg_subtrans の SubTransGetTopmostTransaction() が呼ばれるのは③と④の一部に すぎない。

注意図 7.1 は概念的な順序を示す。実際の HeapTupleSatisfiesMVCC() は MVCC スナップショット引数を受け取り、ステップ③（TransactionIdIsInProgress）は HeapTupleSatisfiesUpdate() など別のバリアントで使われる。MVCC スナップショットを使う HeapTupleSatisfiesMVCC() ではステップ④（XidInMVCCSnapshot）を中心に判定する。

7.2　ステップ①：ヒントビット高速パス

タプルヘッダの t_infomask には、過去の判定結果を再利用するためのヒントビットが格納される。

ビット意味
HEAP_XMIN_COMMITTEDxmin がコミット済みであると確定
HEAP_XMIN_INVALIDxmin が無効（アボートまたはクラッシュ）と確定

いずれかがセットされていれば CLOG や PGPROC 配列を一切参照せずに即座に返れる。これが最も高速なパスで あり、ヒートアップしたワークロードでは大多数のタプルがここで判定される。

heapam_visibility.c（抜粋） 
if (!HeapTupleHeaderXminCommitted(tuple))
{
    if (HeapTupleHeaderXminInvalid(tuple))
        return false;   /* 不可視 */
    /* ヒントビットなし → 以降のステップへ */
    ...
}

7.3　ステップ②：自バックエンド判定（TransactionIdIsCurrentTransactionId）

TransactionIdIsCurrentTransactionId()（xact.c）は、問い合わせ対象の XID が 現在のバックエンドのトランザクション（あるいはそのサブトランザクション）かどうかを判定する。 共有メモリを一切触らないローカル処理である。

内部では図 7.2 の順に探索する。

childXids[] には、すでに SAVEPOINT を越えて subcommit されたサブXIDが昇順で格納される。 これを二分探索するため、深くネストした SAVEPOINT でも高速に判定できる。ただし最大 PGPROC_MAX_CACHED_SUBXIDS（= 64）を超えた分はここには入らず、pg_subtrans による 解決が必要になる（後述）。

7.4　ステップ③：他バックエンドの PGPROC 探索（TransactionIdIsInProgress）

TransactionIdIsInProgress()（src/backend/storage/ipc/procarray.c）は、XID が 他のバックエンドで現在実行中かどうかを調べる。図 7.3 のように最大 4 段階の探索を行い、 早い段階で答えが出れば後の探索はスキップする。

重要な設計上の点が 2 つある。

  • subxid キャッシュ（Step 2）に入りきらなかった分は、overflow フラグが立ったバックエンドの トップXIDを収集しておき、ロック解放後に pg_subtrans で最上位XIDまでさかのぼって 突き合わせる（Step 4）。これが pg_subtrans の主要な呼ばれ方である。
  • Step 4 の前に必ず pg_xact（CLOG）でアボート確認する。すでにアボートしたサブ トランザクションの XID は "実行中でない" と答えるべきで、親が実行中でも false を返す必要がある。
procarray.c（Step 4 の抜粋） 
/* Step 4: overflow ありの場合のみ到達 */
if (TransactionIdDidAbort(xid))
{
    cachedXidIsNotInProgress = xid;
    return false;    /* すでにアボート済み */
}

topxid = SubTransGetTopmostTransaction(xid);  /* pg_subtrans を参照 */
if (!TransactionIdEquals(topxid, xid) &&
    pg_lfind32(topxid, xids, nxids))
    return true;    /* 最上位XIDが実行中 */

cachedXidIsNotInProgress = xid;
return false;

7.5　ステップ④：スナップショット照合（XidInMVCCSnapshot）

MVCC スナップショットを使う可視性判定では、XidInMVCCSnapshot()（snapmgr.c）が 「スナップショット取得時点でその XID が実行中だったか」を返す。これは③とは異なり、現在の状態でなく スナップショット取得時点の状態を問う。

図 7.4 にその処理を示す。オーバーフローの有無によって参照する配列が分岐するのがポイントである。

通常（非リカバリ）のオーバーフローなしのケースが最速で、subxip[] に対する線形探索 （実装は pg_lfind32()、SIMD を活用）だけで完結する。

subxip[] にはスナップショット取得時点で各バックエンドが PGPROC にキャッシュしていた サブXIDがすべて格納される（最大 TOTAL_MAX_CACHED_SUBXIDS）。この配列はスナップショット ごとにコピーされるため、スナップショット取得後に他バックエンドがコミット・アボートしても影響を 受けない。

7.6　ステップ⑤：CLOG 確認とヒントビット更新

①〜④で答えが出なかった場合（ヒントビットなし・自分でもなく・現在実行中でもなく・スナップショット時点でも 実行中でない）、コミットログ（pg_xact、CLOG）を参照する。

heapam_visibility.c（抜粋） 
else if (TransactionIdDidCommit(HeapTupleHeaderGetRawXmin(tuple)))
    SetHintBits(tuple, buffer, HEAP_XMIN_COMMITTED,
                HeapTupleHeaderGetRawXmin(tuple));
else
{
    /* アボートまたはクラッシュ */
    SetHintBits(tuple, buffer, HEAP_XMIN_INVALID,
                InvalidTransactionId);
    return false;
}

TransactionIdDidCommit() は内部で pg_xact の SLRU バッファを参照し、コミット状態 ビットを確認する。CLOG ページがバッファにない場合はディスクから読み込む（待機イベント CLogBuffer）。

判定後に SetHintBits() でヒントビットを書き込むことで、次回同じタプルにアクセスした際には ①の高速パスで即座に返れる。ヒントビットの書き込みは WAL を出さない（クラッシュ後はヒントビットが 消えるが、再度ステップを踏めばよい）。

7.7　pg_subtrans が呼ばれる条件のまとめ

以上をまとめると、SubTransGetTopmostTransaction() が実際に呼ばれる条件は以下の 2 つに 限定される。

呼び出し元条件
TransactionIdIsInProgress() ステップ4 いずれかのバックエンドの subxid キャッシュがオーバーフローしており、かつ PGPROC の トップXIDおよびキャッシュ内サブXID どちらにも対象 XID が見つからなかった
XidInMVCCSnapshot() スナップショット取得時点でいずれかのバックエンドが overflow しており、スナップショットの suboverflowed フラグが立っている

逆に言えば、サブトランザクションの数が全バックエンドにわたって合計 TOTAL_MAX_CACHED_SUBXIDS（= PGPROC_MAX_CACHED_SUBXIDS × 最大バックエンド数）以内に 収まっている間は、pg_subtrans は一度も参照されない。オーバーフローは 「1 つのバックエンドが 64 個を超えるサブトランザクションを生む」ことで発生する （6.1節）。

まとめバックエンドがサブXIDの状態を判定するとき、まずヒントビット・ローカルスタック・ PGPROC キャッシュという共有メモリロック不要ないし軽量ロックで完結する高速パスを順に試みる。 これらをすべて通過してはじめて pg_subtrans の SubTransGetTopmostTransaction() が呼ばれ、SLRU バッファ上の親XIDチェーンをたどる。pg_subtrans の参照はオーバーフロー時の 例外的なパスであり、通常ワークロードでは現れない。
