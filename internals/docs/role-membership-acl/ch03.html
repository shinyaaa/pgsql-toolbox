第3章　メンバーシップキャッシュの仕組み

メンバーシップの展開は syscache 検索を伴う再帰であり、権限チェックのたびに実行するには重い。 そのため acl.c はプロセスローカルなキャッシュを持っている。ただしそれは 「モードごとに 1 スロット」という極端に小さいもので、有効期間も強く限定されている。 本章ではこのキャッシュの構造、無効化の仕組み、そして大規模なロール階層に備えた ブルームフィルタによる高速化を読む。

3.1　cached_role / cached_roles の構造

キャッシュの実体は 2 本のファイルスコープ配列である。要素数 3 は RoleRecurseType の値と対応している。

src/backend/utils/adt/acl.c:82 
static Oid  cached_role[] = {InvalidOid, InvalidOid, InvalidOid};
static List *cached_roles[] = {NIL, NIL, NIL};
static uint32 cached_db_hash;

つまり、探索モードごとに「直前に問い合わせたロール 1 件分」の結果しか保持しない。 図 3.1 にその構造を示す。

キャッシュ利用の可否は関数冒頭の 1 行で決まる。

src/backend/utils/adt/acl.c:5199 
    /* If cache is valid and ADMIN OPTION not sought, just return the list */
    if (cached_role[type] == roleid && !OidIsValid(admin_of) &&
        OidIsValid(cached_role[type]))
        return cached_roles[type];

admin_of が指定されている場合は、たとえ同じロールの結果が残っていても キャッシュを使わない。*admin_role は展開の副産物としてしか得られず、 キャッシュされたリストからは復元できないためである。図 3.1 に示した「再利用」の枝には この条件が加わる。

展開結果は TopMemoryContext にコピーされ、プロセスが生きている限り残る。 代入順序にも注意が払われている。

src/backend/utils/adt/acl.c:5287 
    /*
     * Copy the completed list into TopMemoryContext so it will persist.
     */
    oldctx = MemoryContextSwitchTo(TopMemoryContext);
    new_cached_roles = list_copy(roles_list);
    MemoryContextSwitchTo(oldctx);
    list_free(roles_list);

    /*
     * Now safe to assign to state variable
     */
    cached_role[type] = InvalidOid; /* just paranoia */
    list_free(cached_roles[type]);
    cached_roles[type] = new_cached_roles;
    cached_role[type] = roleid;

cached_role[type] を先に無効化してから古いリストを解放し、最後に有効な OID を書き戻す。 途中でエラーが発生してもダングリングポインタを指した状態のキャッシュが残らないようにするための順序である。

注意関数ヘッダが明言しているとおり、返される List * は 次に roles_is_member_of() が呼ばれるまでしか有効でない （"the result is only guaranteed good until the next call of roles_is_member_of()!"）。 呼び出し側はリストを保持してはならない。select_best_grantor() がリストを ループ中ずっと保持できるのは、ループ内で呼ぶ aclmask_direct() が メンバーシップを一切問い合わせないことを確認したうえでの意図的な設計である （6.3）。

3.2　initialize_acl によるコールバック登録

キャッシュが古くなる契機は 3 つある。pg_auth_members の変更、 pg_authid の変更、そして接続中データベースの所有者変更である。 これらを捕捉するためのコールバック登録が initialize_acl() である。

src/backend/utils/adt/acl.c:5071 
void
initialize_acl(void)
{
    if (!IsBootstrapProcessingMode())
    {
        cached_db_hash =
            GetSysCacheHashValue1(DATABASEOID,
                                  ObjectIdGetDatum(MyDatabaseId));

        /*
         * In normal mode, set a callback on any syscache invalidation of rows
         * of pg_auth_members (for roles_is_member_of()) pg_database (for
         * roles_is_member_of())
         */
        CacheRegisterSyscacheCallback(AUTHMEMROLEMEM,
                                      RoleMembershipCacheCallback,
                                      (Datum) 0);
        CacheRegisterSyscacheCallback(AUTHOID,
                                      RoleMembershipCacheCallback,
                                      (Datum) 0);
        CacheRegisterSyscacheCallback(DATABASEOID,
                                      RoleMembershipCacheCallback,
                                      (Datum) 0);
    }
}

ここで cached_db_hash に自データベースの syscache ハッシュ値を保存している点が要である。 pg_database は共有カタログなので、他データベースの行が更新されても 無効化メッセージが飛んでくる。それを後段で選り分けるための値である。

この関数は InitPostgres の中から、リレーションキャッシュの初期化直後に呼ばれる。

src/backend/utils/init/postinit.c:1229 
    /* set up ACL framework (so CheckMyDatabase can check permissions) */
    initialize_acl();

直後に CheckMyDatabase() が接続権限を確認する。つまり、バックエンドが最初の 権限チェックを行う直前が、ACL フレームワークの初期化タイミングということになる。 図 3.2 にこの順序を示す。

3.3　RoleMembershipCacheCallback による無効化

無効化コールバックの本体はきわめて短い。

src/backend/utils/adt/acl.c:5101 
static void
RoleMembershipCacheCallback(Datum arg, SysCacheIdentifier cacheid,
                            uint32 hashvalue)
{
    if (cacheid == DATABASEOID &&
        hashvalue != cached_db_hash &&
        hashvalue != 0)
    {
        return;                 /* ignore pg_database changes for other DBs */
    }

    /* Force membership caches to be recomputed on next use */
    cached_role[ROLERECURSE_MEMBERS] = InvalidOid;
    cached_role[ROLERECURSE_PRIVS] = InvalidOid;
    cached_role[ROLERECURSE_SETROLE] = InvalidOid;
}

読み取るべき点は 3 つある。

  1. 粒度がない。どのロールが変更されたかにかかわらず、3 モードすべてのキャッシュを 一括で捨てる。1 スロットしか持たないキャッシュに対して、選択的な無効化を実装する意味がないからである。
  2. リストは解放しない。cached_role[] に InvalidOid を 書くだけで、cached_roles[] の List はそのまま残る。コールバックは無効化メッセージ処理中に 呼ばれるため、ここでメモリ解放のような込み入った処理をしない設計になっている。 実際の解放は次回の roles_is_member_of() での上書き時に行われる（3.1）。
  3. hashvalue == 0 は無条件に受け入れる。0 は 「このキャッシュ全体を無効化せよ」を意味する特殊値であり、pg_database 由来であっても 無視してはならない。

DATABASEOID についてだけ cached_db_hash との比較でフィルタするのは、 pg_database が共有カタログであり、無関係なデータベースの ALTER DATABASE ... OWNER TO でも通知が届くためである。 pg_authid（AUTHOID）を購読しているのは、 rolsuper の変更が superuser_arg() の結果を変え、 ひいては全判定関数の早期リターンに影響しうるからである。

3.4　roles_list_append とブルームフィルタ

roles_list_append() は、探索中のリストに重複なく OID を追加するヘルパである。素朴に書けば list_append_unique_oid() で済むが、それは線形探索なので、 リストが大きくなると展開全体が O(n²) になる。これを避けるためブルームフィルタを併用する。

src/backend/utils/adt/acl.c:5118 
/*
 * A helper function for roles_is_member_of() that provides an optimized
 * implementation of list_append_unique_oid() via a Bloom filter.  The caller
 * (i.e., roles_is_member_of()) is responsible for freeing bf once it is done
 * using this function.
 */

判定の核心は次の条件式である。ブルームフィルタの「偽陽性はあるが偽陰性はない」という性質を そのまま利用している。

src/backend/utils/adt/acl.c:5137 
    if ((*bf && bloom_lacks_element(*bf, roleptr, sizeof(Oid))) ||
        !list_member_oid(roles_list, role))
    {

bloom_lacks_element() が真を返せば「確実に未登録」なので、リスト走査を省いて即座に追加できる。偽を返した場合は 偽陽性の可能性があるため、list_member_oid() による線形探索にフォールバックする。 C の || は短絡評価なので、フィルタが効いているときは線形探索が実行されない。 この流れを図 3.3 に示す。

フィルタは最初から作られるわけではない。リスト長がしきい値を超えて初めて作成され、 そのとき既存要素がまとめて投入される。

src/backend/utils/adt/acl.c:5140 
        /*
         * If the list is large, we take on the overhead of creating and
         * populating a Bloom filter to speed up future calls to this
         * function.
         */
        if (*bf == NULL &&
            list_length(roles_list) > ROLES_LIST_BLOOM_THRESHOLD)
        {
            *bf = bloom_create(ROLES_LIST_BLOOM_THRESHOLD * 10, work_mem, 0);
            foreach_oid(roleid, roles_list)
                bloom_add_element(*bf, (unsigned char *) &roleid, sizeof(Oid));
        }

しきい値の値と、その設定理由はソースに明記されている。

src/backend/utils/adt/acl.c:86 
/*
 * If the list of roles gathered by roles_is_member_of() grows larger than the
 * below threshold, a Bloom filter is created to speed up list membership
 * checks.  This threshold is set arbitrarily high to avoid the overhead of
 * creating the Bloom filter until it seems likely to provide a net benefit.
 */
#define ROLES_LIST_BLOOM_THRESHOLD 1024

bloom_create() に渡す想定要素数はしきい値の 10 倍（10240）である。 実際のロール数がこれを大きく超えると偽陽性率が上がり、線形探索へのフォールバックが増えるが、 正しさは損なわれない。フィルタの寿命は roles_is_member_of() の 1 回の呼び出し内に 限られ、展開が終わった時点で解放される。

src/backend/utils/adt/acl.c:5281 
    /*
     * Free the Bloom filter created by roles_list_append(), if there is one.
     */
    if (bf)
        bloom_free(bf);
メモ1024 という値は、ほとんどの環境ではフィルタが一度も作られないことを 意味する。この最適化が効くのは、数千ロールが 1 人のユーザに集約されるような大規模なマルチテナント構成である。
