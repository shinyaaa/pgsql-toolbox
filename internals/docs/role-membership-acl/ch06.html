第6章　GRANT の付与者選択 select_best_grantor

GRANT SELECT ON t TO alice; を実行したとき、pg_class.relacl の ai_grantor には何が記録されるのだろうか。「実行したロール」と答えたくなるが、 実際にはそうとは限らない。本章では select_best_grantor() を読み、なぜ付与者を選び直す必要があるのかを理解する。

6.1　なぜ付与者を選び直すのか

ACL は単なる権限の集合ではなく、権限の由来を辿れる有向グラフである。 REVOKE ... CASCADE が連鎖的に権限を剥がせるのは、各エントリが ai_grantor を持っていて、依存関係を再構成できるからである。 この構造を壊さないための不変条件が、関数ヘッダに明記されている。

src/backend/utils/adt/acl.c:5484 
/*
 * Select the effective grantor ID for a GRANT or REVOKE operation.
 *
 * If the GRANT/REVOKE has an explicit GRANTED BY clause, we always use
 * exactly that role (which may result in granting/revoking no privileges).
 * Otherwise, we seek a "best" grantor, starting with the current user.
 *
 * The grantor must always be either the object owner or some role that has
 * been explicitly granted grant options.  This ensures that all granted
 * privileges appear to flow from the object owner, and there are never
 * multiple "original sources" of a privilege.  Therefore, if the would-be
 * grantor is a member of a role that has the needed grant options, we have
 * to do the grant as that role instead.
 *
 * It is possible that the would-be grantor is a member of several roles
 * that have different subsets of the desired grant options, but no one
 * role has 'em all.  In this case we pick a role with the largest number
 * of desired options.  Ties are broken in favor of closer ancestors.
 */

要点は「すべての権限がオブジェクト所有者から流れ出てくるように見えなければならない （all granted privileges appear to flow from the object owner）」という一文である。 実行者そのものを付与者として記録してしまうと、権限グラフに複数の「源」ができ、 REVOKE の連鎖が正しく働かなくなる。

図 6.1 の状況を実際に作ってみる。

SQL 
CREATE ROLE owner_r;
CREATE ROLE grantor_r;
CREATE ROLE user_r LOGIN;
CREATE ROLE recv_r;
GRANT grantor_r TO user_r;          -- 既定で INHERIT TRUE
GRANT CREATE ON SCHEMA public TO owner_r;

SET ROLE owner_r;
CREATE TABLE gtest (id int);
GRANT SELECT ON gtest TO grantor_r WITH GRANT OPTION;
RESET ROLE;
SQL（user_r で接続して実行） 
GRANT SELECT ON gtest TO recv_r;
実行結果（記録された付与者を確認） 
SELECT pg_get_userbyid(grantee) AS grantee,
       pg_get_userbyid(grantor) AS grantor,
       privilege_type, is_grantable
FROM   pg_class, aclexplode(relacl)
WHERE  relname = 'gtest'
ORDER  BY 1;

  grantee  |  grantor  | privilege_type | is_grantable
-----------+-----------+----------------+--------------
 grantor_r | owner_r   | SELECT         | t
 owner_r   | owner_r   | SELECT         | f
 owner_r   | owner_r   | UPDATE         | f
 ...
 recv_r    | grantor_r | SELECT         | f
(10 rows)

最終行の付与者は user_r ではなく grantor_r である。 結果として、権限は owner_r → grantor_r → recv_r という一本の連鎖になり、 owner_r が grantor_r から REVOKE ... CASCADE すれば recv_r の権限も正しく失われる。

6.2　GRANTED BY 指定時の処理

GRANTED BY 句がある場合、探索は行われず指定されたロールがそのまま使われる。 ただし、実行者がそのロールの権限を継承していることは確認される。

src/backend/utils/adt/acl.c:5521 
    /*
     * If we have GRANTED BY, resolve it and verify current user is allowed to
     * specify that role.
     */
    if (grantedBy)
    {
        Oid         grantor = get_rolespec_oid(grantedBy, false);

        if (!has_privs_of_role(roleId, grantor))
            ereport(ERROR,
                    (errcode(ERRCODE_INSUFFICIENT_PRIVILEGE),
                     errmsg("must inherit privileges of role \"%s\"",
                            GetUserNameFromId(grantor, false))));
        /* Use exactly that grantor, whether it has privileges or not */
        *grantorId = grantor;
        *grantOptions = aclmask_direct(acl, grantor, ownerId,
                                       needed_goptions, ACLMASK_ALL);
        return;
    }

コメントの "whether it has privileges or not" が示すとおり、指定されたロールが 必要なグラントオプションを持っていなくてもエラーにはならない。その場合 *grantOptions が 0 になり、結果として何も付与されない （ヘッダの "which may result in granting/revoking no privileges"）。 これは REVOKE で重要になる。存在しない付与に対する REVOKE は 黙って何もしないのが正しい振る舞いだからである。

次に、探索を省ける単純なケースが処理される。

src/backend/utils/adt/acl.c:5541 
    /*
     * The object owner is always treated as having all grant options, so if
     * roleId is the owner it's easy.  Also, if roleId is a superuser it's
     * easy: superusers are implicitly members of every role, so they act as
     * the object owner.
     */
    if (roleId == ownerId || superuser_arg(roleId))
    {
        *grantorId = ownerId;
        *grantOptions = needed_goptions;
        return;
    }

スーパーユーザが GRANT を実行しても、記録される付与者はスーパーユーザ自身ではなく オブジェクトの所有者になる。これも「権限は所有者から流れ出る」という不変条件の帰結である。 4.4 で見た select_best_admin() の呼び出し元が スーパーユーザに対して BOOTSTRAP_SUPERUSERID を返すのと、狙いは同じで実現方法が異なる。

6.3　最良候補の探索とスコアリング

上記のどれにも当てはまらない場合、実行者が権限を継承しているロールを順に調べる。

src/backend/utils/adt/acl.c:5554 
    /*
     * Otherwise we have to do a careful search to see if roleId has the
     * privileges of any suitable role.  Note: we can hang onto the result of
     * roles_is_member_of() throughout this loop, because aclmask_direct()
     * doesn't query any role memberships.
     */
    roles_list = roles_is_member_of(roleId, ROLERECURSE_PRIVS,
                                    InvalidOid, NULL);

    /* initialize candidate result as default */
    *grantorId = roleId;
    *grantOptions = ACL_NO_RIGHTS;
    nrights = 0;

    foreach(l, roles_list)
    {
        Oid         otherrole = lfirst_oid(l);
        AclMode     otherprivs;

        otherprivs = aclmask_direct(acl, otherrole, ownerId,
                                    needed_goptions, ACLMASK_ALL);
        if (otherprivs == needed_goptions)
        {
            /* Found a suitable grantor */
            *grantorId = otherrole;
            *grantOptions = otherprivs;
            return;
        }

        /*
         * If it has just some of the needed privileges, remember best
         * candidate.
         */
        if (otherprivs != ACL_NO_RIGHTS)
        {
            int         nnewrights = pg_popcount64(otherprivs);

            if (nnewrights > nrights)
            {
                *grantorId = otherrole;
                *grantOptions = otherprivs;
                nrights = nnewrights;
            }
        }
    }

探索の性質を 3 点にまとめる。

  1. 完全一致で即座に打ち切る。 必要なグラントオプションをすべて持つロールが見つかれば、その場で return する。 roles_list の先頭は roleId 自身なので、実行者本人が十分な権限を 持っていればループは 1 回で終わる。
  2. 部分一致は立っているビット数で採点する。 pg_popcount64() でビット数を数え、最大のものを覚えておく。 ヘッダにある「複数のロールが必要なオプションの異なる部分集合を持つ」ケースへの対処である。
  3. 同点は先勝ちで決まる。 比較が nnewrights > nrights（>= ではない）なので、 同じスコアなら先に見つかったほうが残る。roles_is_member_of() の結果が 幅優先順であることと組み合わさり、ヘッダの "Ties are broken in favor of closer ancestors." が実現される。

つまり 2.2 で見た「幅優先であることは仕様である」という宣言は、 ここで消費されている。図 6.2 に判定の流れを示す。

コメント中の "we can hang onto the result of roles_is_member_of() throughout this loop" は、 3.1 で見たキャッシュの制約への明示的な言及である。 返された List は次の roles_is_member_of() 呼び出しで無効化されうるため、 ループ内で間接的にでもメンバーシップ判定が起きればバグになる。 aclmask() ではなく aclmask_direct() を使っているのは、 効率のためだけでなく、この安全性のためでもある。

select_best_grantor() は aclchk.c の各 ExecGrant_*() から呼ばれる。列単位の権限（aclchk.c:1705）と オブジェクト単位の権限（aclchk.c:2205 ほか）で個別に呼び出されるため、 同じ GRANT 文でも対象ごとに異なる付与者が選ばれうる。

6.4　select_best_admin との対比

select_best_grantor() がオブジェクト権限の付与者を選ぶのに対し、 select_best_admin() はロールメンバーシップの付与者を選ぶ。目的は同じでも、実装の置き場所がまったく違う。

select_best_grantor()select_best_admin()
対象オブジェクト権限（relacl ほか）ロールメンバーシップ（pg_auth_members）
「権限あり」の根拠ACL 配列上のグラントオプションビットpg_auth_members.admin_option
探索の実装場所この関数自身のループroles_is_member_of() の内部
部分一致ありうる（ビット単位でスコアリング）ない（admin_option は真偽値）
superuser の扱い関数内で処理（付与者 = オーナー）呼び出し元で処理（BOOTSTRAP_SUPERUSERID）
近さの優先スコア同点時に先勝ち最初に見つけた 1 件のみ採用

select_best_admin() で探索を roles_is_member_of() に埋め込めるのは、 admin_option が真偽値でスコアリングが不要だからである。 一方 select_best_grantor() は権限ビットごとの部分一致を扱う必要があり、 ACL 配列を見なければ判断できない。両者の構造の違いは、この一点から説明できる。
