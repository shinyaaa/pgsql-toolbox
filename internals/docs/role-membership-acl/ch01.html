第1章　ロールとメンバーシップの基礎

PostgreSQL の権限判定は、大きく二つの層に分かれている。ひとつは「あるロールが、どのロールの権限を 行使できるか」を決めるロールメンバーシップ層であり、もうひとつは「あるロールが、あるオブジェクトに 対してどの操作を許されているか」を決める ACL（Access Control List）層である。この二層は独立ではなく、 ACL の評価は必ずメンバーシップ層に問い合わせる。本章では両者の関係を整理し、以降の章で読むことになる 判定関数群の役割を先に俯瞰しておく。

1.1　ロール・メンバーシップ・権限の三層構造

PostgreSQL には「ユーザ」と「グループ」の区別がない。どちらも pg_authid に格納される単一の ロール（role）であり、ロールが別のロールの メンバーになることでグループ的な構造を作る。このメンバーシップ関係は pg_auth_members に 1 行 1 グラント（付与）として記録される。

一方、テーブルや関数といったオブジェクトに対する権限は、各カタログの relacl／proacl といった列に aclitem[] 型の配列として格納される。 ここに現れるのは「誰に、どの権限ビットを、誰が与えたか」という組であり、ロール階層の情報は含まれない。 したがって、あるセッションユーザが権限を持つかどうかを判定するには、ACL 配列の走査と、 ロールメンバーシップグラフの展開という二種類の処理を組み合わせる必要がある。図 1.1 に全体像を示す。

図 1.1 に示すように、ACL のビットマスク判定を行う aclmask() は、ACL 配列に直接一致するエントリがない場合に has_privs_of_role() を呼び、間接的なメンバーシップ経由の権限を拾い上げる。行レベルセキュリティ（RLS）も同じ has_privs_of_role() を使ってポリシーの適用対象を決める。つまり roles_is_member_of() が全体のボトルネックであり、そのためのキャッシュが用意されている（第3章）。

1.2　pg_auth_members と 3 つのオプション

pg_auth_members の定義は次のとおりである。1 行が「member が roleid の メンバーである」というグラント 1 件を表す。

src/include/catalog/pg_auth_members.h 
CATALOG(pg_auth_members,1261,AuthMemRelationId) BKI_SHARED_RELATION ...
{
    /* OID for this record (needed for dependencies) */
    Oid         oid;

    /* ID of a role */
    Oid         roleid BKI_LOOKUP(pg_authid);

    /* ID of a member of that role */
    Oid         member BKI_LOOKUP(pg_authid);

    /* who granted the membership */
    Oid         grantor BKI_DEFAULT(POSTGRES) BKI_LOOKUP(pg_authid);

    /* granted with admin option? */
    bool        admin_option BKI_DEFAULT(f);

    /* exercise privileges without SET ROLE? */
    bool        inherit_option BKI_DEFAULT(t);

    /* use SET ROLE to the target role? */
    bool        set_option BKI_DEFAULT(t);
} FormData_pg_auth_members;

注目すべきは末尾の 3 つの真偽値である。これらはグラント 1 件ごとに独立して設定でき、 それぞれ意味が異なる。

  • inherit_option: メンバーが対象ロールの権限を 自動的に行使できるか。true なら SET ROLE なしで権限が及ぶ。
  • set_option: メンバーが SET ROLE によって 対象ロールになりすませるか。
  • admin_option: メンバーが対象ロールのメンバーシップを 第三者に再付与（GRANT）できるか。
メモinherit_option と set_option がグラント単位で 持てるようになったのは PostgreSQL 16 からである。それ以前は pg_authid.rolinherit による ロール単位の設定しかなく、SET ROLE の可否はメンバーシップの有無と一致していた。

この 3 オプションが独立していることを、実際に確かめてみよう。同じ target ロールに対し、 INHERIT と SET の組み合わせを変えた 4 つのメンバーを作る。

SQL 
CREATE ROLE target;
CREATE ROLE m_both LOGIN;
CREATE ROLE m_inh  LOGIN;
CREATE ROLE m_set  LOGIN;
CREATE ROLE m_none LOGIN;

GRANT target TO m_both WITH INHERIT TRUE,  SET TRUE;
GRANT target TO m_inh  WITH INHERIT TRUE,  SET FALSE;
GRANT target TO m_set  WITH INHERIT FALSE, SET TRUE;
GRANT target TO m_none WITH INHERIT FALSE, SET FALSE;

SELECT r.rolname AS member, g.rolname AS roleid,
       am.admin_option, am.inherit_option, am.set_option
FROM   pg_auth_members am
       JOIN pg_roles r ON r.oid = am.member
       JOIN pg_roles g ON g.oid = am.roleid
WHERE  g.rolname = 'target'
ORDER  BY 1;
実行結果 
 member | roleid | admin_option | inherit_option | set_option
--------+--------+--------------+----------------+------------
 m_both | target | f            | t              | t
 m_inh  | target | f            | t              | f
 m_none | target | f            | f              | f
 m_set  | target | f            | f              | t
(4 rows)

4 つのメンバーはいずれも target のメンバーではある。しかし、実際に target の権限が及ぶかどうかは inherit_option に依存する。 target にテーブルの SELECT 権限を与えて、各メンバーから読んでみる。

SQL 
CREATE TABLE t (id int);
INSERT INTO t VALUES (1);
GRANT SELECT ON t TO target;
実行結果（各ロールで接続して SELECT count(*) FROM t;） 
m_both   : 1
m_inh    : 1
m_set    : ERROR:  permission denied for table t
m_none   : ERROR:  permission denied for table t

m_set は権限こそ自動的には及ばないが、SET ROLE によって target に なることはできる。逆に m_inh は権限が及ぶのに SET ROLE できない。

実行結果 
-- m_set で接続
=> SET ROLE target; SELECT count(*) FROM t;
 count
-------
     1
(1 row)

-- m_inh で接続
=> SET ROLE target;
ERROR:  permission denied to set role "target"

「メンバーである」「権限が及ぶ」「なりすませる」の 3 つが独立した概念であることが、 この実験から確認できる。バックエンドのコードでも、この 3 つは別々の関数として実装されている。

1.3　4 種類の判定関数とその使い分け

acl.h が公開しているロール判定関数は次の 4 つ（＋ superuser を無視する変種）である。 すべて同一の roles_is_member_of() を、異なる展開モードで呼び出す薄いラッパにすぎない。

関数展開モード問い典型的な用途
has_privs_of_role() ROLERECURSE_PRIVS 権限が自動的に及ぶか 権限チェック全般。ACL 判定・RLS ポリシー適用
member_can_set_role() ROLERECURSE_SETROLE SET ROLE できるか SET ROLE、オブジェクトの所有者変更・作成
is_member_of_role() ROLERECURSE_MEMBERS メンバーか（オプション無視） ほぼ使ってはならない（後述）
is_admin_of_role() ROLERECURSE_MEMBERS + admin_of メンバーシップを再付与できるか GRANT role TO ... の権限チェック

is_member_of_role() のコメントは異例なほど強い調子で書かれている。

src/backend/utils/adt/acl.c（is_member_of_role のコメント） 
/*
 * Is member a member of role (directly or indirectly)?
 *
 * This is defined to recurse through grants whether they are inherited or not.
 *
 * Do not use this for privilege checking, instead use has_privs_of_role().
 * Don't use it for determining whether it's possible to SET ROLE to some
 * other role; for that, use member_can_set_role(). And don't use it for
 * determining whether it's OK to create an object owned by some other role:
 * use member_can_set_role() for that, too.
 *
 * In short, calling this function is the wrong thing to do nearly everywhere.
 */

実際、PostgreSQL 20devel のバックエンドにおいて is_member_of_role() の呼び出しは pg_role_aclcheck() の 1 箇所（pg_has_role(..., 'MEMBER') の実装）しか残っていない。 inherit_option／set_option がグラント単位になった結果、 「オプションを無視したメンバーシップ」は SQL レベルの内省以外では使い道がなくなった、ということである。

関連各関数の意味論と実装の詳細は 第4章 判定関数の意味論で扱う。まず次章で、これらが共通して呼び出す roles_is_member_of() の中身を読む。
