第2章　メンバーシップ展開の中核 roles_is_member_of

roles_is_member_of() は、指定したロールが（直接・間接に）メンバーであるロールをすべて列挙する静的関数である。 第1章で見た 4 つの判定関数はいずれもこの関数のラッパであり、権限判定のほぼすべての経路が ここを通る。本章ではその探索アルゴリズムと、探索モードによる振る舞いの違いを読む。

2.1　RoleRecurseType — 3 つの探索モード

この関数の第 2 引数は enum RoleRecurseType であり、 acl.c の冒頭 で定義されている。

src/backend/utils/adt/acl.c:78 
    ROLERECURSE_MEMBERS = 0,    /* recurse unconditionally */
    ROLERECURSE_PRIVS = 1,      /* recurse through inheritable grants */
    ROLERECURSE_SETROLE = 2     /* recurse through grants with set_option */
};
static Oid  cached_role[] = {InvalidOid, InvalidOid, InvalidOid};
static List *cached_roles[] = {NIL, NIL, NIL};
static uint32 cached_db_hash;

モードの違いは、グラフを辿るときに pg_auth_members の行をどうフィルタするかだけである。 探索本体のループ内では次のように判定される。

src/backend/utils/adt/acl.c:5258 
            /* If we're supposed to ignore non-heritable grants, do so. */
            if (type == ROLERECURSE_PRIVS && !form->inherit_option)
                continue;

            /* If we're supposed to ignore non-SET grants, do so. */
            if (type == ROLERECURSE_SETROLE && !form->set_option)
                continue;

ここで重要なのは、フィルタが辺（グラント）単位で効くことである。多段の メンバーシップ連鎖では、途中の 1 本でも inherit_option = false があれば、 その先には権限が伝わらない。図 2.1 にその様子を示す。

図 2.1 の状況を実際に作って確かめる。

SQL 
CREATE ROLE lv3;
CREATE ROLE lv2;
CREATE ROLE lv1;
GRANT lv3 TO lv2;   -- 既定: INHERIT TRUE
GRANT lv2 TO lv1;   -- 既定: INHERIT TRUE

SELECT pg_has_role('lv1','lv3','USAGE')  AS privs_lv1_lv3,
       pg_has_role('lv1','lv3','MEMBER') AS member_lv1_lv3;
実行結果 
 privs_lv1_lv3 | member_lv1_lv3
---------------+----------------
 t             | t
(1 row)
SQL（中間リンクだけを非継承に張り替える） 
REVOKE lv3 FROM lv2;
GRANT  lv3 TO lv2 WITH INHERIT FALSE;

SELECT pg_has_role('lv1','lv3','USAGE')  AS privs_lv1_lv3,
       pg_has_role('lv1','lv3','MEMBER') AS member_lv1_lv3,
       pg_has_role('lv1','lv3','SET')    AS set_lv1_lv3;
実行結果 
 privs_lv1_lv3 | member_lv1_lv3 | set_lv1_lv3
---------------+----------------+-------------
 f             | t              | t
(1 row)

lv1 と lv3 の間の辺は一切変えていないにもかかわらず、 USAGE（＝has_privs_of_role）だけが f になった。 MEMBER と SET は別モードで展開されるため影響を受けていない。

2.2　自己拡張リストによる幅優先探索

探索の実装は、List を「発見済み集合」と「未処理キュー」の両方として使い回す、簡潔だがやや技巧的な ものである。ソースのコメントがその意図を明示している。

src/backend/utils/adt/acl.c:5222 
    /*
     * Find all the roles that roleid is a member of, including multi-level
     * recursion.  The role itself will always be the first element of the
     * resulting list.
     *
     * Each element of the list is scanned to see if it adds any indirect
     * memberships.  We can use a single list as both the record of
     * already-found memberships and the agenda of roles yet to be scanned.
     * This is a bit tricky but works because the foreach() macro doesn't
     * fetch the next list element until the bottom of the loop.
     */
    roles_list = list_make1_oid(roleid);

    foreach(l, roles_list)
    {

foreach() がループ末尾まで次要素を取りに行かないため、ループ本体で roles_list に要素を追加しても走査が破綻しない。追加は末尾に対して行われるので、 結果として探索は幅優先（breadth-first）になる。この順序は偶然の産物ではなく、 関数ヘッダで仕様として明記されている。

src/backend/utils/adt/acl.c:5176 
 * For the benefit of select_best_grantor, the result is defined to be
 * in breadth-first order, ie, closer relationships earlier.

「近い関係ほど先に来る」という保証は、第6章で読む select_best_grantor() や select_best_admin() が「ホップ数の少ない候補を優先する」ために使う。図 2.2 に探索の進行を示す。

図 2.2 の最後のステップに現れる重複排除について、ソースは次のように注記している。 ループそのものはカタログ上あり得ないはずだが、菱形の経路は正当に存在しうる。

src/backend/utils/adt/acl.c:5266 
            /*
             * Even though there shouldn't be any loops in the membership
             * graph, we must test for having already seen this role. It is
             * legal for instance to have both A->B and A->C->B.
             */
            roles_list = roles_list_append(roles_list, &bf, otherid);

この重複判定を担う roles_list_append() は、線形探索がボトルネックにならないようブルームフィルタを併用する。詳細は第3章で扱う。

なお、直接のメンバーシップの取得には syscache のリスト検索 SearchSysCacheList1(AUTHMEMMEMROLE, ...) を使う。 AUTHMEMMEMROLE は pg_auth_members_member_role_index （member, roleid, grantor）に対応するキャッシュであり、 「あるメンバーが所属するロール」を引く方向に最適化されている。

2.3　pg_database_owner の暗黙メンバーシップ

pg_database_owner は、pg_auth_members に行を持たない特殊な定義済みロールである。 「現在接続中のデータベースの所有者」が暗黙のメンバーになる、という規則が roles_is_member_of() の中に直接埋め込まれている。

src/backend/utils/adt/acl.c:5275 
        /* implement pg_database_owner implicit membership */
        if (memberid == dba && OidIsValid(dba))
            roles_list = roles_list_append(roles_list, &bf,
                                           ROLE_PG_DATABASE_OWNER);

dba は関数冒頭で pg_database の syscache から取得される。 ここには、データベースに接続していないバックエンドを考慮した分岐がある。

src/backend/utils/adt/acl.c:5204 
    /*
     * Role expansion happens in a non-database backend when guc.c checks
     * ROLE_PG_READ_ALL_SETTINGS for a physical walsender SHOW command.  In
     * that case, no role gets pg_database_owner.
     */
    if (!OidIsValid(MyDatabaseId))
        dba = InvalidOid;
    else
    {
        HeapTuple   dbtup;

        dbtup = SearchSysCache1(DATABASEOID, ObjectIdGetDatum(MyDatabaseId));
        if (!HeapTupleIsValid(dbtup))
            elog(ERROR, "cache lookup failed for database %u", MyDatabaseId);
        dba = ((Form_pg_database) GETSTRUCT(dbtup))->datdba;
        ReleaseSysCache(dbtup);
    }

この実装から、重要な帰結がひとつ導かれる。メンバーシップの展開結果は、接続中の データベースに依存する。同じロールでも、自分が所有するデータベースに接続しているときだけ pg_database_owner のメンバーになる。実際に確かめてみる。

SQL（postgres で実行） 
CREATE ROLE dbowner LOGIN;
CREATE DATABASE mydb OWNER dbowner;
実行結果（dbowner で mydb に接続） 
=> SELECT current_database(),
          pg_has_role('dbowner','pg_database_owner','USAGE') AS is_dbowner;
 current_database | is_dbowner
------------------+------------
 mydb             | t
(1 row)
実行結果（同じ dbowner で postgres に接続） 
=> SELECT current_database(),
          pg_has_role('dbowner','pg_database_owner','USAGE') AS is_dbowner;
 current_database | is_dbowner
------------------+------------
 postgres         | f
(1 row)
メモこの「データベース依存性」があるため、メンバーシップキャッシュは pg_database の変更にも反応しなければならない。initialize_acl() が DATABASEOID に対してもコールバックを登録している理由がこれである（3.2）。

2.4　ADMIN OPTION 保持者の同時収集

roles_is_member_of() には、単なるメンバーシップ列挙に加えてもうひとつの役割がある。 引数 admin_of に有効な OID が渡された場合、展開の過程で「そのロールに対して ADMIN OPTION を直接持つロール」を探し、最初に見つかったものを *admin_role に返す。

src/backend/utils/adt/acl.c:5179 
 * If admin_of is not InvalidOid, this function sets *admin_role, either
 * to the OID of the first role in the result list that directly possesses
 * ADMIN OPTION on the role corresponding to admin_of, or to InvalidOid if
 * there is no such role.
src/backend/utils/adt/acl.c:5250 
            /*
             * While otherid==InvalidOid shouldn't appear in the catalog, the
             * OidIsValid() avoids crashing if that arises.
             */
            if (otherid == admin_of && form->admin_option &&
                OidIsValid(admin_of) && !OidIsValid(*admin_role))
                *admin_role = memberid;

!OidIsValid(*admin_role) の条件により、最初に見つかった 1 件だけが採用される。 探索が幅優先であることと組み合わせると、これは「ホップ数が最小の ADMIN OPTION 保持者」を 選ぶことになる。is_admin_of_role() はこの値が有効かどうかだけを見て真偽を返し、 select_best_admin() は値そのものを返す（4.4）。

注意すべきは、ADMIN OPTION の判定がモードのフィルタより前に置かれていることである。 上のコードは inherit_option／set_option による continue の 手前にある。したがって、ROLERECURSE_PRIVS モードであっても、 admin_of へ向かう最後の 1 本のグラント自体は継承可能である必要がない。 制約がかかるのは、そこに至るまでの経路（memberid がリストに載るまでの辺）だけである。

注意ただし admin_of が有効な場合はキャッシュが使われない （3.1 参照）。is_admin_of_role() の呼び出しは常に グラフの再展開を伴うため、他の判定関数より重い。
関連展開結果をどうキャッシュし、どう無効化するかは 第3章で扱う。
