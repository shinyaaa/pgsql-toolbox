第5章　ACL のビットマスク判定 aclmask

ここまではロール間の関係を扱ってきた。本章では層をひとつ下りて、オブジェクトに紐づく権限そのもの、 すなわち ACL のデータ構造と、そこから権限ビットを集約する aclmask() を読む。aclmask() こそが、ACL 層とメンバーシップ層をつなぐ接点である。

5.1　Acl と AclItem のレイアウト

ACL は AclItem の 1 次元配列であり、その実体は通常の PostgreSQL 配列（ArrayType）である。

src/include/utils/acl.h:54 
typedef struct AclItem
{
    Oid         ai_grantee;     /* ID that this item grants privs to */
    Oid         ai_grantor;     /* grantor of privs */
    AclMode     ai_privs;       /* privilege bits */
} AclItem;

AclMode は uint64 である。下位 32 ビットが権限そのもの、 上位 32 ビットが グラントオプション（WITH GRANT OPTION）を表す。 同じビット位置が上下で対応しているため、シフトだけで相互変換できる。

src/include/utils/acl.h:61 
/*
 * The upper 32 bits of the ai_privs field of an AclItem are the grant option
 * bits, and the lower 32 bits are the actual privileges.  We use "rights"
 * to mean the combined grant option and privilege bits fields.
 */
#define ACLITEM_GET_PRIVS(item)    ((item).ai_privs & 0xFFFFFFFF)
#define ACLITEM_GET_GOPTIONS(item) (((item).ai_privs >> 32) & 0xFFFFFFFF)
#define ACLITEM_GET_RIGHTS(item)   ((item).ai_privs)

#define ACL_GRANT_OPTION_FOR(privs) (((AclMode) (privs) & 0xFFFFFFFF) << 32)
#define ACL_OPTION_TO_PRIVS(privs)  (((AclMode) (privs) >> 32) & 0xFFFFFFFF)

図 5.1 にレイアウトを示す。

ai_grantee が 0（ACL_ID_PUBLIC）のエントリは PUBLIC への付与を表す。 pg_authid に OID 0 のロールは存在しないため、この値をプレースホルダとして使える。

実際の格納内容は aclitem[] のテキスト表現で観察できる。

SQL 
CREATE TABLE acl_demo (id int);
GRANT SELECT ON acl_demo TO m_both;
GRANT INSERT, UPDATE ON acl_demo TO PUBLIC;
GRANT SELECT ON acl_demo TO target WITH GRANT OPTION;

SELECT relacl FROM pg_class WHERE relname = 'acl_demo';
実行結果 
                                     relacl
--------------------------------------------------------------------------------
 {postgres=arwdDxtm/postgres,m_both=r/postgres,=aw/postgres,target=r*/postgres}
(1 row)

grantee=privs/grantor という書式である。3 番目の =aw/postgres は grantee 部分が空で、これが PUBLIC（ai_grantee == 0）を意味する。 4 番目の r* のアスタリスクが、SELECT のグラントオプション （上位 32 ビット側のビット）が立っていることを示す。

aclexplode() を使うと、このビット表現を行に展開できる。

SQL 
SELECT coalesce(pg_get_userbyid(nullif(grantee, 0)), 'PUBLIC') AS grantee,
       pg_get_userbyid(grantor) AS grantor,
       privilege_type, is_grantable
FROM   pg_class, aclexplode(relacl)
WHERE  relname = 'acl_demo'
ORDER  BY 1, 3;
実行結果（抜粋） 
 grantee  | grantor  | privilege_type | is_grantable
----------+----------+----------------+--------------
 PUBLIC   | postgres | INSERT         | f
 PUBLIC   | postgres | UPDATE         | f
 m_both   | postgres | SELECT         | f
 postgres | postgres | DELETE         | f
 ...
 target   | postgres | SELECT         | t
(12 rows)

5.2　aclmask の三段アルゴリズム

aclmask() は「roleid が持つ権限ビットのうち、mask に含まれるもの」を返す。 第 5 引数 how は打ち切り条件を指定する。

src/backend/utils/adt/acl.c:1395 
/*
 * aclmask --- compute bitmask of all privileges held by roleid.
 *
 * When 'how' = ACLMASK_ALL, this simply returns the privilege bits
 * held by the given roleid according to the given ACL list, ANDed
 * with 'mask'.  (The point of passing 'mask' is to let the routine
 * exit early if all privileges of interest have been found.)
 *
 * When 'how' = ACLMASK_ANY, returns as soon as any bit in the mask
 * is known true.  (This lets us exit soonest in cases where the
 * caller is only going to test for zero or nonzero result.)
 *
 * Usage patterns:
 *
 * To see if any of a set of privileges are held:
 *      if (aclmask(acl, roleid, ownerId, privs, ACLMASK_ANY) != 0)
 *
 * To see if all of a set of privileges are held:
 *      if (aclmask(acl, roleid, ownerId, privs, ACLMASK_ALL) == privs)
 *
 * To determine exactly which of a set of privileges are held:
 *      heldprivs = aclmask(acl, roleid, ownerId, privs, ACLMASK_ALL);
 */

本体は 3 つの段階からなる。図 5.2 に流れを示す。

段1: オーナーの暗黙グラントオプション。 オブジェクトの所有者はすべての権限にグラントオプションを持つ、とみなされる。 ここで has_privs_of_role() が使われるので、所有者ロールを継承しているロールも この恩恵を受ける。

src/backend/utils/adt/acl.c:1443 
    /* Owner always implicitly has all grant options */
    if ((mask & ACLITEM_ALL_GOPTION_BITS) &&
        has_privs_of_role(roleid, ownerId))
    {
        result = mask & ACLITEM_ALL_GOPTION_BITS;
        if ((how == ACLMASK_ALL) ? (result == mask) : (result != 0))
            return result;
    }

段2: 直接付与の走査。 ai_grantee が roleid そのもの、または PUBLIC であるエントリを集める。 メンバーシップの問い合わせは発生しない。

src/backend/utils/adt/acl.c:1455 
    /*
     * Check privileges granted directly to roleid or to public
     */
    for (i = 0; i < num; i++)
    {
        AclItem    *aidata = &aidat[i];

        if (aidata->ai_grantee == ACL_ID_PUBLIC ||
            aidata->ai_grantee == roleid)
        {
            result |= aidata->ai_privs & mask;
            if ((how == ACLMASK_ALL) ? (result == mask) : (result != 0))
                return result;
        }
    }

段3: 間接付与の走査。 ここで初めてメンバーシップ層に問い合わせる。段2 と段3 を分けているのは、 高価なメンバーシップ判定の回数を減らすための意図的な設計である。

src/backend/utils/adt/acl.c:1471 
    /*
     * Check privileges granted indirectly via role memberships. We do this in
     * a separate pass to minimize expensive indirect membership tests.  In
     * particular, it's worth testing whether a given ACL entry grants any
     * privileges still of interest before we perform the has_privs_of_role
     * test.
     */
    remaining = mask & ~result;
    for (i = 0; i < num; i++)
    {
        AclItem    *aidata = &aidat[i];

        if (aidata->ai_grantee == ACL_ID_PUBLIC ||
            aidata->ai_grantee == roleid)
            continue;           /* already checked it */

        if ((aidata->ai_privs & remaining) &&
            has_privs_of_role(roleid, aidata->ai_grantee))
        {
            result |= aidata->ai_privs & mask;
            if ((how == ACLMASK_ALL) ? (result == mask) : (result != 0))
                return result;
            remaining = mask & ~result;
        }
    }

条件式の順序が本質的である。aidata->ai_privs & remaining を先に評価し、 それが 0 なら has_privs_of_role() を呼ばずに次のエントリへ進む。 && の短絡評価により、まだ必要としている権限を実際に与えうるエントリだけが メンバーシップ判定の対象になる。remaining はビットが埋まるたびに更新され、 走査が進むほど残りの候補が減っていく。

この段3 の存在によって、「target に付与された権限が、target を継承する ロールにも及ぶ」という挙動が実現される。1.2 で作った 4 つのメンバーで確かめる。

SQL 
SELECT has_table_privilege('m_inh','acl_demo','SELECT') AS m_inh_select_via_target,
       has_table_privilege('m_set','acl_demo','SELECT') AS m_set_select_via_target;
実行結果 
 m_inh_select_via_target | m_set_select_via_target
-------------------------+-------------------------
 t                       | f
(1 row)

acl_demo の ACL に m_inh も m_set も現れないが、 target=r* のエントリに対する段3 の判定結果が分かれる。 m_inh は inherit_option = true なので has_privs_of_role(m_inh, target) が真になり、SELECT を得る。 m_set は偽なので得られない。

5.3　aclmask_direct との違い

aclmask_direct() は、同じ引数を取りながらメンバーシップを一切考慮しない静的関数である。

src/backend/utils/adt/acl.c:1501 
/*
 * aclmask_direct --- compute bitmask of all privileges held by roleid.
 *
 * This is exactly like aclmask() except that we consider only privileges
 * held *directly* by roleid, not those inherited via role membership.
 */

差分は 2 箇所に集約される。

aclmask()aclmask_direct()
段1（オーナー判定）has_privs_of_role(roleid, ownerId)roleid == ownerId
段2（直接付与）ai_grantee が roleid または PUBLICai_grantee == roleid のみ（PUBLIC を含まない）
段3（間接付与）ありなし

PUBLIC を除外している点は見落としやすい。この関数の唯一の呼び出し元である select_best_grantor() は「誰が権限の出所たりうるか」を探しており、 PUBLIC は付与者になれないため、含めては困るのである。

メモaclmask_direct() がメンバーシップを問い合わせないという性質は、 select_best_grantor() が roles_is_member_of() の返す List を ループ中ずっと保持できる根拠になっている（6.3）。 キャッシュが「次の呼び出しまでしか有効でない」以上、これは正しさに直結する前提である。

5.4　aclmembers と oid_cmp

ACL には、権限判定とは別に「この ACL が言及しているロールをすべて列挙する」という需要がある。 依存関係（pg_shdepend）の記録に使うためで、これを担うのが aclmembers() である。

src/backend/utils/adt/acl.c:1562 
/*
 * aclmembers
 *      Find out all the roleids mentioned in an Acl.
 *      Note that we do not distinguish grantors from grantees.
 *
 * *roleids is set to point to a palloc'd array containing distinct OIDs
 * in sorted order.  The length of the array is the function result.
 */

実装は「最悪ケースの領域を確保 → grantee と grantor を無差別に詰める → ソート → 重複除去」 という素直な流れである。ここで oid_cmp() が使われる。

src/backend/utils/adt/acl.c:1586 
    /* Allocate the worst-case space requirement */
    list = palloc(ACL_NUM(acl) * 2 * sizeof(Oid));
    acldat = ACL_DAT(acl);

    /*
     * Walk the ACL collecting mentioned RoleIds.
     */
    j = 0;
    for (i = 0; i < ACL_NUM(acl); i++)
    {
        const AclItem *ai = &acldat[i];

        if (ai->ai_grantee != ACL_ID_PUBLIC)
            list[j++] = ai->ai_grantee;
        /* grantor is currently never PUBLIC, but let's check anyway */
        if (ai->ai_grantor != ACL_ID_PUBLIC)
            list[j++] = ai->ai_grantor;
    }

    /* Sort the array */
    qsort(list, j, sizeof(Oid), oid_cmp);

    /* ... */

    /* Remove duplicates from the array */
    return qunique(list, j, sizeof(Oid), oid_cmp);

oid_cmp() 自体は oid.c にある汎用の qsort() 比較関数で、ACL 専用ではない。

src/backend/utils/adt/oid.c:285 
/* qsort comparison function for Oids */
int
oid_cmp(const void *p1, const void *p2)
{
    Oid         v1 = *((const Oid *) p1);
    Oid         v2 = *((const Oid *) p2);

    return pg_cmp_u32(v1, v2);
}

pg_cmp_u32() を使うのは、単純な v1 - v2 が Oid（uint32） では桁あふれしうるためである。qunique() は隣接する等値要素を潰す関数なので、 事前のソートが前提になる。qsort() と qunique() が同じ比較関数を 共有しているのはそのためである。
