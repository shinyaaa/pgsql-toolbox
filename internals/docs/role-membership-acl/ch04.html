第4章　判定関数の意味論

第2章で見たとおり、公開されているロール判定関数はいずれも roles_is_member_of() の 薄いラッパである。しかし「薄い」からこそ、どのラッパを選ぶかがそのまま権限モデルの意味を決める。 本章では 4 つの関数（および superuser を無視する変種）を順に読み、どこで使うべきかを整理する。

4.1　has_privs_of_role — 権限の自動継承

has_privs_of_role() は「member が role の権限を自動的に行使できるか」を返す。 権限チェックで使うべき唯一の関数である。

src/backend/utils/adt/acl.c:5308 
/*
 * Does member have the privileges of role (directly or indirectly)?
 *
 * This is defined not to recurse through grants that are not inherited,
 * and only inherited grants confer the associated privileges automatically.
 *
 * See also member_can_set_role, below.
 */
bool
has_privs_of_role(Oid member, Oid role)
{
    /* Fast path for simple case */
    if (member == role)
        return true;

    /* Superusers have every privilege, so are part of every role */
    if (superuser_arg(member))
        return true;

    /*
     * Find all the roles that member has the privileges of, including
     * multi-level recursion, then see if target role is any one of them.
     */
    return list_member_oid(roles_is_member_of(member, ROLERECURSE_PRIVS,
                                              InvalidOid, NULL),
                           role);
}

構造は 3 段である。図 4.1 に示すように、自己一致の早期リターン、スーパーユーザの早期リターン、そして ROLERECURSE_PRIVS モードでの展開結果に対する線形探索、という順で進む。 4 つの関数すべてがこの型を共有しており、違うのは第 2 段の有無と展開モードだけである （4.5 の対応表にまとめる）。

superuser_arg() による早期リターンは、「スーパーユーザはすべてのロールの 暗黙のメンバーである」という規則の実装である。これは pg_auth_members に行を作らずに 実現されており、pg_database_owner（2.3）と並ぶ 暗黙メンバーシップのもう一つの例である。

4.2　member_can_set_role — SET ROLE 可能性

member_can_set_role() は ROLERECURSE_SETROLE モードを使う。コメントが用途を明示している。

src/backend/utils/adt/acl.c:5336 
/*
 * Can member use SET ROLE to this role?
 *
 * There must be a chain of grants from 'member' to 'role' each of which
 * permits SET ROLE; that is, each of which has set_option = true.
 *
 * It doesn't matter whether the grants are inheritable. That's a separate
 * question; see has_privs_of_role.
 *
 * This function should be used to determine whether the session user can
 * use SET ROLE to become the target user. We also use it to determine whether
 * the session user can change an existing object to be owned by the target
 * user, or create new objects owned by the target user.
 */

最後の一文が重要である。SET ROLE の可否だけでなく、 オブジェクトの所有者になれるかの判定にも使われる。理屈としては、 「そのロールになりすませるなら、そのロールが所有するオブジェクトを作るのと変わらない」ということである。

そのためのエラー送出付きラッパが check_can_set_role() であり、各 DDL の OWNER TO 処理から広く呼ばれている。

src/backend/utils/adt/acl.c:5373 
void
check_can_set_role(Oid member, Oid role)
{
    if (!member_can_set_role(member, role))
        ereport(ERROR,
                (errcode(ERRCODE_INSUFFICIENT_PRIVILEGE),
                 errmsg("must be able to SET ROLE \"%s\"",
                        GetUserNameFromId(role, false))));
}
呼び出し元文脈
tablecmds.cALTER TABLE ... OWNER TO
schemacmds.cCREATE SCHEMA ... AUTHORIZATION
dbcommands.cCREATE DATABASE ... OWNER
variable.cSET ROLE の GUC assign フック
usercontext.cメンテナンス処理での一時的なロール切り替え

variable.c の呼び出しでは、判定の主体が GetUserId() ではなく GetSessionUserId() である点に注意したい。SET ROLE を重ねても 権限が積み上がらないよう、常にセッションユーザを基準に判定している。

4.3　is_member_of_role と is_member_of_role_nosuper

is_member_of_role() は ROLERECURSE_MEMBERS、すなわちオプションを一切無視した展開を行う。 第1章で引用したとおり、コメントは「ほぼどこで呼んでも間違い」と断じている。

一方 is_member_of_role_nosuper() は、is_member_of_role() から superuser_arg() の早期リターンを取り除いただけの関数である。

src/backend/utils/adt/acl.c:5416 
/*
 * Is member a member of role, not considering superuserness?
 *
 * This is identical to is_member_of_role except we ignore superuser
 * status.
 *
 * Do not use this for privilege checking, instead use has_privs_of_role()
 */

「スーパーユーザを特別扱いしない」ことが望ましい場面は限られるが、確かに存在する。 20devel における呼び出しは 2 箇所だけである。

  • hba.c — pg_hba.conf の +role 記法によるマッチング。認証はまだ通っていないので、 「実質的にスーパーユーザだから全ロールに一致する」という扱いをしてはならない。
  • user.c — メンバーシップ付与時の循環検出。スーパーユーザを全ロールのメンバーとみなすと、 あらゆる付与が循環と判定されてしまう。
注意is_member_of_role() 系は「メンバーではあるが権限は及ばない」 ケースを真と判定する。1.2 の実験における m_none がまさにそれで、 MEMBER は t だがテーブルは読めない。権限チェックにこれを使うと権限昇格になる。

4.4　is_admin_of_role と select_best_admin

この 2 つは、2.4 で見た admin_of／admin_role の 仕組みを使う姉妹関数である。

src/backend/utils/adt/acl.c:5441 
bool
is_admin_of_role(Oid member, Oid role)
{
    Oid         admin_role;

    if (superuser_arg(member))
        return true;

    /* By policy, a role cannot have WITH ADMIN OPTION on itself. */
    if (member == role)
        return false;

    (void) roles_is_member_of(member, ROLERECURSE_MEMBERS, role, &admin_role);
    return OidIsValid(admin_role);
}

/* ... */

Oid
select_best_admin(Oid member, Oid role)
{
    Oid         admin_role;

    /* By policy, a role cannot have WITH ADMIN OPTION on itself. */
    if (member == role)
        return InvalidOid;

    (void) roles_is_member_of(member, ROLERECURSE_PRIVS, role, &admin_role);
    return admin_role;
}

戻り値の型以外に、3 つの違いがある。

  1. 展開モード: is_admin_of_role() は ROLERECURSE_MEMBERS、 select_best_admin() は ROLERECURSE_PRIVS。後者は 「権限が及ぶロールのうち ADMIN OPTION を持つもの」を探すので条件が厳しい。
  2. superuser の扱い: select_best_admin() にはスーパーユーザの 早期リターンがない。呼び出し元がスーパーユーザを別扱いするためである（後述）。
  3. 自己に対する扱い: 両者とも member == role を偽とする。 コメントの "By policy, a role cannot have WITH ADMIN OPTION on itself." がそれを述べている。

この「自分自身の ADMIN にはなれない」規則は、SQL からも観察できる。 agent_r は admin_r を継承し、admin_r は club に ADMIN OPTION を持つ、という階層で確かめる。

SQL 
CREATE ROLE club;
CREATE ROLE admin_r;
CREATE ROLE agent_r LOGIN;
GRANT club    TO admin_r WITH ADMIN OPTION;
GRANT admin_r TO agent_r;              -- 既定で INHERIT TRUE

SELECT pg_has_role('agent_r','club','MEMBER WITH ADMIN OPTION') AS agent_is_admin,
       pg_has_role('admin_r','club','MEMBER WITH ADMIN OPTION') AS admin_is_admin,
       pg_has_role('club',   'club','MEMBER WITH ADMIN OPTION') AS self_is_admin;
実行結果 
 agent_is_admin | admin_is_admin | self_is_admin
----------------+----------------+---------------
 t              | t              | f
(1 row)

select_best_admin() の実際の使われ方は user.c にある。GRANT role TO ... を実行したとき、pg_auth_members.grantor に 何を記録するかを決める処理である。

src/backend/commands/user.c:2228 
        if (superuser_arg(currentUserId))
            return BOOTSTRAP_SUPERUSERID;

        /*
         * Otherwise, the grantor must either have ADMIN OPTION on the role or
         * inherit the privileges of a role which does. In the former case,
         * record the grantor as the current user; in the latter, pick one of
         * the roles that is "most directly" inherited by the current role
         * (i.e. fewest "hops").
         *
         * (We shouldn't fail to find a best grantor, because we've already
         * established that the current user has permission to perform the
         * operation.)
         */
        grantorId = select_best_admin(currentUserId, roleid);
        if (!OidIsValid(grantorId))
            elog(ERROR, "no possible grantors");
        return grantorId;

スーパーユーザは select_best_admin() を経ずに BOOTSTRAP_SUPERUSERID を返す。 これが、select_best_admin() 側に superuser の早期リターンがない理由である。 実際に agent_r として GRANT を実行すると、記録される付与者は agent_r ではなく admin_r になる。

SQL（agent_r で接続して実行） 
GRANT club TO newbie;
実行結果（postgres で確認） 
SELECT pg_get_userbyid(roleid)  AS role,
       pg_get_userbyid(member)  AS member,
       pg_get_userbyid(grantor) AS grantor,
       admin_option
FROM   pg_auth_members
WHERE  roleid = 'club'::regrole
ORDER  BY 2;

 role | member  | grantor  | admin_option
------+---------+----------+--------------
 club | admin_r | postgres | t
 club | newbie  | admin_r  | f
(2 rows)

この「実行者ではなく、権限の出所となるロールを付与者として記録する」考え方は、 オブジェクト権限側の select_best_grantor() と同じ設計思想に基づく。 第6章で対比しながら読む。

4.5　pg_has_role() との対応関係

ここまでの 4 関数は、SQL 関数 pg_has_role() の第 3 引数を通じてすべて呼び分けられる。 ディスパッチを行うのが pg_role_aclcheck() である。

src/backend/utils/adt/acl.c:5041 
static AclResult
pg_role_aclcheck(Oid role_oid, Oid roleid, AclMode mode)
{
    if (mode & ACL_GRANT_OPTION_FOR(ACL_CREATE))
    {
        if (is_admin_of_role(roleid, role_oid))
            return ACLCHECK_OK;
    }
    if (mode & ACL_CREATE)
    {
        if (is_member_of_role(roleid, role_oid))
            return ACLCHECK_OK;
    }
    if (mode & ACL_USAGE)
    {
        if (has_privs_of_role(roleid, role_oid))
            return ACLCHECK_OK;
    }
    if (mode & ACL_SET)
    {
        if (member_can_set_role(roleid, role_oid))
            return ACLCHECK_OK;
    }
    return ACLCHECK_NO_PRIV;
}

権限文字列から AclMode への変換は convert_role_priv_string() が行う。ロールには本来対応する ACL ビットがないため、既存のビットを流用する「ごまかし」が行われている。

src/backend/utils/adt/acl.c:5008 
/*
 * convert_role_priv_string
 *      Convert text string to AclMode value.
 *
 * We use USAGE to denote whether the privileges of the role are accessible
 * (has_privs_of_role), MEMBER to denote is_member, and MEMBER WITH GRANT
 * (or ADMIN) OPTION to denote is_admin.  There is no ACL bit corresponding
 * to MEMBER so we cheat and use ACL_CREATE for that.  This convention
 * is shared only with pg_role_aclcheck, below.
 */

以上をまとめると、SQL レベルの権限文字列と C レベルの判定関数は次の対応になる。 本書の実行例で pg_has_role() を多用しているのは、この対応があるためである。
