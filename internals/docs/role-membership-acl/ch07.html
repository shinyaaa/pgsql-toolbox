第7章　行レベルセキュリティへの適用

行レベルセキュリティ（Row Level Security, RLS）は、テーブルに定義されたポリシーの USING／WITH CHECK 式をクエリに注入する仕組みである。 ポリシーには適用対象ロールを指定できるため、ここでもロールメンバーシップの判定が必要になる。 本章では、RLS がどの判定関数を選び、その選択が運用上どう効いてくるかを見る。

7.1　get_policies_for_relation — コマンド種別による選別

クエリ書き換え時、 get_row_security_policies() が RLS の適用を統括する。その最初の仕事が、対象リレーションに定義されたポリシーのうち 「このコマンドとこのロールに適用されるもの」を選び出すことであり、これを担うのが get_policies_for_relation() である。

src/backend/rewrite/rowsecurity.c:547 
/*
 * get_policies_for_relation
 *
 * Returns lists of permissive and restrictive policies to be applied to the
 * specified relation, based on the command type and role.
 *
 * This includes any policies added by extensions.
 */

ポリシーは relcache に RowSecurityDesc として保持されており、 各要素は RowSecurityPolicy である。

src/include/rewrite/rowsecurity.h:20 
typedef struct RowSecurityPolicy
{
    char       *policy_name;    /* Name of the policy */
    char        polcmd;         /* Type of command policy is for */
    ArrayType  *roles;          /* Array of roles policy is for */
    bool        permissive;     /* restrictive or permissive policy */
    Expr       *qual;           /* Expression to filter rows */
    Expr       *with_check_qual;    /* Expression to limit rows allowed */
    bool        hassublinks;    /* If either expression has sublinks */
} RowSecurityPolicy;

選別は 2 条件の AND である。ひとつはコマンド種別の一致、もうひとつがロールの一致。

src/backend/rewrite/rowsecurity.c:610 
        /*
         * Add this policy to the relevant list of policies if it applies to
         * the specified role.
         */
        if (cmd_matches && check_role_for_policy(policy->roles, user_id))
        {
            if (policy->permissive)
                *permissive_policies = lappend(*permissive_policies, policy);
            else
                *restrictive_policies = lappend(*restrictive_policies, policy);
        }

polcmd が '*'（FOR ALL）のポリシーは無条件に一致し、 それ以外は CmdType との対応表で判定される。MERGE だけは特別扱いで、 専用のポリシーを持たない。

7.2　check_role_for_policy — ロール適用判定

本書の主題からすると、RLS における最重要の 20 行が check_role_for_policy() である。

src/backend/rewrite/rowsecurity.c:926 
/*
 * check_role_for_policy -
 *   determines if the policy should be applied for the current role
 */
static bool
check_role_for_policy(ArrayType *policy_roles, Oid user_id)
{
    int         i;
    Oid        *roles = (Oid *) ARR_DATA_PTR(policy_roles);

    /* Quick fall-thru for policies applied to all roles */
    if (roles[0] == ACL_ID_PUBLIC)
        return true;

    for (i = 0; i < ARR_DIMS(policy_roles)[0]; i++)
    {
        if (has_privs_of_role(user_id, roles[i]))
            return true;
    }

    return false;
}

読み取るべき点は 3 つある。

  1. 使うのは has_privs_of_role() である。 is_member_of_role() ではない。つまり、ポリシーが適用されるのは 「そのロールの権限が自動的に及ぶ」ロールに限られる。単なるメンバーでは適用されない。
  2. PUBLIC の判定は先頭要素だけを見る。 polroles に他のロールと PUBLIC が混在しないことが保証されているためである。 policy_role_list_to_array() は TO 句の省略時も TO PUBLIC 指定時も配列を {0} の 1 要素にし、 TO PUBLIC, alice のような混在指定に対しては "ignoring specified roles other than PUBLIC" という WARNING を出して {0} に切り詰める。
  3. 複数ロール指定は OR である。 TO a, b と書いたポリシーは、a か b のどちらかの権限が 及べば適用される。

1 番目の帰結は、運用上かなり大きな意味を持つ。実験してみよう。 sales ロールにポリシーを付け、INHERIT の有無だけが違う 2 人のメンバーを作る。

SQL 
CREATE TABLE docs (id int, dept text);
INSERT INTO docs VALUES (1,'sales'), (2,'hr');
ALTER TABLE docs ENABLE ROW LEVEL SECURITY;

CREATE ROLE sales;
CREATE ROLE alice LOGIN;
CREATE ROLE bob   LOGIN;
GRANT sales TO alice WITH INHERIT TRUE,  SET TRUE;
GRANT sales TO bob   WITH INHERIT FALSE, SET TRUE;
GRANT SELECT ON docs TO alice, bob, sales;

CREATE POLICY p_sales ON docs FOR SELECT TO sales USING (dept = 'sales');
実行結果（alice で接続） 
=> SELECT * FROM docs;
 id | dept
----+-------
  1 | sales
(1 row)
実行結果（bob で接続） 
=> SELECT * FROM docs;
 id | dept
----+------
(0 rows)

bob は sales のメンバーであり、テーブルへの SELECT 権限も 直接持っている。それでも 1 行も見えない。has_privs_of_role(bob, sales) が偽なので p_sales が適用対象から外れ、bob に適用される permissive ポリシーが ひとつも残らなかったからである。SET ROLE すれば見えるようになる。

実行結果（bob で接続） 
=> SET ROLE sales;
SET
=> SELECT * FROM docs;
 id | dept
----+-------
  1 | sales
(1 row)

7.4　継承がもたらす直感に反する挙動

本章の実験が示したのは、RLS の適用範囲が pg_auth_members の inherit_option に左右されるという事実である。これは仕様として一貫しているが、 運用上は次のような落とし穴になりうる。

注意NOINHERIT なロールや WITH INHERIT FALSE の グラントを使っている環境では、「グループにポリシーを付けたのにメンバーに適用されない」ことが起きる。 テーブル権限（GRANT SELECT）も同じ has_privs_of_role() を経由するため 通常は両方まとめて効かなくなるが、メンバーに直接テーブル権限を与えていると、 アクセスはできるが 1 行も見えないという状態になる。

逆向きの注意点もある。ポリシーの適用判定に使われるのは user_id であり、 これは SET ROLE で切り替わる現在ユーザである。ただし RLS には BYPASSRLS 属性や、テーブル所有者に対する既定の適用除外があり、 適用の可否そのものは check_enable_rls() が先に判定する。本章で見た get_policies_for_relation() は、 その判定を通過したあとの処理である。

最後に、ロール変更に対するプランキャッシュの扱いに触れておく。 get_row_security_policies() は書き換えを行った場合に *hasRowSecurity = true を立てる。

src/backend/rewrite/rowsecurity.c:540 
    /*
     * Mark this query as having row security, so plancache can invalidate it
     * when necessary (eg: role changes)
     */
    *hasRowSecurity = true;

SET ROLE によって適用されるポリシーが変わりうる以上、 生成済みのプランをそのまま再利用してはならない。このフラグがその再計画を促す。 ロールメンバーシップの判定結果が、権限チェックだけでなくプランの有効性にまで 波及していることが分かる。

7.5　まとめ

本書で読んだ 15 の関数の関係を整理して締めくくる。

図 7.3 が示すとおり、権限判定のほぼすべての経路が roles_is_member_of() に収束する。 acl.c がモードごとに 1 スロットという小さなキャッシュしか持たないのは、 「同じロールについて連続して問い合わせる」というアクセスパターンが支配的だからである。 そして、そのキャッシュが「次の呼び出しまでしか有効でない」という制約は、 select_best_grantor() が aclmask() ではなく aclmask_direct() を使う理由にまで影響している。 小さな設計判断が層をまたいで結びついているのが、この領域のコードの特徴である。
