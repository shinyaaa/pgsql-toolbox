第5章　GRAPH_TABLE の書き換え

本章がドキュメントの核心である。解析変換で得た RTE_GRAPH_TABLE を、 rewriteGraphTable() が どのようにして辺と頂点の等結合からなる通常のサブクエリへ書き換えるのかを、 1 本のパス（単一パス）を例に段階的に追う。ラベル選言による複数パスと UNION は第6章で扱う。

5.1　書き換えの起点

書き換えは、リライタ本体 rewriteHandler.c が 範囲テーブルを走査して RTE_GRAPH_TABLE を見つけたときに rewriteGraphTable() を呼ぶことで始まる。この関数は次の 3 手で RTE を その場で書き換える。

  1. パスパターンからパスクエリのリストを生成する （generate_queries_for_path_pattern()）。
  2. それらを UNION ALL で束ねて 1 つの Query にする （generate_union_from_pathqueries()）。
  3. RTE の種別を RTE_SUBQUERY へ変え、subquery に上記クエリを差し込み、 lateral = true にする。グラフ固有フィールドは NULL でクリアする。

これで RTE は「ラテラルなサブクエリ」に化ける。以降のプランナ・実行器はグラフのことを 一切知らずに処理できる。

5.2　path_factor と path_element

書き換えロジックは 2 つの内部構造体を軸に進む。

  • path_factor: パス中の 1 つの「位置」を表す。非循環パスでは 1 要素パターンに対応する。kind・ variable・labelexpr・whereClause、パス内での位置 factorpos、そして辺の場合の隣接頂点へのリンク src_pf／dest_pf を持つ。
  • path_element: 1 つの path_factor のラベル式が解決した具体的な要素を表す。要素 OID・ 実体テーブル OID、辺なら始点・終点頂点 OID と結合条件（src_quals／ dest_quals）を持つ。

つまり path_factor は「パターン上の位置」、path_element は「その位置に当てはまる実要素」であり、 1 つの path_factor は複数の path_element を生みうる（ラベルが複数要素に付いている場合）。

5.3　パスの列挙（K-partite グラフの DFS）

generate_queries_for_path_pattern() は、まずパターン中の要素パターン列を path_factor のリストに変換し、隣接する頂点と辺の リンク（src_pf／dest_pf）を張る。次に各 path_factor について、 そのラベル式を満たす要素を集めて path_element のリストを作る （get_path_elements_for_path_factor()）。

図 5.1 に示すように、K 個の要素パターンからなるパスパターンは、各パターンが要素の集合を 持つ K 部グラフ（K-partite graph）とみなせる。1 本の具体的なパスは、 各パターンから 1 要素ずつ選んだ組み合わせであり、可能なパスの総数は要素数の積になる。 これを深さ優先探索（DFS）で列挙するのが generate_queries_for_path_pattern_recurse() である。

DFS が末端に達するたびに、そのとき選ばれている要素の並び（1 本のパス）を generate_query_for_graph_path() へ渡して 1 つのクエリを作る。辺が隣接頂点を実際には結んでいない組み合わせは 「壊れたパス」として捨てられ（NULL を返す）、クエリは作られない。

5.4　1 本のパスから 1 クエリを生成する

generate_query_for_graph_path() は、与えられた 1 本のパスから SELECT クエリを組み立てる。生成物は次の 3 要素からなる。

  • FROM リスト: パス中の各要素の実体テーブルに対する RangeTblEntry。 path_factor.factorpos + 1 が各 RTE の rtindex になるよう、 要素の順に追加される。
  • WHERE（qual）: (a) 辺と隣接頂点を結ぶ等結合条件、(b) 各要素の WHERE 句、(c) GRAPH_TABLE 全体の WHERE 句、の論理積。
  • ターゲットリスト: COLUMNS の各式。

辺-頂点の結合条件は、5.2 節で述べたとおり path_element 生成時に create_pe_for_element() が あらかじめ組み立てておいた src_quals／dest_quals を使う（詳細は第6章）。

一方、COLUMNS や WHERE に現れる GraphPropertyRef（プロパティ参照）は、 replace_property_refs_mutator() が 実際の列参照へ展開する。手順はこうだ。参照された変数（path_element）に対応する ラベル群をたどり、そのプロパティの値式（pg_propgraph_label_property.plpexpr）を カタログから取り出し、式中の Var の varno をこの要素の rtindex （= factorpos + 1）へ付け替える（ChangeVarNodes()）。図 5.2 に流れを示す。

プロパティがその要素のどのラベルにも値式を持たない場合でも、パターンが選んだ いずれかのラベルにそのプロパティが関連していれば、SQL/PGQ 標準（6.5 節）に従って NULL 定数へ展開される。どのラベルとも無関係なプロパティを参照した場合にのみ エラーとなる。最後に、参照された列に SELECT 権限フラグを立てる処理があり、 これは「プロパティグラフ所有者ではなく実行ユーザーの権限で基底表へアクセスする」 という設計（SECURITY INVOKER なビューに相当）を裏付けている。
