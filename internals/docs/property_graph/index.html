PostgreSQL プロパティグラフ（SQL/PGQ）の内部構造

PostgreSQL 19beta1 · 031904048aa · 2026-07-09 12:21

プロパティグラフ（property graph）は、頂点（vertex）と辺（edge）にラベルとプロパティを 持たせたグラフデータモデルであり、SQL/PGQ（ISO/IEC 9075-16, SQL:2023 の一部）として 標準化されている。本ドキュメントは、SQL/PGQ を実装した PostgreSQL 19beta1 （shinyaaa/postgres フォーク）を題材に、CREATE PROPERTY GRAPH による グラフ定義がどのようにシステムカタログへ格納され、GRAPH_TABLE クエリが どのようにして通常のリレーショナルクエリへ書き換えられて実行されるのかを、 ソースコードに即して解説する。対象読者は PostgreSQL のパーサ・リライタの基本的な仕組みを 知っているエンジニアである。専用の実行器ノードは一切追加されておらず、グラフ照会機能の 本質が「クエリ書き換え」に凝縮されている点が最大の見どころである。

第1章 プロパティグラフ概論 SQL/PGQ とは何か、プロパティグラフの論理オブジェクトとしての性質、そしてクエリ処理パイプライン全体像を俯瞰する。 第2章 グラフの定義: CREATE PROPERTY GRAPH 構文・頂点/辺・KEY・SOURCE/DESTINATION・ラベルとプロパティ・一貫性チェック、そしてカタログ登録の流れを解説する。 第3章 システムカタログ 定義を格納する 5 つの pg_propgraph_* カタログの役割・列・索引と、pg_get_propgraphdef による復元を詳解する。 第4章 GRAPH_TABLE の構文解析 MATCH パターンの構文、RangeGraphTable/GraphPattern/GraphElementPattern のパースツリー、解析変換と参照解決を追う。 第5章 GRAPH_TABLE の書き換え RTE_GRAPH_TABLE をリレーショナルなサブクエリへ変換する中核。path_factor/path_element、パス列挙、クエリ生成を解剖する。 第6章 複数パス・UNION・循環パターン ラベル選言による複数パスの UNION、辺-頂点結合条件の生成、双方向辺・循環パターン・空パスの扱いを掘り下げる。 第7章 実践: 観察と制限事項 myshop グラフの完全例、書き換え結果の観察方法、権限・ビュー・準備文との相互作用、そして現在の実装上の制限をまとめる。
