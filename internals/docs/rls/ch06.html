第6章　バイパス・FORCE・キャッシュ無効化の運用面

本章では RLS の運用面で誤解されやすいポイント、すなわちバイパス系の経路と、 GUC・キャッシュ・最適化との相互作用を整理する。いずれも check_enable_rls() と書き換え経路に実装の根拠がある。

6.1　BYPASSRLS とテーブル所有者

BYPASSRLS はロール属性で、付与されたロールはすべての RLS を無効化する。 スーパーユーザーは has_bypassrls_privilege() が常に true を返すため 自動的にバイパス対象である。check_enable_rls() はこの判定を行うと RLS_NONE_ENV を返す。これは「現在の環境では適用しない」というシグナルであり、 ロール切り替えで状況が変わりうるためプランキャッシュは無効化対象として残る。

テーブル所有者の挙動はこれと似て非なる。所有者は既定で自テーブルの RLS をバイパスするが、 これは「自分のテーブルだから無条件で全行アクセス」というよりは「DDL 主体への保護」に近い。 実装上は object_ownercheck() の結果と relforcerowsecurity フラグを組み合わせ、 FORCE 未指定のときに限ってバイパスとなる（次節）。

rls.c 抜粋 
if (has_bypassrls_privilege(user_id))
    return RLS_NONE_ENV;

amowner = object_ownercheck(RelationRelationId, relid, user_id);
if (amowner)
{
    if (!relforcerowsecurity || InNoForceRLSOperation())
        return RLS_NONE_ENV;
}

6.2　FORCE ROW LEVEL SECURITY

ALTER TABLE … FORCE ROW LEVEL SECURITY は所有者にも RLS を強制するスイッチで、 pg_class.relforcerowsecurity に格納される。RLS が有効かつ FORCE がセットされていれば、 所有者であっても RLS_ENABLED 経路に進む。

ただし「FORCE 状態でも RLS を適用しない」例外として、参照整合性制約のチェックなどで 使われる InNoForceRLSOperation() がある。これが真のあいだは、所有者経由の RI 検査が RLS で阻害されない。ソースコメントは次のように説明している。

rls.c コメント抜粋 
InNoForceRLSOperation indicates that we should not apply RLS even
if the table has FORCE RLS set - IF the current user is the owner.
This is specifically to ensure that referential integrity checks
are able to still run correctly.

この設計は「所有者でない通常ユーザーが INSERT したとき、外部キー先のテーブルにも RLS が掛かっていると整合性検査が失敗しうる」という問題を回避するためのものである。

6.3　row_security GUC と plan invalidation

row_security は接続ローカルの GUC で、既定は on。ユーザーが off に設定すると、 RLS が適用されるはずのクエリで 明示的にエラー を発生させる。

rls.c より 
if (!row_security && !noError)
    ereport(ERROR,
            (errcode(ERRCODE_INSUFFICIENT_PRIVILEGE),
             errmsg("query would be affected by row-level security policy for table \"%s\"",
                    get_rel_name(relid)),
             amowner ? errhint("To disable the policy for the table's owner, use ALTER TABLE NO FORCE ROW LEVEL SECURITY.") : 0));

目的は「ダンプや一括ロード時に、暗黙の行フィルタによってデータが欠落することを利用者に気付かせる」 ことにある。pg_dump はデフォルトでこの GUC を off にしてダンプを試み、 RLS で阻まれる行があれば即座にエラーとして検出する。

プラン無効化の観点では次の流れが鍵となる。

  1. 書き換え時、get_row_security_policies() は最後に *hasRowSecurity = true を立てる。
  2. rewriter はこの値を Query->hasRowSecurity へ伝搬する （rewriteHandler.c L504 ほか）。
  3. plancache はこのフラグを持つプランを「ロール変更・GUC 変更・カタログ無効化のいずれかで再計画必須」と扱う。
  4. RelationBuildRowSecurity() の再実行は relcache 無効化で誘発される （3.3 節）。

6.4　LEAKPROOF と最適化との相互作用

RLS の述語（securityQuals）はセキュリティバリアとして扱われる。これは、 プランナがフィルタやインデックス条件を行レベル安全境界の 内側 に押し下げてよいかを 関数の leakproof 属性で判定するという意味である。LEAKPROOF でない関数は バリアより上に置かれ、見せたくない行の値を「副作用つきの関数」が観測する可能性を排除する。

実装上は、get_row_security_policies() 自体は単に「securityQuals に式を入れる」だけで、 バリアとしての扱いはプランナ側の securityQuals 取り扱いに委ねられている。 これによって RLS とセキュリティバリアビューの取り扱いを同じ枠組みで実現している。

注意 運用上、独自関数を RLS の述語で使う場合は、安全性を確認したうえで CREATE FUNCTION ... LEAKPROOF として宣言しないと、 プランナによる qual の押し下げが効かず、性能上の不利益が出ることがある。 LEAKPROOF を付けるにはスーパーユーザー権限が必要。

まとめ

全章を通じて、RLS は次の 3 層に分けて理解すると見通しが良い。

  1. カタログ層: pg_policy と pg_class.relrowsecurity / relforcerowsecurity がポリシーの永続表現を担う。
  2. キャッシュ層: RowSecurityDesc が relcache 上にロードされ、 DDL 完了時の CacheInvalidateRelcache() で再構築される。
  3. クエリ層: rewriter が get_row_security_policies() 経由で securityQuals と withCheckOptions を Query に注入し、 hasRowSecurity フラグで plancache と協調する。

この三層がそれぞれ独立しているおかげで、ポリシーは DDL から書き換え、最適化、実行までを 一貫したインターフェースで扱える。RLS を運用する際の挙動は、ほぼすべてこの 3 層のどこに 対応するかを意識すれば説明可能である。
