第3章　CREATE/ALTER/DROP POLICY の実装

本章では CREATE POLICY / ALTER POLICY / DROP POLICY の 実装を src/backend/commands/policy.c を中心に追う。これらの DDL は カタログへの書き込み、述語のパース、依存関係の登録、relcache の無効化までを 一気通貫で行う。

3.1　CREATE POLICY の処理フロー

CREATE POLICY はパーサから CreatePolicyStmt として 実行器（utility）に渡され、CreatePolicy() が処理する。図 3.1 は CreatePolicy() 内部の主要ステップである。

対象テーブルは AccessExclusiveLock で開かれる。これは 「クエリのプランがポリシー集合に依存しうるため、ポリシー変更中の問い合わせを すべて締め出す」ためで、RemovePolicyById() のコメントにも同様の説明がある。

対象コマンドや WITH CHECK の組み合わせに対する妥当性検査もこの関数で行われる。 たとえば SELECT / DELETE に WITH CHECK は意味がないため、明示すると 「WITH CHECK cannot be applied to SELECT or DELETE」というエラーになる （policy.c L599）。 INSERT で USING を指定するのも禁止される。

3.2　USING 句と WITH CHECK 句の解析

USING / WITH CHECK 句は テーブル 1 つを範囲表（range table）に持つ独立クエリ として 解釈される。CreatePolicy() は次のように 2 つの ParseState を用意し、 それぞれに対象テーブルを RTE として登録してから transformWhereClause() を呼ぶ。

policy.c 抜粋 
qual_pstate = make_parsestate(NULL);
with_check_pstate = make_parsestate(NULL);
...
nsitem = addRangeTableEntryForRelation(qual_pstate, target_table,
                                       AccessShareLock,
                                       NULL, false, false);
addNSItemToQuery(qual_pstate, nsitem, false, true, true);
...
qual = transformWhereClause(qual_pstate, stmt->qual,
                            EXPR_KIND_POLICY, "POLICY");
with_check_qual = transformWhereClause(with_check_pstate, stmt->with_check,
                                       EXPR_KIND_POLICY, "POLICY");
assign_expr_collations(qual_pstate, qual);
assign_expr_collations(with_check_pstate, with_check_qual);

EXPR_KIND_POLICY という式種別を渡すことで、集約関数やウィンドウ関数など ポリシーで使えない構文をエラーとして弾けるようにしている。最終的に nodeToString() で polqual / polwithcheck 列に格納される テキスト表現が得られる。

ロール集合は policy_role_list_to_array() が RoleSpec リストを Oid 配列に変換する。PUBLIC が単独で 指定されていれば ACL_ID_PUBLIC の 1 要素配列に正規化され、 他のロールと併記すると WARNING を出して PUBLIC に縮退させる。

3.3　依存関係の登録と relcache 無効化

ポリシーは「テーブル」「式中に現れるオブジェクト」「ロール」と複数の依存先を持つ。 CreatePolicy() はこれを次のように記録する。

policy.c 抜粋 
recordDependencyOn(&myself, &target, DEPENDENCY_AUTO);
recordDependencyOnExpr(&myself, qual, qual_pstate->p_rtable,
                       DEPENDENCY_NORMAL);
recordDependencyOnExpr(&myself, with_check_qual,
                       with_check_pstate->p_rtable, DEPENDENCY_NORMAL);
/* Register role dependencies */
target.classId = AuthIdRelationId;
for (i = 0; i < nitems; i++)
{
    target.objectId = DatumGetObjectId(role_oids[i]);
    if (target.objectId != ACL_ID_PUBLIC)
        recordSharedDependencyOn(&myself, &target,
                                 SHARED_DEPENDENCY_POLICY);
}

依存関係の整理として次の点が重要である。

  • テーブルとは DEPENDENCY_AUTO。テーブルが DROP されればポリシーも自動で消える。
  • 式中のオブジェクト（例: USING 句で参照される関数や型）は DEPENDENCY_NORMAL。 これらを DROP しようとすると、依存先のポリシーが残っている限りエラーになる。
  • ロール依存は SHARED_DEPENDENCY_POLICY として shared dependency に記録される。 ロールを DROP すると RemoveRoleFromObjectPolicy() が呼ばれ、ポリシーの polroles 配列から該当ロールを除外する。 結果としてロールが 0 になる場合はポリシー自体を削除する。

最後に CacheInvalidateRelcache(target_table) を発行し、対象リレーションの relcache エントリを無効化する。これによって既存セッションが次にこのテーブルを参照したとき、 RelationBuildRowSecurity() が再実行され、新しいポリシーが反映される。

注意 RelationBuildRowSecurity() 単体には plancache の無効化処理はない。 プラン側の無効化は「クエリ Query に hasRowSecurity = true が立っているとき、 リレーションキャッシュ無効化イベントを契機に再計画される」仕組みで実現される （6.3 節）。

3.4　ポリシーの削除とロールからの除去

DROP POLICY は最終的に RemovePolicyById() に到達する。対象テーブルを AccessExclusiveLock で開き、CatalogTupleDelete() でタプルを削除したあと、CacheInvalidateRelcache() で無効化を発行する。 コメントが「relrowsecurity は単にポリシーの有無を示すフラグではなく、ユーザーが 明示的にセットしたものである」と述べているとおり、ポリシーが 0 件になっても RLS フラグ自体は自動では落とされない。すなわち、ポリシー全削除後の RLS 有効テーブルは default-deny の状態になる。

ロールが DROP された場合の動作も実装上重要である。 RemoveRoleFromObjectPolicy() は polroles 配列から該当ロール OID を 除いた新しい配列を作成し、heap_modify_tuple() + CatalogTupleUpdate() で書き戻す。再構築後の配列要素数が 0 ならば「ポリシー自体を破棄する」ことを示す false を返し、呼び出し側で RemovePolicyById() 相当の処理を行う。
