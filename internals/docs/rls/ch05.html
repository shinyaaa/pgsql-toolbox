第5章　ポリシー結合の意味論とコマンド別の挙動

本章では、テーブルに複数のポリシーが定義されているときの結合規則と、 SQL コマンド種別ごとの適用順を、rowsecurity.c の実装に即して整理する。 RLS の動作直感は「OR で広げ、AND で絞る」「足りなければ default-deny」の 2 点に集約される。

5.1　permissive と restrictive の結合

ポリシーには permissive（既定）と restrictive（AS RESTRICTIVE 指定）の 2 種があり、結合規則が異なる。add_security_quals() と add_with_check_options() における実装は次のとおり。

  • permissive 同士は OR で結合: 列挙したいずれかの permissive ポリシーが true を返せば行は許可される。
  • restrictive はそれぞれ独立に AND で適用: すべての restrictive ポリシーが true を返さなければ行は許可されない。
  • permissive が 0 件なら結果は常に拒否: 「最低 1 つは行を許可する側のポリシーが 必要」というのが基本原則である。

なお restrictive ポリシーは「ポリシー名でソートしてから 1 つずつ WithCheckOption として 追加する」実装になっており（sort_policies_by_name()）、 違反時のエラーメッセージにどのポリシーが原因かを安定して特定できるよう配慮されている。

5.2　コマンド別のポリシー収集

get_policies_for_relation() は対象テーブルの全ポリシーを走査し、polcmd と要求されたコマンドが一致するもののみを permissive / restrictive のリストに振り分ける。polcmd == '*'（ALL）は無条件で対象に含まれる。

ロール一致判定は check_role_for_policy() が行う。polroles[0] が ACL_ID_PUBLIC のときは即時 true を返し、 そうでなければ各 OID について has_privs_of_role() をチェックする。 ロール継承を含むので、ある親ロールに付与されたポリシーは継承元のメンバーにも適用される。

さらに、拡張モジュールが row_security_policy_hook_permissive / row_security_policy_hook_restrictive を介して追加のポリシーを差し込める。 ビルトインポリシーに後追いで連結され、restrictive 側は同様にポリシー名でソートされる。

5.3　default-deny の生成

permissive ポリシーが 1 件も収集されなかった場合、add_security_quals() は 常に FALSE を返す定数ノードを 1 つだけ securityQuals に追加する。

rowsecurity.c より 
/*
 * A permissive policy must exist for rows to be visible at all.
 * Therefore, if there were no permissive policies found, return a
 * single always-false clause.
 */
*securityQuals = lappend(*securityQuals,
                         makeConst(BOOLOID, -1, InvalidOid,
                                   sizeof(bool), BoolGetDatum(false),
                                   false, true));

これが「ポリシー未定義 + RLS 有効 = すべて見えない」という挙動の正体である。 add_with_check_options() も同様に、permissive ポリシーが無いときは FALSE な WithCheckOption を 1 つ追加し、INSERT/UPDATE が必ず失敗するようにする。

5.4　ON CONFLICT と MERGE の特殊処理

get_row_security_policies() の本体（L98 以降）は、 SELECT / UPDATE / DELETE / INSERT に加え、INSERT ... ON CONFLICT と MERGE の組み合わせを 漏れなく扱うために長くなっている。要点は次のとおり。

  • SELECT FOR UPDATE/SHARE: perminfo->requiredPerms に ACL_UPDATE が含まれるとき、まず UPDATE 用 USING ポリシーをかけてから SELECT 用ポリシーを重ねる。これにより、ロック取得対象が UPDATE 経由でも 可視である行に限定される。
  • UPDATE/DELETE と RETURNING: 行更新／削除の許可判定（UPDATE/DELETE の USING）に加え、 RETURNING や WHERE で列を参照する場合は SELECT 用 USING も適用される。
  • INSERT/UPDATE: INSERT の WITH CHECK と、UPDATE の WITH CHECK が add_with_check_options() 経由で挿入される。WITH CHECK が未指定なら QUAL_FOR_WCO マクロで USING が代用される（L823）。
  • INSERT ... ON CONFLICT DO SELECT/UPDATE: 競合行に対する WCO_RLS_CONFLICT_CHECK が追加され、行が見えない／更新できない場合は 静かに無視されるのではなくエラーになる。UPDATE 経路に進む場合はさらに WITH CHECK と SELECT の WCO が重ねられる。
  • MERGE: MERGE は単独のポリシーを持たず、UPDATE / DELETE / INSERT それぞれの ポリシーを「WCO」として束ねて検査する。USING 句も WCO_RLS_MERGE_UPDATE_CHECK / WCO_RLS_MERGE_DELETE_CHECK として WCO 化される点が他コマンドと異なる （L419 以降）。 ソース内コメントは「ON CONFLICT のように、対象行が RLS で許可されない場合は 暗黙にスキップするのではなくエラーにする」設計意図を明記している。
メモ 最適化ヒントとして、securityQuals はセキュリティバリアとして扱われ、 LEAKPROOF 関数だけが押し下げられる。詳細は6.4 節を参照。
