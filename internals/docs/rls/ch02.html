第2章　pg_policy カタログとデータ構造

ポリシーはディスク上ではシステムカタログ pg_policy に保存され、 メモリ上ではリレーションキャッシュ（relcache）に RowSecurityDesc として 付随する形で保持される。本章では、この 2 つの表現を行き来する経路を中心に、 RLS が依存するデータ構造をひととおり確認する。

2.1　pg_policy の物理レイアウト

pg_policy は 1 ポリシーにつき 1 行のシステムカタログで、定義は src/include/catalog/pg_policy.h に置かれている。BKI_LOOKUP マクロの存在から分かる通り、リレーション OID とロール OID への 参照を持つ、関係カタログである。

ポイントは次のとおりである。

  • polrelid: ポリシーが属するリレーションの OID。 pg_policy_polrelid_polname_index が (polrelid, polname) 上のユニークインデックスとして定義されており、 relcache 構築時のシーケンシャルスキャンを避け、また同一テーブル内のポリシー名重複を防ぐ。
  • polcmd: '*'（ALL）、'r'（SELECT）、 'a'（INSERT）、'w'（UPDATE）、'd'（DELETE）の 5 通り。
  • polpermissive: true なら permissive、false なら restrictive。 複数ポリシーの結合規則を分ける（5.1 節）。
  • polroles: 適用対象ロールの OID 配列。空配列は許されず、 PUBLIC（ACL_ID_PUBLIC == 0）が単独で入っているのが「全ロール対象」の意味になる。
  • polqual / polwithcheck: USING 句／WITH CHECK 句を パース・トランスフォーム済みのノードツリーへ変換し、nodeToString() で テキスト化したもの。読み出し時には stringToNode() で復元する （3.2 節）。

2.2　RowSecurityPolicy 構造体

カタログ上のタプルをそのまま使うのは不便なので、relcache 上では RowSecurityPolicy というメモリ構造体に変換される。

構造体定義（rowsecurity.h） 
typedef struct RowSecurityPolicy
{
    char       *policy_name;     /* Name of the policy */
    char        polcmd;          /* Type of command policy is for */
    ArrayType  *roles;           /* Array of roles policy is for */
    bool        permissive;      /* restrictive or permissive policy */
    Expr       *qual;            /* Expression to filter rows */
    Expr       *with_check_qual; /* Expression to limit rows allowed */
    bool        hassublinks;     /* If either expression has sublinks */
} RowSecurityPolicy;

カタログにおける pg_node_tree（テキスト）が Expr *（パースツリー）に 既にデシリアライズされている点が重要である。クエリ書き換え時の get_row_security_policies() からは、この Expr を copyObject() でコピーしてクエリツリーへ織り込むだけで済む （第4章）。

もう 1 つの注目点は hassublinks フラグである。これは USING / WITH CHECK 句に サブクエリ（SubLink）が含まれているかを事前に判定したキャッシュであり、 rewriter が「ポリシー内のサブクエリに対しても RIR ルールを発火させ、無限再帰を検知する」必要が あるかを高速に判定するために使われる（4.2 節）。

2.3　RowSecurityDesc と relcache への紐付け

リレーションに紐づくポリシー群は RowSecurityDesc でまとめられ、Relation->rd_rsdesc から辿れる。

構造体定義 
typedef struct RowSecurityDesc
{
    MemoryContext rscxt;     /* row security memory context */
    List         *policies;  /* list of row security policies */
} RowSecurityDesc;

rscxt は専用のメモリコンテキストで、relcache のフラッシュ時にここを丸ごと解放できるよう 分離されている。policies は RowSecurityPolicy * のリストで、 pg_policy のスキャン順、すなわち (polrelid, polname) インデックス順で並ぶ。

このロードを担当するのが RelationBuildRowSecurity() である。 概略は次のとおり。

  1. "row security descriptor" という名前で AllocSetContextCreate によりサブコンテキストを作る。
  2. pg_policy を polrelid でインデックススキャンする。
  3. 各タプルから polqual / polwithcheck を取り出し、stringToNode() で Expr へ復元する。
  4. checkExprHasSubLink() でサブリンクの有無を求め、hassublinks に格納する。
  5. 処理成功後にコンテキストを CacheMemoryContext の子に付け替え、Relation->rd_rsdesc にセットする。

途中でエラーが発生してもメモリリークが起きないよう、コンテキストの親付け替えは 「ロードが最後まで完走したあと」に行われている点が特徴である。

関連 RLS の有効性判定（pg_class.relrowsecurity の確認）と RelationBuildRowSecurity() の呼び出しは独立しており、 relrowsecurity = false のテーブルではこのロード自体が行われない （4.1 節）。
