第1章　行レベルセキュリティの概観

本章では PostgreSQL の行レベルセキュリティ（Row Level Security, RLS）の動機と 全体像を概観する。RLS は「テーブル単位ではなく行単位で、誰がどの行を見られる／更新できるか」を サーバ側で強制する仕組みであり、GRANT 系の特権モデルとは別の層で動作する。 具体的な実装はカタログ pg_policy・relcache・クエリ書き換え（rewrite）に分散しており、 後続の章でその内部を順に追っていく。

1.1　RLS とは何か

RLS は、テーブルに対して ポリシー（policy） と呼ばれる述語を定義し、 クエリ実行時にその述語をクエリツリーへ自動的に注入することで、行の可視性と 書き込み可否を制御する機能である。ALTER TABLE … ENABLE ROW LEVEL SECURITY によりテーブルでフラグを立て、CREATE POLICY でポリシーを定義する。 RLS が有効なテーブルへ通常ユーザーがアクセスすると、SELECT / UPDATE / DELETE では ポリシーの USING 句が、INSERT / UPDATE では WITH CHECK 句が 自動的に適用される。

実装ファイルの先頭コメントでは、RLS の役割は次のように要約されている （src/backend/rewrite/rowsecurity.c）。

ソースコメント（要約） 
Policies in PostgreSQL provide a mechanism to limit what records are
returned to a user and what records a user is permitted to add to a table.

Policies can be defined for specific roles, specific commands, or provided
by an extension.  Row security can also be enabled for a table without any
policies being explicitly defined, in which case a default-deny policy is
applied.

注目すべきは「ポリシーを 1 つも作らずに RLS を有効化した場合、暗黙の default-deny が 適用される」点である。実装上、これは add_security_quals() と add_with_check_options() の中で「permissive ポリシーが 1 つも見つからなければ常に FALSE な qual / WCO を 1 件だけ追加する」 という形で表現されている。RLS を有効化した直後は誰にも何も見えなくなるのが正しい挙動である。

1.2　列・テーブルレベル権限との関係

PostgreSQL の権限は大きく次の 3 層に分かれる。RLS はこのうち最も内側の「行レベル」を担当する。 図 1.1 に示すように、テーブルアクセスは外側から順に評価され、すべて通過した行だけが クエリ結果に含まれる。

GRANT 系の権限はカタログ pg_class.relacl や pg_attribute.attacl に ACL として保持され、パーサ／プランナで参照される。一方 RLS の述語は pg_policy に格納され、クエリ書き換え段階で対象テーブルの参照に対して合成される （詳細は第4章）。

メモ RLS はテーブルの所有者と BYPASSRLS 属性を持つロールに対して既定でバイパスされる。 所有者であっても RLS を効かせたい場合は ALTER TABLE … FORCE ROW LEVEL SECURITY を使う。 詳細は第6章を参照。

1.3　ポリシーの基本構成要素と最小例

各ポリシーは「名前」「対象テーブル」「対象コマンド」「対象ロール」「USING 句」「WITH CHECK 句」 「permissive / restrictive 区分」から成る。SQL 文では次のように記述する。

SQL 
CREATE TABLE accounts (
    id    int PRIMARY KEY,
    owner text NOT NULL,
    balance numeric NOT NULL
);

ALTER TABLE accounts ENABLE ROW LEVEL SECURITY;

CREATE POLICY accounts_owner ON accounts
    FOR ALL                       -- 対象コマンド
    TO PUBLIC                     -- 対象ロール
    USING (owner = current_user)  -- 既存行に対するフィルタ
    WITH CHECK (owner = current_user); -- 新規/更新行への制約

ポリシー定義は pg_policy 1 行に対応する。生成された行は次のように観察できる。

SQL 
SELECT polname, polcmd, polpermissive,
       pg_get_expr(polqual, polrelid)      AS using_expr,
       pg_get_expr(polwithcheck, polrelid) AS check_expr
FROM pg_policy
WHERE polrelid = 'accounts'::regclass;
実行結果 
    polname     | polcmd | polpermissive |       using_expr        |       check_expr
----------------+--------+---------------+-------------------------+-------------------------
 accounts_owner | *      | t             | (owner = CURRENT_USER)  | (owner = CURRENT_USER)

polcmd は対象コマンドを 1 文字で表現する。'*' は ALL、 'r' は SELECT、'a' は INSERT、'w' は UPDATE、 'd' は DELETE で、これは parse_policy_command() の対応表と一致する。

以降の章では、この 1 行のポリシーがどのようにカタログから読み出され （第2章）、DDL で書き込まれ （第3章）、クエリへ織り込まれ （第4章）、複数ポリシーが組み合わされる （第5章）かを順に見ていく。
