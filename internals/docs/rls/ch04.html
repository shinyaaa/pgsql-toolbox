第4章　クエリ書き換え時の RLS 適用

RLS の「効かせる側」の実装はクエリ書き換え（rewrite）段階に集中している。 本章では、書き換え器が範囲表エントリ（RTE）ごとに RLS を適用する全体経路と、 ポリシーを Query ノードに織り込むまでの流れを追う。

4.1　check_enable_rls による有効性判定

リレーションに対して RLS を適用すべきかは check_enable_rls() が一手に決定する。返り値は CheckEnableRlsResult の 3 値である。

  • RLS_NONE: そもそも RLS とは無関係。プランキャッシュも再評価不要。
  • RLS_NONE_ENV: 現環境では適用しないが、ロール変更や GUC 変更で結果が変わるため、 プランキャッシュには「RLS 依存」のフラグを立てる必要がある。
  • RLS_ENABLED: RLS を実際に注入する。プランキャッシュは無効化の対象。

典型的なバイパス条件は次の 3 つである。

  1. システムカタログ（relid < FirstNormalObjectId）には RLS を適用しない。 これはカタログ参照経路を壊さないためのガード。
  2. BYPASSRLS 属性を持つロール（スーパーユーザーは常に BYPASSRLS）。
  3. テーブル所有者が、FORCE ROW LEVEL SECURITY 未設定、または InNoForceRLSOperation() が真の場合。後者は外部キー制約のチェックなどで一時的に RLS を抑止するための文脈である。

バイパスケースであっても RLS_NONE_ENV を返すことで、後段の plancache が 「ロールが変わったら再計画が必要」と認識できるようにしている点が重要である （6.3 節）。

4.2　fireRIRrules から get_row_security_policies へ

RLS のクエリへの注入は、rewriter の RIR（Retrieve-In-Retrieve；ビュー展開などを行う） 処理を担当する fireRIRrules() の末尾で行われる。すなわち、ビュー展開や CTE 走査が終わってから、 各 RTE に対し RLS の述語を載せる。

rewriteHandler.c より 
foreach(lc, parsetree->rtable)
{
    RangeTblEntry *rte = (RangeTblEntry *) lfirst(lc);
    ...
    if (rte->rtekind != RTE_RELATION ||
        (rte->relkind != RELKIND_RELATION &&
         rte->relkind != RELKIND_PARTITIONED_TABLE))
        continue;

    rel = relation_open(rte->relid, NoLock);

    /* Fetch any new security quals that must be applied to this RTE. */
    get_row_security_policies(parsetree, rte, rt_index,
                              &securityQuals, &withCheckOptions,
                              &hasRowSecurity, &hasSubLinks);

RTE がリレーションでなければスキップする（サブクエリや関数 RTE には RLS は付かない）。 なお RTE_RELATION でも、relkind が通常テーブルとパーティションテーブル以外なら無視される。 ビューは「ビュー自体は relcache 上のフラグで RLS を持てない」が、ビュー展開の結果として 下位の RTE がテーブルになれば、その時点で RLS が掛かる仕組みである。

サブリンクを含むポリシーの場合、fireRIRrules は次の 2 つの追加処理を行う。

  1. activeRIRs リストに自テーブルが入っていないかを確認し、入っていれば 「infinite recursion detected in policy for relation」エラーを出す。 これはポリシー内サブクエリが自テーブルを参照することによる再帰展開を防ぐためである。
  2. acquireLocksOnSubLinks() でサブリンクが参照するリレーションをロックし、 fireRIRonSubLink で再帰的に RIR ルールを発火させる。 これはパース時点ではポリシー側のサブクエリにロックが取られていないためで、 RLS 経路で追加処理する必要がある。

4.3　securityQuals と withCheckOptions の注入

得られた securityQuals は対象 RTE の rte->securityQuals リストの 先頭に 連結される。これは既存のセキュリティバリアビュー由来の qual より RLS を優先させるためで、 rewriter のコメントに明記されている。

rewriteHandler.c より 
rte->securityQuals = list_concat(securityQuals, rte->securityQuals);
parsetree->withCheckOptions = list_concat(withCheckOptions,
                                          parsetree->withCheckOptions);

securityQuals は Query->rtable[i].securityQuals に、 withCheckOptions は Query->withCheckOptions に、それぞれ クエリツリー上の異なる場所に格納される。前者はスキャン時のフィルタとして働き、 後者は INSERT/UPDATE 直後の行に対する制約として executor が検査する。

WithCheckOption のレイアウトは parsenodes.h に定義されている。kind フィールドは WCO_RLS_INSERT_CHECK、WCO_RLS_UPDATE_CHECK、 WCO_RLS_CONFLICT_CHECK、WCO_RLS_MERGE_UPDATE_CHECK、 WCO_RLS_MERGE_DELETE_CHECK のいずれかで、エグゼキュータがどのタイミングで どのコマンドの WITH CHECK として扱うべきかを区別する。

最後に Query->hasRowSecurity が立つことで、plancache が「このプランは ロール・GUC・ポリシーの変化で再計画が必要」と判断できる （6.3 節）。

関連 ポリシー述語の中に他テーブルへの参照があるとき、setRuleCheckAsUser() によって checkAsUser をコピーしておくのも本経路の役割である （rowsecurity.c L537）。 これによりサブクエリ評価も「ポリシー実行者」の権限で行われる。
