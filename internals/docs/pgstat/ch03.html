第3章　共有メモリ上のデータ構造

統計データはプロセス間で共有されなければならない。この章では、累積統計システムの 共有メモリ表現の中心である PgStat_ShmemControl と、可変数統計を収める動的共有メモリ（DSM）上のハッシュテーブルの構造を解説する。

3.1　PgStat_ShmemControl と DSA

累積統計システムの共有メモリへの入口は PgStat_ShmemControl である。 図 3.1 に示すように、ここから動的共有メモリ領域（DSA）、可変数統計のハッシュテーブル、 固定数統計の実体すべてに到達できる。

固定数統計（archiver〜wal）は PgStat_ShmemControl に直接埋め込まれている。 一方、可変数統計は DSA 上に確保され、hash_handle から到達する dshash テーブルで管理される。 各バックエンドは pgstat_attach_shmem() で dsa_attach_in_place() により DSA へアタッチし、dshash_attach() で ハッシュテーブルへ接続する。アタッチ後の状態はバックエンドローカルの PgStat_LocalState （グローバル変数 pgStatLocal）に保持される。

3.2　dshash と PgStat_HashKey

可変数統計は dshash（動的共有メモリ向けハッシュテーブル）に格納される。 各エントリは PgStat_HashKey で識別される。キーは3つのフィールドからなる。

図 3.2 のとおり、kind・dboid・objid の組で1つの統計オブジェクトを 一意に指す。dboid は共有オブジェクトでは InvalidOid となる。 たとえば「データベース A のテーブル B のリレーション統計」は (RELATION, A の OID, B の OID) で表される。 このキーはパディングを持たないことが前提で、 ソース上でも StaticAssertDecl によりサイズが各フィールドの和と一致することを コンパイル時に検査している。実行時はこのキーで統計をアドレッシングするが、 ディスクへシリアライズする際には（レプリケーションスロット統計のように）より広い識別子を 使う場合がある。

3.3　HashEntry と Common ヘッダ

dshash のエントリ自体は統計データを直接含まない。図 3.3 に示すように、 PgStatShared_HashEntry は統計本体への dsa_pointer（body）を持つだけで、 本体は別途 DSA から確保される。この分離により、種別ごとに大きさの異なる統計を 同じハッシュテーブルへ格納しても PgStatShared_HashEntry の領域を無駄にしない。

統計本体は必ず PgStatShared_Common で始まる。これは妥当性検査用の magic（0xdeadbeef）と、 データ部分を保護する LWLock を持つ共通ヘッダである。 その後ろに種別固有の統計（例: PgStatShared_Relation は PgStat_StatTabEntry）が続く。 dropped・refcount・generation はエントリのライフタイム管理に使われ、 これは第4章で詳述する。

メモ共有メモリに置かれる構造体は PgStatShared 接頭辞を持ち、 バックエンドローカルやディスク向けの構造体（PgStat 接頭辞）と区別される。

3.4　固定数統計の格納

固定数統計は dshash を介さず、PgStat_ShmemControl に直接埋め込まれた 専用構造体に格納される。たとえば PgStatShared_Checkpointer や PgStatShared_Wal などである。これらの多くは 単一ライタの changecount 機構を用い、ロックなしで低オーバヘッドの 書き込みを実現している（読み取り側は changecount の前後比較で一貫性を確認する）。

ただしリセットは別バックエンドが書き込む必要があるため、リセット時に共有統計を直接 書き換えるのではなく、現在のカウンタ値を reset_offset として控え、 読み取り時に差し引く方式を採る。 stat_reset_timestamp だけは別バックエンドが書くため ->lock で保護される。 I/O 統計（PgStatShared_IO）はバックエンド種別ごとにロック配列を持つなど、 種別に応じた最適化が施されている。
