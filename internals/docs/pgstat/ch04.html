第4章　ローカル参照とペンディング更新

共有ハッシュテーブルへ毎回アクセスすると、ロック競合がシステム全体の性能を律速しかねない。 累積統計システムはこれを避けるため、バックエンドローカルのキャッシュ層を 共有ハッシュの手前に置く。この章では、そのキャッシュとペンディング更新の仕組み、 そしてエントリのライフタイム管理を解説する。

4.1　二段構えのハッシュ

各バックエンドは、共有ハッシュへの参照を保持するローカルハッシュ pgStatEntryRefHash を持つ。図 4.1 に示すように、 統計エントリを引くときはまずローカルハッシュを探し、見つかればそのまま使う。 見つからないときだけ共有ハッシュへアクセスする。

このキャッシュ参照の取得は pgstat_get_entry_ref() が担う。まず pgstat_get_entry_ref_cached() でローカルハッシュを引き、 ヒットすればロックを一切取らずに参照を返す。ミスのときだけ dshash_find() で 共有ハッシュを引き、必要なら新規エントリを作成する。 大半の統計操作が共有ハッシュに触れずに済むため、競合が大幅に減る。

4.2　PgStat_EntryRef とローカルキャッシュ

ローカルハッシュのエントリが指す PgStat_EntryRef は、共有エントリへの参照とペンディングデータを束ねる構造体である（図 4.2）。

shared_entry は共有ハッシュのエントリを指し、shared_stats は 統計本体（body をローカルアドレスに解決したもの）を指す。 dsa_get_address() の繰り返し呼び出しを避けるためにローカルアドレスを キャッシュしている点がポイントである。pending は未反映の統計更新を溜める領域、 pending_node は後述のペンディングリストへの連結に使われる。

4.3　refcount と generation

統計エントリのライフタイムは、ハッシュ表のエントリ寿命とは別に管理される。 バックエンドがロックを持たずに統計本体を指せるようにするためである。 PgStatShared_HashEntry の refcount と generation がこれを担う。

refcount はエントリへの参照数を数える原子変数である。エントリが ドロップされていない限り「ドロップ不可」を表す 1 が加算され、さらに参照を持つ 各バックエンドが 1 ずつ加算する。DROP TABLE 等で統計が落とされても、 参照がすべて解放されるまでエントリのメモリは解放されない。 pgstat_init_entry() は refcount を 1、generation を 0 で初期化する。

generation はエントリが再利用された回数である。ドロップ済みエントリが pgstat_reinit_entry() で再初期化されるたびに 1 増える。これにより、古い世代を指したままの ローカル参照は PgStat_EntryRef->generation との不一致で「陳腐化」を検知できる。 参照が残って解放できなかったドロップは、gc_request_count を契機に 各バックエンドが pgstat_gc_entry_refs() を実行して回収する。

4.4　ペンディング更新リスト

統計更新の大半は、まず各プロセス内にペンディングエントリとして蓄積され、 後でまとめて共有メモリへフラッシュされる。これにより個々の統計エントリへの競合が 実質的に排除される。ペンディングデータを持つエントリはすべて、グローバルな pgStatPending リストに連結される（図 4.3）。

ペンディングエントリの準備は pgstat_prep_pending_entry() が行い、PgStat_EntryRef->pending を確保してリストへ追加する。 新規のペンディングは必ずリスト末尾に足す決まりで、これは後述のフラッシュ処理が 反復中に追加された分も漏れなく処理できるようにするためである。 ペンディングの蓄積から共有メモリへのフラッシュまでの流れは、次章で扱う。
