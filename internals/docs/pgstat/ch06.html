第6章　読み取りとスナップショット

統計の読み取りには、更新とは別の一貫性の問題がある。同じクエリの中で 複数の統計を読むとき、それらが整合した時点のものであってほしいことがあるからだ。 pgstat.c はこれを stats_fetch_consistency という GUC で制御する。 この章では読み取り経路とスナップショットの仕組みを解説する。

6.1　stats_fetch_consistency の3モード

stats_fetch_consistency は読み取りの一貫性レベルを3段階で選べる（図 6.1）。 デフォルトは cache である。

none はキャッシュせず毎回共有メモリから読むため最も軽量だが、 同一トランザクション内でも値が変わり得る。cache は一度読んだ統計を トランザクション内で固定する。snapshot は初回アクセス時に 全統計の一貫したスナップショットを作る。キャッシュ・スナップショットはともに PgStat_Snapshot （pgStatLocal.snapshot）へ格納される。

6.2　pgstat_fetch_entry() の流れ

可変数統計の読み取り入口は pgstat_fetch_entry() である。図 6.2 に処理の流れを示す。

snapshot モードなら先に全体スナップショットを構築し、以降はそこから引く。 cache モードではキャッシュを引き、ヒットすればそれを返す。ミス時は 共有エントリをロック下でローカルへコピーし、none 以外ではキャッシュへ 保存する。none モードは呼び出し元のメモリコンテキストへ確保し、 may_free を真にして「呼び出し元が pfree() してよい」ことを伝える。

6.3　スナップショットの構築

snapshot モードの一貫したスナップショットは pgstat_build_snapshot() が構築する。dshash 全体を走査し、対象となる可変数統計をローカルへコピーする。 このとき、種別が accessed_across_databases でない限り、 現在のデータベースまたはデータベース非依存のオブジェクトだけを含める。

固定数統計は pgstat_build_snapshot_fixed() が種別ごとの snapshot_cb を呼んで構築する。 fixed_valid[] / custom_valid[] フラグで構築済みかを管理し、 cache モードでは一度構築すれば再構築しない。 スナップショット採取時刻は snapshot_timestamp に記録され、 pgstat_get_stat_snapshot_timestamp() で参照できる。

6.4　スナップショットの破棄

キャッシュ／スナップショットはトランザクション境界などで破棄される。 pgstat_clear_snapshot() がスナップショット用メモリコンテキストを解放し、mode や各 valid フラグを リセットする。これにより次回アクセス時には最新の統計が読み直される。

stats_fetch_consistency が変更された場合は、次回利用前に既存スナップショットを 強制クリアするフラグ（force_stats_snapshot_clear）が立つ。 こうしてモード変更が確実に反映される。

関連共有メモリ上の統計表現は第3章、永続化は第7章を参照。
