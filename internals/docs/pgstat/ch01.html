第1章　累積統計システムの全体像

PostgreSQL は稼働中にさまざまな統計を蓄積する。テーブルが何回スキャンされたか、 チェックポイントが何回起きたか、WAL を何バイト書いたか――こうした情報は 累積統計システム（cumulative statistics system）が収集し、 pg_stat_user_tables や pg_stat_bgwriter といった 統計ビュー群を通じて参照できる。この章ではまず、その基盤コードである pgstat.c が何を担い、システム全体がどう組み立てられているかを俯瞰する。

1.1　累積統計システムとは

累積統計システムは、異なる種類のオブジェクトについて統計を累積する仕組みである。 ここでいう「累積」とは、カウンタが単調に増え続け（リセットされるまで）、 個々のセッションをまたいで合算されることを指す。プランナが使うテーブルの行数推定 （pg_statistic）とは別物であり、こちらはあくまで「いつ・何回・どれだけ」を数える 稼働統計である。

統計はサーバ起動時に startup プロセスがディスクから読み込み、 通常シャットダウン時に checkpointer プロセスが書き出す。 直前にクラッシュがあった場合は、整合性が保証できないためすべての統計が破棄される。 この読み込み・書き出し・破棄の制御も pgstat.c の責務である。

1.2　固定数統計と可変数統計

pgstat.c は統計を大きく2つに分類する。図 1.1 に示すように、 オブジェクト数があらかじめ決まっている固定数統計（fixed-numbered stats）と、 実行時に増減する可変数統計（variable-numbered stats）である。

固定数統計は、たとえば checkpointer のようにシステム全体で1つしか存在しない （あるいは I/O 統計のようにバックエンド種別ごとに固定個の）統計である。 これらは通常の（非動的な）共有メモリにそのまま置かれる。一方、テーブルや関数の統計は オブジェクトの作成・削除に伴って数が変わるため、動的共有メモリ（DSM）上の ハッシュテーブルに格納される。それぞれの内部表現は第3章で詳しく扱う。

1.3　ファイル構成と pgstat.c の役割

統計処理は規模が大きいため、複数ファイルに分割されている。pgstat.c のファイル先頭コメントは この分割方針を明記している。pgstat.c は全体を束ねるインフラ層であり、 特定の統計種別に依存するコードはできるだけ各種別専用ファイルへ追い出す方針が貫かれている。

図 1.2 のとおり、共有メモリとハッシュエントリの保守はほぼ pgstat_shmem.c に、トランザクションに連動した統計エントリの作成・削除は pgstat_xact.c に置かれる。database・relation といった各統計種別は、 それぞれ pgstat_database.c・pgstat_relation.c のように専用ファイルを持つ。 本ドキュメントが主に扱うのはインフラ層、とりわけ pgstat.c である。

1.4　更新・読み取り・永続化の全体像

pgstat.c が担う処理は、大きく「更新（フラッシュ）」「読み取り（スナップショット）」 「永続化」の3系統に分けられる。図 1.3 にその全体像を示す。

各バックエンドは統計更新をまずローカルに溜め込み（pending）、コミット後などに pgstat_report_stat() で共有メモリへフラッシュする。読み取り側は pgstat_fetch_entry() を通じて統計を取得し、stats_fetch_consistency の設定に応じてスナップショットへ キャッシュする。永続化は startup プロセスによる読み込みと checkpointer による書き出しが担う。 以降の章では、この全体像を構成要素ごとに掘り下げていく。
