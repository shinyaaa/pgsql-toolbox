第4章　PGPROCのライフサイクル

PGPROC は postmaster 起動時に共有メモリ上へまとめて確保され、各プロセスの 起動・終了に応じて貸し出し・返却される。本章ではこのライフサイクルを、 確保 → 割り当て → ProcArray 登録 → 解放の順に追う。

4.1　共有メモリの確保

必要な PGPROC の総数は ProcGlobalShmemRequest() で計算される。 TotalProcs = MaxBackends + NUM_AUXILIARY_PROCS + max_prepared_xacts であり、PGPROC 本体に加えて密配列 3 本ぶんの共有メモリも同時に予約される。

実際の初期化は ProcGlobalShmemInit() が行う。この関数は、フリーリストの初期化、allProcs と 密配列の領域切り出し、バックエンドあたりのセマフォ作成、そして各 PGPROC の 共通初期化（ファストパス配列の割り当て、ラッチ・セマフォ・fpInfoLock の セットアップ、myProcLocks[] の初期化）を行う。セマフォを起動時に まとめて確保するのは、カーネルのセマフォ上限超過を「あとで」ではなく「すぐに」 検知させるためである。

関連各 PGPROC はこのループの末尾でいずれかのフリーリストへ push される（3.2 節）。プリペアドトランザクション用の PGPROC だけは TwoPhaseShmemInit() が別途フリーリストに繋ぐ。

4.2　PGPROC の割り当て

バックエンドは起動時に InitProcess() を呼び、自分の種別に対応するフリーリストから PGPROC を 1 つ取り出して MyProc にセットする。リストが空なら 「sorry, too many clients already」エラーになる ― これが接続数上限を 検知する地点の一つである。

proc.c — InitProcess() のフリーリスト選択（要約） 
if (AmAutoVacuumWorkerProcess() || AmSpecialWorkerProcess())
    procgloballist = &ProcGlobal->autovacFreeProcs;
else if (AmBackgroundWorkerProcess())
    procgloballist = &ProcGlobal->bgworkerFreeProcs;
else if (AmWalSenderProcess())
    procgloballist = &ProcGlobal->walsenderFreeProcs;
else
    procgloballist = &ProcGlobal->freeProcs;
...
MyProc = dlist_container(PGPROC, freeProcsLink,
                         dlist_pop_head_node(procgloballist));
MyProcNumber = GetNumberFromPGProc(MyProc);

PGPROC を取得した後、InitProcess() は xid、xmin、 pid、vxid などほぼ全フィールドを初期値にリセットし、 共有ラッチの所有権を取得して（OwnLatch → SwitchToSharedLatch）、 終了時クリーンアップとして ProcKill を登録する。補助プロセスは 代わりに InitAuxiliaryProcess() を呼び、フリーリストではなく線形探索で空きスロットを得る。

4.3　ProcArray への登録

PGPROC を確保しただけでは、まだ他プロセスから「実行中トランザクション」として 見えない。InitProcessPhase2() が ProcArrayAdd() を呼び、PGPROC を共有 ProcArray に追加して初めて可視になる。 この関数は ProcArrayLock と XidGenLock の両方を排他で取得する （この順序が重要）。

ProcArrayAdd は、参照局所性を高めるため配列を PGPROC ポインタ順にソート状態で 保つ。挿入位置を見つけたら、それ以降の pgprocnos と 3 本の密配列を memmove で 1 つずらし、新しいエントリを書き込み、後続すべての pgxactoff を振り直す。これが「ProcArrayAdd/Remove で pgxactoff が 変化する」（3.3 節の注意）の正体である。 ロックは取得と逆順（XidGenLock → ProcArrayLock）で解放される。

4.4　PGPROC の解放

バックエンド終了時には、登録の逆順で後始末が行われる。まず RemoveProcFromArray（内部で ProcArrayRemove()）が PGPROC を ProcArray から外し、 続いて ProcKill() が PGPROC をフリーリストへ返す。

ProcKill() は、保持中の LWLock の解放、同期レプリケーション リストからの離脱、共有ラッチの返却（DisownLatch）などを行う。 特に注意深いのがロックグループからの離脱である。自分が グループのリーダーかフォロワーか、最後のメンバーかどうかに応じて、自分自身と リーダーのどちらをフリーリストへ戻すかを決める。リーダーが先に抜けても PGPROC は最後のフォロワーが返すまで解放されない。

注意DisownLatch() は PGPROC がフリーリストへ 載るより前に行わなければならない。さもないと、空いたスロットを pop した新しい バックエンドが OwnLatch() で「まだ所有されているラッチ」に出くわして PANIC する。解放処理の順序にはこうした制約が織り込まれている。
