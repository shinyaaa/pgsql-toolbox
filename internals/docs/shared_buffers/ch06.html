第6章　観察と運用

最終章では、これまで見てきた共有バッファの内部状態を実際に観察する手段と、 shared_buffers のサイズを決めるうえでの考え方を扱う。観察には標準で同梱される 拡張 pg_buffercache と、システムビュー pg_shmem_allocations が役立つ。

6.1　pg_buffercache でバッファプールを覗く

拡張 pg_buffercache を導入すると、共有バッファの各スロットの状態（タグ・ダーティ・ ピン数・使用回数）を SQL で一覧できる。これは第3章・第4章で見た BufferDesc の フィールドをほぼそのまま表に展開したものである。

SQL 
CREATE EXTENSION IF NOT EXISTS pg_buffercache;

-- 何件のバッファが使われ、うちダーティはいくつか
SELECT count(*) AS total,
       count(*) FILTER (WHERE relfilenode IS NOT NULL) AS used,
       count(*) FILTER (WHERE isdirty) AS dirty
FROM pg_buffercache;
実行結果 
 total | used | dirty
-------+------+-------
 16384 |  142 |     7

total が NBuffers（既定 16384）と一致する点に注目してほしい。 バッファ単位で usagecount（使用回数、第3章）も見える。クロックスイープが 参照するこの値が、ホットなページで高くなることを確認できる。

SQL 
SELECT c.relname, count(*) AS buffers,
       round(avg(b.usagecount), 1) AS avg_usage
FROM pg_buffercache b
JOIN pg_class c ON b.relfilenode = pg_relation_filenode(c.oid)
WHERE b.reldatabase = (SELECT oid FROM pg_database
                       WHERE datname = current_database())
GROUP BY c.relname
ORDER BY buffers DESC
LIMIT 5;
実行結果 
   relname    | buffers | avg_usage
--------------+---------+-----------
 accounts     |      88 |       4.2
 accounts_pkey|      31 |       3.0
 tellers      |       5 |       5.0

6.2　共有メモリの割り当てを確認する

第2章で見た各配列が共有メモリ上で実際にどれだけの領域を占めているかは、 pg_shmem_allocations で確認できる。BufferManagerShmemRequest() が 申告した名前（"Buffer Blocks" など）がそのまま行として現れる。

SQL 
SELECT name, pg_size_pretty(allocated_size) AS size
FROM pg_shmem_allocations
WHERE name LIKE 'Buffer%' OR name = 'Buffer Descriptors'
ORDER BY allocated_size DESC;
実行結果（shared_buffers = 128MB の例） 
         name          |  size
-----------------------+--------
 Buffer Blocks         | 128 MB
 Buffer Descriptors    | 1024 kB
 Buffer IO Condition Variables | 256 kB
 Checkpoint BufferIds  | 320 kB

"Buffer Blocks" が shared_buffers とほぼ等しく、これがページ本体の領域である。 残りは記述子や条件変数などの管理領域で、いずれも NBuffers に比例する。 shared_buffers を増やすと、本体だけでなくこれらの付随領域も増える点を 見積もりに入れておくとよい。

6.3　サイズ設定の指針と注意

shared_buffers のサイズ選定は、本ドキュメントで見た仕組みから次のように整理できる。

  • 再起動が必要：PGC_POSTMASTER のため、変更はサーバ再起動でのみ反映される（第1章）。
  • 大きすぎる弊害：物理メモリを圧迫し、OS のページキャッシュと二重キャッシュになりうる。クロックスイープが一周するコストや、チェックポイント時の書き出し量も増える。
  • 小さすぎる弊害：ヒット率が下がり、置換のたびにダーティページの書き出し（第5章）やディスク読み込みが頻発する。
  • 付随コスト：記述子・バッファテーブルなどの管理領域も NBuffers に比例して増える（6.2）。
注意一般によく言われる「物理メモリの 25% 程度から始める」という目安は 出発点にすぎない。実際には pg_buffercache によるヒット状況や使用回数の分布、 ワークロードの作業集合（working set）の大きさを観察し、計測に基づいて調整するのが望ましい。

本ドキュメントでは、shared_buffers（=NBuffers）が共有メモリ上の バッファプールを定めること、そのプールが記述子・ページ本体・バッファテーブルといった 構造から成ること、そしてページ探索とクロックスイープによる置換がどう動くかを、 PostgreSQL 19beta1 のソースに即して見てきた。共有バッファはバッファマネージャという より大きな仕組みの中核であり、WAL・チェックポイント・バックグラウンドライタとの連携を さらに追うことで、PostgreSQL の I/O 全体像の理解が深まるだろう。
