第3章　バッファ記述子 BufferDesc

バッファ記述子（buffer descriptor）は、1つのバッファスロットのメタデータを保持する構造体 BufferDesc である。どのページを保持しているか、いま誰かが使用中か、内容は変更されたか といった情報をすべてここに集約する。本章では BufferDesc の各フィールドと、その大半を 占める state 変数のビット割り当てを見る。定義は buf_internals.h にある。

3.1　BufferDesc の構造

BufferDesc は図 3.1 のフィールドを持つ。構造体全体は、性能のため一般的な CPU の キャッシュライン（64 バイト）に収まるよう注意深く設計されている。実際、配列要素は BufferDescPadded としてキャッシュライン境界に整列される。

  • tag：このバッファが保持しているディスクページの識別子（BufferTag）。第4章で詳述する。タグの変更にはバッファヘッダロックが必要。
  • buf_id：バッファの添字（0始まり）。初期化後は不変なのでロック不要で参照できる。
  • state：フラグ・ピン数（refcount）・使用回数（usagecount）・コンテンツロックを1つの 64bit 値に詰め込んだ状態変数。アトミック操作で更新する。次節で詳説。
  • wait_backend_pgprocno：排他的なピン解放を待っているバックエンドの番号（BM_PIN_COUNT_WAITER と併用）。
  • io_wref：非同期 I/O（AIO）が進行中のときに設定される待機参照。
  • lock_waiters：コンテンツロックを待っている PGPROC のリスト。バッファヘッダスピンロックで保護される。
メモPostgreSQL 19 では、以前は別の LWLock で実装していた コンテンツロック（ページ内容の読み書きを保護するロック）が state に統合された。 これにより「ピンを外しつつアンロックする」といった操作を1回のアトミック操作で行えるなど、 いくつかの最適化が可能になっている。

3.2　state に詰め込まれた状態

state は pg_atomic_uint64（64bit）で、図 3.2 のように複数の情報を ビットフィールドとして格納する。1つの値にまとめることで、たとえばピン数の増減を バッファヘッダをロックせずに CAS（compare-and-swap）ループだけで行える。

  • refcount（18bit）：このバッファをピン（pin）しているプロセス数。ピンが1以上のバッファは置換対象にできない。取り出しには BUF_STATE_GET_REFCOUNT() を使う。
  • usagecount（4bit）：直近でどれだけ使われたかを表すカウンタ。クロックスイープ（第5章）で参照され、最大値は BM_MAX_USAGE_COUNT = 5。
  • flags（12bit）：ダーティか、有効か、タグが割り当て済みかなどを表す各種フラグ（次節）。
  • lock（20bit）：コンテンツロックの状態。共有ロック数（最大 MAX_BACKENDS）と、共有排他・排他のビットからなる。

usagecount の上限が小さい点には理由がある。ソースのコメント （buf_internals.h 付近） によれば、usagecount を NBuffers 並みに大きくすれば LRU に近い挙動になるが、 空きバッファを見つけるのにクロックの針が最大で BM_MAX_USAGE_COUNT+1 周する必要があり、 大きすぎると探索コストが増す。精度と速度のトレードオフとして 5 が選ばれている。

3.3　フラグ一覧

flags 部分には、図 3.2 の12ビットに対応する BM_* フラグが格納される。主なものを表に示す。 これらは BufferAlloc() やチェックポイント、I/O 処理の各所で確認・設定される。

buf_internals.h（主なフラグ） 
BM_LOCKED            バッファヘッダがロック中
BM_DIRTY             ページが変更済み（書き出しが必要）
BM_VALID             ページ内容が有効（読み込み済み）
BM_TAG_VALID         タグが割り当て済み（=バッファテーブルに登録あり）
BM_IO_IN_PROGRESS    読み書き I/O が進行中
BM_IO_ERROR          直前の I/O が失敗
BM_PIN_COUNT_WAITER  単独ピンの解放待ちプロセスあり
BM_CHECKPOINT_NEEDED チェックポイントで書き出す必要あり
BM_PERMANENT         永続バッファ（UNLOGGED でない）

とくに重要なのが BM_DIRTY と BM_VALID である。BM_DIRTY が立った バッファは、ディスクの内容より新しい変更を保持しており、置換やチェックポイントの前に 書き出さなければならない。BM_TAG_VALID は、そのバッファがバッファテーブル （第4章）にエントリを持つことを意味する。図 3.3 に、バッファ1スロットがたどる代表的な 状態遷移を示す。

記述子の構造を把握したところで、次章では tag フィールド、すなわちバッファが 「どのページを保持しているか」を表す BufferTag と、タグからバッファを引くための バッファテーブルを見る。
