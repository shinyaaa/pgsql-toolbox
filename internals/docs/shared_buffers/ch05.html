第5章　ページ探索と置換アルゴリズム

これまでに見たデータ構造（記述子・タグ・バッファテーブル）が、実際のページ読み込みで どう協調するかを本章で追う。あるページがバッファにあればそれを返し（ヒット）、無ければ 空きスロットを確保してディスクから読み込む（ミス）。空きが無ければ既存のページを 追い出す必要があり、その犠牲を選ぶのがクロックスイープである。

5.1　ReadBuffer の流れ

バックエンドがページを要求する入口は ReadBuffer() / ReadBufferExtended() で、 実体は ReadBuffer_common() に集約される。要求は最終的に BufferAlloc() へ渡され、ヒット／ミスの判定と必要なスロット確保が行われる。図 5.1 に全体像を示す。

5.2　BufferAlloc — 探索と登録

BufferAlloc() はまず要求ページのタグを作り、ハッシュ値とパーティションロックを 求める（第4章）。そのうえで図 5.2 の手順を踏む。要点は、まず共有ロックで素早く存在確認し、 ミス時のみ犠牲バッファを確保して排他ロックで登録する点である。

ヒット時は PinBuffer() でピンを立て、マッピングロックを解放して即座に返す。 ミス時は GetVictimBuffer() （bufmgr.c） で置換先（犠牲バッファ）を1つ確保する。確保後、排他ロックのもとで BufTableInsert() を 試みるが、その間に別バックエンドが同じページを登録してしまうこともある。その場合は確保した 犠牲バッファを返却し、既存のものを使う（図 5.2 の「衝突」分岐）。ロックを保持したまま ディスク I/O をしないこの設計が、競合下でも高い並行性を保つ鍵である。

メモ犠牲バッファが BM_DIRTY だった場合、再利用する前にその内容を ディスクへ書き出す必要がある。GetVictimBuffer() はこの書き出しも面倒を見る。したがって 共有バッファが小さくダーティページが多いと、読み込みのたびに書き出しが誘発され遅くなりうる。

5.3　クロックスイープによる犠牲選択

置換先を選ぶアルゴリズムがクロックスイープ（clock sweep）である。実装は StrategyGetBuffer() にある。全バッファを円環に見立て、針（nextVictimBuffer）を回しながら、各バッファの usagecount を1ずつ減らしていく。ピンされておらず usagecount が 0 になった バッファが犠牲として選ばれる。図 5.3 にこの動作を示す。

針を進めるのは ClockSweepTick() で、nextVictimBuffer をアトミックに加算し NBuffers で割った剰余を次の候補とする。 バッファはアクセスされるたびに usagecount が増える（上限 BM_MAX_USAGE_COUNT = 5） ため、よく使われるページほど針が来ても生き残りやすい。これにより、おおまかに「最近よく 使われたページを残す」LRU 的な挙動を、軽量に近似する。

関連PostgreSQL 18 以降、かつて存在したフリーリスト （未使用バッファの連結リスト）は廃止され、空きバッファの取得もクロックスイープに一本化された。 本バージョン（19beta1）の BufferDesc に freeNext フィールドが無く、 StrategyControl も針（nextVictimBuffer）と周回数（completePasses）を 中心とした構造になっているのはそのためである。

もし全バッファがピンされていて一周しても犠牲が見つからない場合、StrategyGetBuffer() は 無限ループを避けるため "no unpinned buffers available" のエラーを発生させる。通常の 運用でこの状態に陥ることはまず無いが、極端に小さい shared_buffers と高い並行度の 組み合わせでは理論上あり得る。次章では、こうした内部状態を実際に観察する方法を扱う。
