第1章　shared_buffers とは

shared_buffers は、PostgreSQL がディスク上のページ（既定 8KB）を メモリ上にキャッシュするために確保する共有バッファプールの大きさを決める 設定パラメータである。すべてのバックエンドプロセスはこの共有メモリ領域を通じてテーブルや インデックスのページを読み書きするため、shared_buffers はインスタンス全体の I/O 効率を左右する最重要パラメータのひとつである。

本章では、まず共有バッファが果たす役割を概観し、続いて shared_buffers が PostgreSQL 内部でどのように定義され（GUC 変数 NBuffers）、共有メモリのどこに 位置づけられるかを確認する。バッファプールの内部データ構造そのものは第2章以降で扱う。

1.1　共有バッファの役割

PostgreSQL はテーブルやインデックスのデータを 8KB 単位のページ（ブロック）で管理する。 バックエンドがあるページを必要とするとき、毎回ディスクへアクセスしていては遅い。そこで、 一度読み込んだページを共有メモリ上に保持し、次回以降はメモリから直接返す仕組みを持つ。 この共有メモリ上のキャッシュ領域が共有バッファプールであり、その管理を担うサブシステムを バッファマネージャ（buffer manager）と呼ぶ。

共有バッファは図 1.1 に示すように、バックエンドとストレージの間に位置する。読み込み時は まず共有バッファを探索し（キャッシュヒット）、なければストレージから読み込んで空きスロットへ 格納する。書き込みも一旦共有バッファ上のページを変更（ダーティ化）するだけで完了し、 実際のディスクへの反映はチェックポイントやバックグラウンドライタが後で行う。

共有バッファのおかげで、ホットなページへの繰り返しアクセスはディスク I/O を伴わずに済む。 一方で、確保したメモリは PostgreSQL 起動中ずっと占有されるため、物理メモリとの兼ね合いで 適切な大きさを選ぶ必要がある。サイズ設定の指針は第6章で扱う。

1.2　GUC パラメータとしての定義

shared_buffers は他の設定パラメータと同様、GUC（Grand Unified Configuration） として定義されている。PostgreSQL 19 では GUC 定義はデータファイル guc_parameters.dat に記述され、ビルド時にテーブルが生成される。shared_buffers の定義は次のとおりである。

guc_parameters.dat（抜粋） 
{ name => 'shared_buffers', type => 'int', context => 'PGC_POSTMASTER', group => 'RESOURCES_MEM',
  short_desc => 'Sets the number of shared memory buffers used by the server.',
  flags => 'GUC_UNIT_BLOCKS',
  variable => 'NBuffers',
  boot_val => '16384',
  min => '16',
  max => 'INT_MAX / 2',
},

この定義から、shared_buffers の性質が読み取れる。

  • type = int / variable = NBuffers：整数型で、値はC変数 NBuffers に格納される。
  • context = PGC_POSTMASTER：postmaster 起動時にのみ設定可能で、稼働中の変更にはサーバ再起動が必要である。
  • flags = GUC_UNIT_BLOCKS：値の単位はブロック（既定 8KB）であり、128MB のように単位付きで指定すると内部でブロック数へ換算される。
  • boot_val = 16384：既定値は 16384 ブロック、すなわち 16384 × 8KB = 128MB。
  • min = 16：最小16ブロック。max = INT_MAX / 2：上限はおよそ 10 億ブロック。
  • group = RESOURCES_MEM：「資源／メモリ」カテゴリに属する。
メモcontext が PGC_POSTMASTER であることは重要である。共有バッファプールは 起動時に一括して共有メモリ上に確保され、その後サイズを変えられない。postgresql.conf で 値を変更しても、再起動するまで反映されない。

1.3　NBuffers と単位の換算

設定値の実体は int NBuffers という大域変数である。宣言は miscadmin.h にあり、バッファマネージャ全体がこの値を「バッファの個数」として参照する。たとえば後述する バッファ記述子配列やバッファブロック配列は、いずれも要素数 NBuffers で確保される。

shared_buffers が単位付き（バイト）で表示・指定されるのに対し、NBuffers は 純粋な個数である。両者は図 1.2 の関係で換算される。ブロックサイズ BLCKSZ は ビルド時定数で、既定では 8192 バイトである。

設定値は pg_settings や SHOW で確認できる。shared_buffers は ブロック単位だが、表示は人間に読みやすい単位へ整形される。

SQL 
SHOW shared_buffers;
SELECT setting, unit FROM pg_settings WHERE name = 'shared_buffers';
実行結果 
 shared_buffers
----------------
 128MB

 setting | unit
---------+------
 16384   | 8kB

pg_settings では setting が 16384、unit が 8kB と表示され、 内部の NBuffers の値そのものが見えている。16384 × 8kB = 128MB が SHOW の表示と一致することが確認できる。

1.4　共有メモリ上の位置づけ

PostgreSQL は起動時に1つの大きな共有メモリセグメントを確保し、その中に各サブシステムが 必要とする領域を切り出していく。共有バッファプールはその中でも最大の領域を占めることが多い。 バッファマネージャは BufferManagerShmemRequest()（後述）で必要量を申告し、 確保された領域へ各配列のポインタを割り付ける。

図 1.3 に共有メモリの大まかな内訳を示す。shared_buffers を大きくすると、 バッファブロック本体だけでなく、それに付随するバッファ記述子やバッファテーブルなどの 管理領域も NBuffers に比例して増える。これらの内訳は第2章で詳説する。

以上で shared_buffers の正体（GUC 変数 NBuffers）と、それが共有メモリ上の バッファプールを決めることを確認した。次章では、そのバッファプールが具体的にどのような 配列の集まりで構成されているかを見ていく。
