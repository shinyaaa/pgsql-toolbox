第4章　バッファタグとバッファテーブル

バックエンドが「あるテーブルの N 番目のブロック」を求めたとき、それが共有バッファのどの スロットにあるのか（あるいは無いのか）を即座に判定しなければならない。この対応づけを 担うのが、ページの識別子である BufferTag と、タグからバッファ番号を引くハッシュ表 バッファテーブルである。本章ではこの2つと、競合を抑えるパーティション機構を見る。

4.1　BufferTag — ページの識別子

BufferTag は、共有バッファが保持するディスクページを一意に特定する構造体である。 定義は buf_internals.h にあり、図 4.1 の5つのフィールドからなる。

  • spcOid / dbOid / relNumber：どのリレーション（テーブル・インデックス）の物理ファイルかを表す3つ組（RelFileLocator 相当）。
  • forkNum：フォーク種別。本体（main）・空き領域マップ（fsm）・可視性マップ（vm）などを区別する。
  • blockNum：そのフォーク内のブロック番号（0始まり）。

ソースのコメントが強調しているように、BufferTag は pg_class や pg_tablespace を引かずとも、そのページをディスクのどこへ書き戻すかを決定できるだけの 情報を含んでいる。これはバッファを書き出すバックエンドが、対象リレーションをまだ可視と 認識していない場合でも正しく動作させるためである。

メモBufferTag はハッシュ表のキーとして使われるため、構造体に パディング（隙間）があってはならない。隙間に不定値が残るとキー比較が壊れるため、 InitBufferTag() や ClearBufferTag() で全フィールドを明示的に設定する。

4.2　バッファテーブル — タグからバッファを引く

バッファテーブルは、BufferTag をキー、バッファ番号（buf_id）を値とする 共有ハッシュ表である。実体は SharedBufHash という HTAB で、 buf_table.c が薄いラッパ関数を提供する。図 4.2 に役割を示す。

主な操作は次の3つである。

  • BufTableLookup()：タグに対応する buf_id を返す。無ければ −1。
  • BufTableInsert()：タグと buf_id の対応を登録する。すでに別バッファが登録済みなら、その buf_id を返す。
  • BufTableDelete()：タグの対応を削除する（バッファを追い出すとき）。

ハッシュ表のエントリ数は、最悪ケースに備えて NBuffers + NUM_BUFFER_PARTITIONS 個で 確保される。バッファを置換する際、新タグの挿入と旧タグの削除を一時的に同時に保持し得るため、 バッファ数ぴったりでは足りない場面があるからである。

4.3　パーティションとマッピングロック

バッファテーブルはインスタンス内の全バックエンドが共有するため、アクセスの集中は競合を生む。 これを緩和するため、ハッシュ表は NUM_BUFFER_PARTITIONS = 128 個のパーティションに 分割され（lwlock.h）、 パーティションごとに専用の LWLock（マッピングロック）を持つ。

どのパーティションに属するかは、タグのハッシュ値から決まる。図 4.3 のように、 BufTableHashCode() で得たハッシュ値を NUM_BUFFER_PARTITIONS で割った余りが パーティション番号となり、対応するロックを BufMappingPartitionLock() が返す。 異なるパーティションに属するタグへのアクセスは、互いに待たされない。

このパーティション分割により、別々のページに同時アクセスする多数のバックエンドが、 それぞれ別のロックを取れるようになり、バッファテーブルがボトルネックになりにくい。 実際のページ探索でこれらのロックがどう使われるかは、次章の BufferAlloc() の流れで見る。
