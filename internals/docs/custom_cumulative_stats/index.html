Custom Cumulative Statistics API の内部構造

PostgreSQL 19beta1 · 79c65b9 · 2026-06-14 13:18

PostgreSQL の累積統計システム（cumulative statistics system）を拡張し、拡張機能（extension）が独自の統計種別を定義するための仕組みが Custom Cumulative Statistics API である。本ドキュメントは、種別を記述する PgStat_KindInfo、登録関数 pgstat_register_kind()、および公式テストモジュール src/test/modules/test_custom_stats を題材に、API の構造と内部動作を PostgreSQL 19beta1 のソースコードに基づいて図と実例で解説する。対象読者は累積統計を拡張する開発者、および pgstat 内部に関心のあるエンジニア。

第1章 累積統計システムと拡張点 固定個数・可変個数の統計、pending から共有メモリへの流れ、統計種別 ID の空間、そして本 API がどこに位置づくかを概観する。 第2章 PgStat_KindInfo — 種別を定義するメタデータ 統計種別の振る舞いをすべて決める PgStat_KindInfo 構造体を、共通フィールド・可変個数向けコールバック・固定個数向けコールバックに分けて読み解く。 第3章 種別の登録 — pgstat_register_kind() shared_preload_libraries と _PG_init での登録、ID の割り当てと実験枠、登録時に課される検証、そしてレジストリ配列による種別の解決を追う。 第4章 可変個数の統計を実装する test_custom_var_stats を題材に、共有エントリと pending 構造体の設計、pending から共有メモリへの反映、ディスク永続化、そして SQL からの観察までを追う。 第5章 固定個数の統計を実装する test_custom_fixed_stats を題材に、固定共有メモリのレイアウトと changecount による低オーバーヘッド更新、init/snapshot/reset の各コールバック、SQL からの観察とリセットを追う。 第6章 ライフサイクルと運用 登録から起動・スナップショット・永続化までのタイミング、クラッシュリカバリでの可変／固定の違い、既存の統計機構との統合、そして ID 衝突をはじめとする運用上の注意点をまとめる。
