第3章　種別の登録 — pgstat_register_kind()

PgStat_KindInfo を用意したら、それを統計システムに登録する必要がある。 登録の入口は pgstat_register_kind() ただ一つで、 本章ではその前提条件・検証・登録先のレジストリを順に見ていく。

3.1　登録の前提と流れ

カスタム種別は、shared_preload_libraries に指定されたモジュールの初期化中にのみ登録できる。 具体的には、共有ライブラリの _PG_init() の中で、 process_shared_preload_libraries_in_progress が真であることを確認してから pgstat_register_kind() を呼ぶ。これにより、すべてのバックエンドが起動の早い段階で同一の種別を共有できる。

C — test_custom_var_stats.c:129 
void
_PG_init(void)
{
    /* Must be loaded via shared_preload_libraries */
    if (!process_shared_preload_libraries_in_progress)
        return;

    /* Register custom statistics kind */
    pgstat_register_kind(PGSTAT_KIND_TEST_CUSTOM_VAR_STATS, &custom_stats);
}

図 3.1 に登録の流れを示す。サーバ起動時にポストマスタが shared_preload_libraries を読み込み、 各モジュールの _PG_init() を呼ぶ。その中で pgstat_register_kind() が呼ばれ、検証を通過すると カスタムレジストリ pgstat_kind_custom_infos[] にメタデータのポインタが格納される。

3.2　ID の割り当てと PGSTAT_KIND_EXPERIMENTAL

カスタム種別の ID（24〜32）は、全拡張機能を通じて一意でなければならない。複数の拡張が同じ ID を使うと衝突する。 そのため PostgreSQL は、ID を予約するための公式 Wiki ページ （https://wiki.postgresql.org/wiki/CustomCumulativeStats）を用意している。 リリース時にはここで自分の拡張用の一意な ID を予約する。

一方、開発段階でまだ ID を予約していない拡張のために、実験用の枠 PGSTAT_KIND_EXPERIMENTAL（= 24）が用意されている。 公式ドキュメント（doc/src/sgml/xfunc.sgml）も「開発中は kind に PGSTAT_KIND_EXPERIMENTAL を使い、 ユーザへ公開する段になったら Wiki で正式な ID を予約せよ」と案内している。

注意　実験枠 24 は「一時的な共有枠」である。複数の開発中拡張を同時にロードすると 24 が衝突するため、 本番配布前には必ず専用 ID を予約すること。

3.3　登録時の検証

pgstat_register_kind() は登録に先立ち、いくつもの前提を ereport(ERROR) で厳格に検査する。 主な検査は次のとおり。

  • 名前の非空：kind_info->name が NULL または空文字列なら拒否。
  • ID 範囲：pgstat_is_kind_custom(kind) が偽（24〜32 の外）なら「out of range」で拒否。
  • 登録タイミング：process_shared_preload_libraries_in_progress が偽なら、 「shared_preload_libraries の初期化中に登録せよ」と拒否。
  • 固定個数の条件：fixed_amount が真なら shared_size は 0 であってはならず、 かつ track_entry_count を使ってはならない。
  • 重複の禁止：同じ ID への二重登録、および同名の種別が既に登録されている場合は拒否。
C — pgstat.c:1530 付近（固定個数の検査） 
if (kind_info->fixed_amount)
{
    if (kind_info->shared_size == 0)
        ereport(ERROR,
                (errmsg("custom cumulative statistics property is invalid"),
                 errhint("Custom cumulative statistics require a shared memory size for fixed-numbered objects.")));
    if (kind_info->track_entry_count)
        ereport(ERROR,
                (errmsg("custom cumulative statistics property is invalid"),
                 errhint("Custom cumulative statistics cannot use entry count tracking for fixed-numbered objects.")));
}

すべての検査を通過すると、メタデータのポインタがレジストリに格納され、LOG レベルで 「registered custom cumulative statistics "..." with ID ...」が出力される。

3.4　レジストリと種別の解決

カスタム種別のメタデータは、TopMemoryContext 上に確保されるポインタ配列 pgstat_kind_custom_infos[] （要素数 PGSTAT_KIND_CUSTOM_SIZE = 9）に格納される。配列の添字は kind − PGSTAT_KIND_CUSTOM_MIN で、 最初の登録時に遅延確保される。図 3.2 に、ID から添字へのマッピングを示す。

種別の解決は次の二つの関数が担う。いずれも組み込み種別とカスタム種別を透過的に扱う。

  • pgstat_get_kind_info(kind) — ID からメタデータを引く。組み込みなら pgstat_kind_builtin_infos[kind]、カスタムなら pgstat_kind_custom_infos[idx] を返す（未登録なら NULL）。
  • pgstat_get_kind_from_str(name) — 種別名（文字列）から ID を引く。組み込みを走査した後、カスタム配列を名前で線形探索する。

名前による探索が存在することからも、登録時に同名種別を禁止している理由が分かる。名前は種別を一意に指す識別子として機能する。 次章からは、この API を使って実際に統計種別を実装する二つの例を読み解いていく。
