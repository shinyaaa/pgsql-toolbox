第5章　固定個数の統計を実装する

固定個数の統計は、サーバ全体で決まった数（この例では 1 つ）の対象を数えるものである。テストモジュールの test_custom_fixed_stats.c は、可変個数版と対をなす最小の固定個数例で、pending を介さず共有メモリを直接更新する点が特徴である。

5.1　共有メモリレイアウトと changecount

固定個数の統計エントリは、通常の共有メモリ領域に置かれる。読み取りより書き込みが性能上重要なため、組み込みの PgStatShared_Archiver などと同じく changecount 機構を使う。これは「書き込み開始・終了でカウンタを増やし、 読み手は読み取り前後でカウンタが一致するまで再読する」ことで、ロックなしの一貫読み取りを実現する仕組みである。 リセット時のために、現在値 stats とは別にリセット基準 reset_offset を保持する。

C — test_custom_fixed_stats.c:29 
typedef struct PgStat_StatCustomFixedEntry
{
    PgStat_Counter numcalls;          /* update 関数が呼ばれた回数 */
    TimestampTz    stat_reset_timestamp;
} PgStat_StatCustomFixedEntry;

typedef struct PgStatShared_CustomFixedEntry
{
    LWLock      lock;                 /* カウンタを保護 */
    uint32      changecount;          /* アトミック読み取り用 */
    PgStat_StatCustomFixedEntry stats;        /* 現在のカウンタ */
    PgStat_StatCustomFixedEntry reset_offset; /* リセット基準 */
} PgStatShared_CustomFixedEntry;

図 5.1 に固定エントリのレイアウトと、更新・読み取り・リセットの関係を示す。KindInfo では fixed_amount = true とし、init_shmem_cb / snapshot_cb / reset_all_cb を設定する。 固定個数では track_entry_count は使えない（3.3 節の検証を参照）。

5.2　init / snapshot / reset コールバック

固定個数の共有メモリは pgstat が確保し、その直後に init_shmem_cb が呼ばれる。ここで LWLock を初期化する。 共有メモリ／スナップショット領域へのアクセスには、2.4 節で触れた専用アクセサ pgstat_get_custom_shmem_data() / pgstat_get_custom_snapshot_data() を使う。

C — test_custom_fixed_stats.c:87（init_shmem_cb） 
static void
test_custom_stats_fixed_init_shmem_cb(void *stats)
{
    PgStatShared_CustomFixedEntry *stats_shmem =
        (PgStatShared_CustomFixedEntry *) stats;

    LWLockInitialize(&stats_shmem->lock, LWTRANCHE_PGSTATS_DATA);
}

更新関数 test_custom_stats_fixed_update() は、LWLock を排他で取り、changecount の書き込み区間でカウンタを増やす。可変個数と違い pending を経由しない。

C — test_custom_fixed_stats.c:154（update） 
stats_shmem = pgstat_get_custom_shmem_data(PGSTAT_KIND_TEST_CUSTOM_FIXED_STATS);

LWLockAcquire(&stats_shmem->lock, LW_EXCLUSIVE);
pgstat_begin_changecount_write(&stats_shmem->changecount);
stats_shmem->stats.numcalls++;
pgstat_end_changecount_write(&stats_shmem->changecount);
LWLockRelease(&stats_shmem->lock);

読み取りの一貫性は snapshot_cb が担う。 test_custom_stats_fixed_snapshot_cb() は changecount を使って現在値をスナップショット領域へコピーし、リセット基準を差し引いて「リセット以降の増分」に直す。

C — test_custom_fixed_stats.c:120（snapshot_cb 抜粋） 
pgstat_copy_changecounted_stats(stat_snap,
                                &stats_shmem->stats,
                                sizeof(stats_shmem->stats),
                                &stats_shmem->changecount);

LWLockAcquire(&stats_shmem->lock, LW_SHARED);
memcpy(&reset, reset_offset, sizeof(stats_shmem->stats));
LWLockRelease(&stats_shmem->lock);

/* リセット基準を適用 */
#define FIXED_COMP(fld) stat_snap->fld -= reset.fld;
FIXED_COMP(numcalls);
#undef FIXED_COMP

リセットは reset_all_cb （100 行）が、 現在値を reset_offset へ退避し、stat_reset_timestamp を更新することで実現する。 共有メモリ上の値そのものは書き換えない（他バックエンドが書き込み中かもしれないため）。これは組み込みの PgStatShared_Archiver と同じリセットプロトコルである。

5.3　SQL からの観察とリセット

観察関数 test_custom_stats_fixed_report() は、pgstat_snapshot_fixed() でスナップショットを取り、(numcalls, stats_reset) を返す。リセット前は stats_reset が NULL になる。

SQL 
CREATE EXTENSION test_custom_fixed_stats;

SELECT test_custom_stats_fixed_update();
SELECT test_custom_stats_fixed_update();
SELECT test_custom_stats_fixed_update();

SELECT * FROM test_custom_stats_fixed_report();
実行結果 
 numcalls | stats_reset
----------+-------------
        3 |

3 回の更新が numcalls = 3 として観察され、まだリセットしていないので stats_reset は NULL である。 test_custom_stats_fixed_reset() は pgstat_reset_of_kind() を呼んでリセットする。

SQL 
SELECT test_custom_stats_fixed_reset();
SELECT * FROM test_custom_stats_fixed_report();
実行結果 
 numcalls |          stats_reset
----------+-------------------------------
        0 | 2026-06-14 13:20:00.123456+09

リセット後は増分が 0 になり、stats_reset にリセット時刻が入る。固定個数の統計は通常再起動では保持され、 クラッシュリカバリでは（破棄ではなく）リセットされた状態になる。この違いは次章で整理する。
