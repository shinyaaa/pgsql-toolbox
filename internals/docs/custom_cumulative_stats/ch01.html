第1章　累積統計システムと拡張点

PostgreSQL の累積統計システム（cumulative statistics system）は、テーブルへのアクセス回数、 チェックポイントの実行回数、WAL の生成量といった「起動から積み上がっていく」統計を収集・公開する仕組みである。 本章では、この仕組みの全体像と、外部の拡張機能が独自の統計を差し込むための拡張点 Custom Cumulative Statistics API がどこに位置づくかを概観する。実装の中心は src/backend/utils/activity/pgstat.c にある。

1.1　累積統計システムの全体像

累積統計は収集対象の「個数」によって二種類に分かれる。固定個数（fixed-numbered）の統計は、 収集対象がサーバ全体で決まった数（多くは 1 つ）しか存在しないもので、チェックポインタや WAL の統計が該当する。 一方、可変個数（variable-numbered）の統計は、テーブルや関数のように対象の数が実行時に増減するものである。 この区別は pgstat.c 冒頭の説明コメントで明言されている。

性能を確保するため、ほとんどの統計更新はまず各プロセスのローカルメモリに pending（保留）エントリとして蓄積され、 後でまとめて共有メモリへ反映（flush）される。これにより、個々の統計エントリへのロック競合がほぼ解消される。 保留分の反映は pgstat_report_stat() が担う。可変個数の統計は動的共有メモリ（DSA）上の dshash ハッシュテーブルに、固定個数の統計は通常の共有メモリ領域に格納される。 図 1.1 に示すように、統計はバックエンドのローカル領域から共有メモリへ、最終的にはディスク上の統計ファイルへと流れる。

統計ファイルへの書き出しは、シャットダウン直前にチェックポインタプロセスが行う（種別が永続化を許可している場合）。 起動時には起動プロセスがファイルを読み込むが、直前にクラッシュしていた場合はすべての統計が破棄される。 この「クラッシュ時は捨てる」という性質は、後述する固定個数・可変個数それぞれの挙動（6.2 節）に影響する。

1.2　統計種別と ID 空間

各統計は統計種別（statistics kind）で区別される。種別は src/include/utils/pgstat_kind.h で定義される 32 ビット整数（PgStat_Kind）で表され、有効な ID は 1〜32 に限られる。 0 は PGSTAT_KIND_INVALID としてゼロ初期化データを検出するために予約される。

C — pgstat_kind.h 
#define PgStat_Kind uint32

#define PGSTAT_KIND_MIN  1    /* Minimum ID allowed */
#define PGSTAT_KIND_MAX  32   /* Maximum ID allowed */
#define PGSTAT_KIND_INVALID 0

/* stats for variable-numbered objects */
#define PGSTAT_KIND_DATABASE      1
#define PGSTAT_KIND_RELATION      2
...
/* stats for fixed-numbered objects */
#define PGSTAT_KIND_WAL           13

#define PGSTAT_KIND_BUILTIN_MIN PGSTAT_KIND_DATABASE   /* = 1  */
#define PGSTAT_KIND_BUILTIN_MAX PGSTAT_KIND_WAL        /* = 13 */

/* Range of IDs allowed for custom stats kinds */
#define PGSTAT_KIND_CUSTOM_MIN   24
#define PGSTAT_KIND_CUSTOM_MAX   PGSTAT_KIND_MAX       /* = 32 */
#define PGSTAT_KIND_EXPERIMENTAL 24

ID 空間は図 1.2 のように区分される。組み込み種別（built-in）は 1〜13 を占有し、カスタム種別（custom）には 24〜32 が割り当てられている。両者の間（14〜23）は将来の組み込み種別のために空けられている。 カスタム範囲の判定は pgstat_is_kind_custom() が、組み込み範囲の判定は pgstat_is_kind_builtin() が行う。

メモ　カスタム枠は PGSTAT_KIND_CUSTOM_SIZE = PGSTAT_KIND_CUSTOM_MAX − PGSTAT_KIND_CUSTOM_MIN + 1 = 9 個。 このサイズが、後述するレジストリ配列 pgstat_kind_custom_infos の要素数になる。

1.3　Custom Cumulative Stats API の位置づけ

pgstat.c の設計コメントは「外部モジュールがカスタム統計種別を定義でき、組み込み種別と同じ性質をすべて利用できる」と述べている。 すなわち拡張機能は、固定個数・可変個数のいずれの統計でも、pending／共有メモリ／ディスク永続化／スナップショットといった 既存インフラをそのまま再利用して自前の統計を実装できる。種別ごとの振る舞いはすべて、種別を記述するメタデータ構造体 PgStat_KindInfo に集約される（第2章）。

組み込み種別のメタデータは配列 pgstat_kind_builtin_infos[] に静的に並んでいるのに対し、カスタム種別のメタデータは起動時に pgstat_register_kind() で登録され、ポインタ配列 pgstat_kind_custom_infos[] に格納される（第3章）。両者を透過的に引くのが pgstat_get_kind_info() である。

登録は shared_preload_libraries 経由でロードされたモジュールの初期化中にのみ許可される。 これは、すべてのバックエンドが同じカスタム種別を同じ ID で共有する必要があるためである。 本 API の実例は src/test/modules/test_custom_stats に可変個数版・固定個数版の両方が用意されており、 本ドキュメントでも第4章・第5章でこれらを題材に用いる。次章ではまず、種別を定義する PgStat_KindInfo を詳しく見る。
