第2章　PgStat_KindInfo — 種別を定義するメタデータ

カスタム統計種別の性質は、ただ一つの構造体 PgStat_KindInfo に集約される。固定個数か可変個数か、ディスクに書き出すか、エントリのサイズはいくつか、 保留データをどう反映するか――こうした問いへの答えがすべてこの構造体のフィールドとして表現される。 本章ではこの構造体を 3 つのグループに分けて読み解く。

2.1　構造体の全体像

PgStat_KindInfo は、先頭にビットフィールドのフラグ群、続いてサイズ・オフセット群、 そして多数のコールバック関数ポインタ、最後に種別名 name を持つ。図 2.1 に主要フィールドの分類を示す。 コールバックの多くは可変個数向けか固定個数向けのいずれかであり、種別の性質に応じて必要なものだけを設定する。

2.2　共通フィールド

種別の基本的な性格を決めるのが共通フィールドである。代表的なものを挙げる。

  • fixed_amount — この種別が固定個数か可変個数かを表すブール値。API の二分岐の起点。
  • write_to_file — 統計をディスクの統計ファイルへ書き出すか。
  • shared_size — 共有メモリ上の 1 エントリのサイズ。固定個数では PgStat_ShmemControl->custom_data の要素サイズになる。
  • shared_data_off / shared_data_len — エントリ内で「統計データ本体」が占めるオフセットと長さ。 ディスクとの直列化（serialize）で用いる。LWLock などのメモリ上の状態を除くため shared_size とは別に持つ。
  • name — 種別名。空文字列は許されない（登録時に検査される）。

shared_data_off と shared_data_len は、エントリ構造体を offsetof と sizeof で記述するのが定石である。 たとえば可変個数の例では次のように設定する（詳細は 第4章）。

C — test_custom_var_stats.c の KindInfo（抜粋） 
static const PgStat_KindInfo custom_stats = {
    .name = "test_custom_var_stats",
    .fixed_amount = false,        /* variable number of entries */
    .write_to_file = true,        /* persist across restarts */
    .shared_size     = sizeof(PgStatShared_CustomVarEntry),
    .shared_data_off = offsetof(PgStatShared_CustomVarEntry, stats),
    .shared_data_len = sizeof(((PgStatShared_CustomVarEntry *) 0)->stats),
    ...
};

2.3　可変個数向けのコールバック

可変個数の統計は、各プロセスのローカルな pending エントリを共有メモリへ反映する必要があるため、 それに対応するフィールドとコールバックを持つ。

  • pending_size — PgStat_EntryRef->pending のサイズ。0 はこの種別が pending を持たないことを示す。
  • track_entry_count — エントリ数を PgStat_ShmemControl.entry_counts で追跡するか。
  • flush_pending_cb — pending を共有統計へ統合する必須コールバック（pending を使う場合）。
  • delete_pending_cb — pending を破棄する（任意）。
  • to_serialized_data / from_serialized_data — エントリに付随する補助データをファイルへ読み書きする（任意）。
  • to_serialized_name / from_serialized_name、reset_timestamp_cb — 直列化名やリセット時刻の扱い（任意）。

flush_pending_cb のシグネチャは bool (*)(PgStat_EntryRef *sr, bool nowait) で、 ロックを取れなかった場合に false を返す。実装例は 4.2 節で詳述する。

2.4　固定個数向けのコールバック

固定個数の統計は pending を介さず共有メモリを直接更新するのが一般的で、代わりに共有メモリの初期化やスナップショット生成のための コールバックを持つ。

  • init_shmem_cb — 割り当てられた共有メモリ領域を初期化する（LWLock の初期化など）。
  • snapshot_cb — 読み取り一貫性のために、現在値をスナップショット領域へコピーする。
  • reset_all_cb — その種別の統計をすべてリセットする。
  • flush_static_cb — PgStat_EntryRef->pending を使わない種別が保留統計を反映する（任意）。
  • init_backend_cb / finish — バックエンド初期化時・統計ファイル処理完了時の任意フック（固定・可変共通）。

固定個数の統計は、共有メモリ制御構造体 PgStat_ShmemControl の custom_data[] に、 スナップショットは PgStat_Snapshot の custom_data[] （同 625 行）に格納される。 これらは kind − PGSTAT_KIND_CUSTOM_MIN を添字とする配列で、アクセスには専用のインライン関数 pgstat_get_custom_shmem_data() と pgstat_get_custom_snapshot_data() を用いる。 実装例は 第5章で扱う。

関連　PgStat_KindInfo の全フィールドの API 詳細は pgstat_internal.h のコメントに記されている。 公式ドキュメントも「API の詳細は pgstat_internal.h を参照」と案内している。
