第6章　ライフサイクルと運用

最後に、カスタム統計種別のライフサイクルと運用上の論点を整理する。 登録から永続化までのタイミング、クラッシュ時の挙動、既存ツールとの関係、そして衝突回避のための作法を順に見る。

6.1　起動から永続化までのタイミング

カスタム統計種別は、サーバの起動から停止まで一連のフェーズに沿って動く（図 6.1）。 登録は shared_preload_libraries のロード時（第3章）、 読み込みは起動プロセス、書き出しはチェックポインタが担当する。

読み取り一貫性は GUC stats_fetch_consistency に従う。snapshot や cache を選ぶと、 固定個数では snapshot_cb が、可変個数ではエントリ単位のコピーが、ローカルのスナップショット領域 pgStatLocal.snapshot に取り込まれる。これにより同一トランザクション内で安定した値が読める。

6.2　クラッシュリカバリでの扱い

統計ファイルは「直前にクラッシュした場合は破棄する」（1.1 節）。この性質により、 可変個数と固定個数で観察される結果が異なる。公式 TAP テスト t/001_custom_stats.pl はこの差を明示的に検証している。

  • 通常のシャットダウン → 再起動：可変個数・固定個数とも値が保持される（テストは entry1=2、固定 numcalls=3 の維持を確認）。
  • クラッシュ（stop immediate）→ 再起動：統計ファイルが破棄されるため、可変個数のエントリは消失し （_report が空を返す）、固定個数は初期状態すなわちリセット相当になる。

6.3　既存の統計機構との統合

カスタム種別は組み込み種別と同じインフラに乗るため、汎用の統計関数からも透過的に扱える。種別名から ID を引く pgstat_get_kind_from_str() を通じて、pg_stat_have_stats() のような関数に種別名（例: 'test_custom_var_stats'）を渡してエントリの有無を問い合わせられる。 リセットも pgstat_reset_of_kind() を種別 ID で呼ぶだけでよい（5.3 節）。

拡張側が用意すべきなのは、内部統計を SQL に見せるための観察用関数である。テストモジュールが _report 関数を提供しているのはこのためで、収集（pgstat 側）と公開（拡張の SQL 関数）は分離して設計する。

6.4　注意点とベストプラクティス

  • shared_preload_libraries 必須：登録はプリロード初期化中のみ。通常の CREATE EXTENSION だけでは 種別は登録されない。モジュールを shared_preload_libraries に入れること。
  • ID は一意に：本番配布前に Wiki で専用 ID を予約する。開発中は PGSTAT_KIND_EXPERIMENTAL（24）を使い、 他の実験中拡張と同時にロードしない（3.2 節）。
  • 名前も一意に：同名種別は登録時に拒否される。種別名は SQL から種別を指す識別子になるため、衝突しにくい名前を選ぶ。
  • 固定個数の制約：fixed_amount では shared_size > 0 が必須で、track_entry_count は使えない。
  • 直列化の範囲：標準の永続化は shared_data_off〜shared_data_len の固定サイズ部分のみ。 エントリ外の可変長データは to/from_serialized_data で自前処理する（4.3 節）。
  • クラッシュ耐性を過信しない：累積統計はクラッシュで失われ得る前提で設計する。厳密な永続性が必要なデータには不向き。

以上で、Custom Cumulative Statistics API の構造（PgStat_KindInfo）、登録（pgstat_register_kind()）、 可変個数・固定個数それぞれの実装、そしてライフサイクルと運用までを一巡した。拡張機能は、これらの部品を組み合わせることで、 組み込み統計と同等の品質を持つ独自統計を最小限のコードで実装できる。
