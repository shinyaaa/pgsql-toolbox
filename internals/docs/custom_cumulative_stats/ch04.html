第4章　可変個数の統計を実装する

可変個数の統計とは、テーブルや関数のように対象の数が実行時に増減する統計である。 公式テストモジュールの test_custom_var_stats.c は、名前付きのエントリごとに「呼び出し回数」を数える最小の例で、可変個数 API の使い方を一通り示している。 本章ではこれを題材に実装の流れを読み解く。

4.1　データ構造の設計

可変個数の統計には二つの構造体を用意する。一つは共有メモリ上のエントリ、もう一つは各プロセスのローカルな pending である。 共有エントリは必ず先頭に共通ヘッダ PgStatShared_Common を置き、その後ろに統計データ本体を続ける。

C — test_custom_var_stats.c:54 
/* バックエンドローカルの pending 統計（共有へ反映される前） */
typedef struct PgStat_StatCustomVarEntry
{
    PgStat_Counter numcalls;    /* インクリメント回数 */
} PgStat_StatCustomVarEntry;

/* 全バックエンドから見える共有メモリ上のエントリ */
typedef struct PgStatShared_CustomVarEntry
{
    PgStatShared_Common header;         /* 標準の pgstat エントリヘッダ */
    PgStat_StatCustomVarEntry stats;    /* カスタム統計データ */
    dsa_pointer description;            /* DSA 上の説明文字列へのポインタ */
} PgStatShared_CustomVarEntry;

これらを PgStat_KindInfo に結びつける（図 4.1）。shared_data_off / shared_data_len は offsetof と sizeof で stats 部分だけを指すようにし、pending_size に pending 構造体のサイズを設定する。 track_entry_count を真にしてエントリ数を追跡し、accessed_across_databases を真にしてデータベース横断でアクセス可能にしている。

C — test_custom_var_stats.c:108（KindInfo） 
static const PgStat_KindInfo custom_stats = {
    .name = "test_custom_var_stats",
    .fixed_amount = false,
    .write_to_file = true,
    .track_entry_count = true,
    .accessed_across_databases = true,
    .shared_size     = sizeof(PgStatShared_CustomVarEntry),
    .shared_data_off = offsetof(PgStatShared_CustomVarEntry, stats),
    .shared_data_len = sizeof(((PgStatShared_CustomVarEntry *) 0)->stats),
    .pending_size    = sizeof(PgStat_StatCustomVarEntry),
    .flush_pending_cb     = test_custom_stats_var_flush_pending_cb,
    .to_serialized_data   = test_custom_stats_var_to_serialized_data,
    .from_serialized_data = test_custom_stats_var_from_serialized_data,
    .finish               = test_custom_stats_var_finish,
};

4.2　更新：pending から共有メモリへ

可変個数の統計更新は二段階で進む。図 4.2 に示すように、まず更新側はローカルの pending を増やすだけで、共有メモリには触れない。 後で pgstat_report_stat() が走るタイミングで、flush_pending_cb が pending を共有エントリへ加算する。

更新を行う SQL 関数 test_custom_stats_var_update() は、pgstat_prep_pending_entry() で pending を確保し、カウンタを増やすだけである。

C — test_custom_var_stats.c:571 
entry_ref = pgstat_prep_pending_entry(PGSTAT_KIND_TEST_CUSTOM_VAR_STATS, InvalidOid,
                                      PGSTAT_CUSTOM_VAR_STATS_IDX(stat_name), NULL);

pending_entry = (PgStat_StatCustomVarEntry *) entry_ref->pending;
pending_entry->numcalls++;

反映を担う test_custom_stats_var_flush_pending_cb() は、エントリをロックしてから pending の値を共有側へ加算する。ロックを取れない（nowait）場合は false を返し、後で再試行される。

C — test_custom_var_stats.c:155 
pending_entry = (PgStat_StatCustomVarEntry *) entry_ref->pending;
shared_entry  = (PgStatShared_CustomVarEntry *) entry_ref->shared_stats;

if (!pgstat_lock_entry(entry_ref, nowait))
    return false;

/* pending を共有の合計へ加算 */
shared_entry->stats.numcalls += pending_entry->numcalls;

pgstat_unlock_entry(entry_ref);
return true;

エントリの生成は test_custom_stats_var_create() が pgstat_get_entry_ref_locked() で行い、削除は test_custom_stats_var_drop() が pgstat_drop_entry() で行う。エントリのキーには統計名のハッシュ値を用いている。

4.3　ディスクへの永続化

write_to_file = true の種別は、統計データ本体（shared_data_off〜shared_data_len の範囲）が 標準の統計ファイルへ自動的に直列化される。ただしこの例では、エントリに付随する可変長の説明文字列を DSA 上に保持しているため、それは標準の直列化に乗らない。そこで to_serialized_data / from_serialized_data コールバックを用い、補助ファイル pg_stat/test_custom_var_stats_desc.stats へ説明文字列を別途読み書きしている（実装は test_custom_stats_var_to_serialized_data() 以降）。 finish コールバック （409 行）は、 読み書き・破棄といったファイル操作の完了時に補助ファイルの後始末を行う。

メモ　固定サイズの統計データは標準の直列化だけで永続化できる。 to/from_serialized_data が必要になるのは、この例の説明文字列のようにエントリ外の可変長データを持つ場合である。

4.4　SQL からの観察

拡張は観察用の SQL 関数も提供する。 test_custom_stats_var_report() は名前を受け取り、(name, calls, description) を返す。公式 TAP テスト t/001_custom_stats.pl が示す再現可能な手順を以下に示す。前提として両モジュールを shared_preload_libraries に登録しておく。

SQL 
CREATE EXTENSION test_custom_var_stats;

SELECT test_custom_stats_var_create('entry1', 'Test entry 1');
SELECT test_custom_stats_var_update('entry1');
SELECT test_custom_stats_var_update('entry1');

SELECT * FROM test_custom_stats_var_report('entry1');
実行結果 
  name  | calls | description
--------+-------+--------------
 entry1 |     2 | Test entry 1

2 回の _update が pending を経て共有エントリへ反映され、numcalls = 2 として観察される。 エントリを削除すると以後の _report は行を返さない。

SQL 
SELECT test_custom_stats_var_drop('entry1');
SELECT * FROM test_custom_stats_var_report('entry1');   -- 0 行

この可変個数の統計は、通常のシャットダウン／再起動では保持されるが、クラッシュリカバリでは失われる （6.2 節）。次章では、もう一方の固定個数の統計を見る。
